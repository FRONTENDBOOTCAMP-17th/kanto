## 4. 개선 작업 내용 (문제 → 기대 효과 → 그래서 택한 방법)

### 4-1. 지도 지연 로딩 (서드 파티 Google Maps 497KiB)

- **왜 고쳐야 했나**: 지도는 접힌 영역(스크롤 아래)에 있는데도 Google Maps JS(~497KiB)가 초기 로드부터 실행돼 메인 스레드를 점유 → LCP 요소 렌더링 지연 680ms의 주범.
- **개선 안 했을 때**: 첫 화면에 필요 없는 지도 스크립트가 LCP 구간에 끼어들어 텍스트 렌더가 밀림.
- **개선했을 때 (기대)**: 지도 스크립트가 뷰포트 진입 시점으로 밀리면서 초기 메인 스레드가 비워져 LCP 텍스트가 먼저 렌더 → LCP·TBT 개선.
- **그래서 택한 방법**: `CompanyLocationMapLazy.tsx` 신규 추가.
  - `dynamic(() => import("./CompanyLocationMap"), { ssr: false })`로 지도 컴포넌트를 클라이언트 비동기 청크로 분리.
  - `IntersectionObserver`(`rootMargin: "200px"`)로 뷰포트 200px 앞에 들어올 때 `visible=true`가 되면 그때 마운트. 그 전에는 동일 높이(`h-56`) placeholder를 렌더해 **레이아웃 시프트(CLS) 없이** 자리만 잡아둠.
  - `CompanyInfo.tsx`의 import를 `CompanyLocationMap` → `CompanyLocationMapLazy`로 교체(호출부 변경 없음).

### 4-2. 회사 로고 이미지 최적화 (이미지 전송 245KiB)

- **왜 고쳐야 했나**: 로고가 순수 `<img>`라 원본 720×960이 그대로 전송됨(표시 크기 71×95). next/image 파이프라인(리사이즈·AVIF/WebP)을 못 탐.
- **개선했을 때 (기대)**: `next/image`로 바꾸면 표시 크기에 맞게 리사이즈되고 AVIF/WebP로 변환돼 로고 1장당 전송량이 크게 줄어듦.
- **그래서 택한 방법**: `CompanyInfo.tsx`의 `CompanyLogo`에서 `<img>` → `next/image`의 `<Image width={56} height={56}>`로 교체. (표시 크기 71×95에 근접한 명시 크기로 다운로드 용량 축소 + 레이아웃 확정.)

### 4-3. 관련 구인 쿼리 병렬화 (서버 응답 시간)

- **왜 고쳐야 했나**: 기존에는 `getUserLikeReportStatus`·번역 로드를 `Promise.all`로 묶어 받은 **뒤에** 관련 구인 쿼리를 순차로 다시 await → 서버 응답이 두 단계로 직렬화됨.
- **개선했을 때 (기대)**: 관련 구인 쿼리를 같은 `Promise.all`에 합치면 네트워크 왕복이 겹쳐 실행돼 서버 렌더 시간이 단축됨(TTFB·초기 HTML 도달 시간 개선).
- **그래서 택한 방법**: `page.tsx`에서 관련 구인 조회를 `fetchRelated()` 함수로 감싸 `Promise.all([getUserLikeReportStatus, getTranslations×2, fetchRelated()])`에 포함. 결과는 구조 분해로 받음.
