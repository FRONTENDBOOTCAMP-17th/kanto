## 4. 개선 작업 내용 (문제 → 기대 효과 → 그래서 택한 방법)

### 4-1. 좋아요·신고 상태 조회를 관련 매물 조회와 병렬 실행

- **왜 고쳐야 했나**: `getUserLikeReportStatus`와 관련 매물 조회는 서로 결과를 참조하지 않는 독립 쿼리인데도 순차 `await`로 묶여 서버 렌더 시간이 불필요하게 직렬 누적됐음.
- **개선 안 했을 때**: 두 쿼리 지연 시간이 그대로 더해져(=직렬) 초기 HTML 응답이 늦어지고, 모바일에서 요소 렌더링 지연·SI로 드러남.
- **개선했을 때 (기대)**: 두 쿼리가 겹쳐 실행돼 서버 렌더 시간이 둘 중 **느린 쪽 하나** 수준으로 수렴 → TTFB·초기 렌더 도달 단축 → 모바일 LCP/SI 개선 방향.
- **그래서 택한 방법**: `page.tsx`에서 `getUserLikeReportStatus(postId)`를 **await하지 않고 promise만 먼저 시작**(`const likeStatusPromise = getUserLikeReportStatus(postId)`), 관련 매물 조회 블록을 그대로 지나간 뒤 필요한 지점에서 `const { userId, initialLiked, initialReported } = await likeStatusPromise`로 수확. 관련 매물 조회가 도는 동안 좋아요·신고 조회가 백그라운드에서 함께 진행됨.

> 변경 규모는 작지만(핵심 2줄) 서버 컴포넌트의 waterfall을 제거하는 것이 목적. 관련 매물이 없는 분기(`rental.location` 부재)에서도 promise를 미리 띄워두므로 손해가 없다.
