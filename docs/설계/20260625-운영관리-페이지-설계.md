# 운영관리 페이지 설계 문서

## Context

현재 Kanto 어드민(`/admin`)은 단일 `role = "admin"` 체계로 운영되며, 게시글·유저·신고·채팅 조회 등 기본 관리 기능만 존재한다.
기획 스펙에 따라 5개 중분류(공지 & 점검 / 콘텐츠 관리 / 통계 & 모니터링 / 권한 관리 / 감사 로그)를 추가 구현한다.
이 문서는 기존 패턴(Supabase, Server Actions, `createAdminClient`, `requireAdmin`)을 최대한 재사용하는 방향으로 설계한다.

## 1. 역할 체계 확장 (권한 관리 전제)

### 변경 내용

- `users.role` 값에 `"super_admin"` 추가 (기존 `"admin"` 그대로 유지)
- 새 테이블 `admin_permissions`: 어드민별 허용 기능 목록 저장

### 새 테이블: `admin_permissions`

```sql
create table admin_permissions (
  id          bigint generated always as identity primary key,
  admin_id    bigint references users(id) on delete cascade,
  permission  text not null,  -- 'delete_post' | 'handle_report' | 'sanction_user'
                               -- | 'write_notice' | 'view_stats' | 'view_chat'
  granted_by  bigint references users(id),
  created_at  timestamptz default now()
);
```

### 기존 코드 수정

- `requireAdmin()` 옆에 `requireSuperAdmin()` 추가 (`src/services/user/user.ts`)
- `requirePermission(permission: AdminPermission)` 추가 — 슈퍼어드민은 전체 통과, 어드민은 `admin_permissions` 조회
- 어드민 레이아웃: `super_admin` / `admin` 분기 처리, 사이드바 메뉴 조건부 렌더링

### 사이드바 메뉴 접근 규칙

| 메뉴 | super_admin | admin |
|------|:-----------:|:-----:|
| 공지 & 점검 | ✅ | 권한 있는 경우만 |
| 콘텐츠 관리 | ✅ | 권한 있는 경우만 |
| 통계 & 모니터링 | ✅ | 권한 있는 경우만 |
| 권한 관리 | ✅ | ❌ |
| 감사 로그 | ✅ | ❌ |

## 2. 공지 & 점검

### 2-1. DB 테이블

```sql
-- 전체 공지
create table system_notices (
  id          bigint generated always as identity primary key,
  title       text not null,
  created_by  bigint references users(id),
  starts_at   timestamptz not null,
  ends_at     timestamptz not null,       -- 라이프 사이클 종료 시각
  created_at  timestamptz default now()
);

-- 점검 일정
create table maintenance_windows (
  id             bigint generated always as identity primary key,
  starts_at      timestamptz not null,
  ends_at        timestamptz not null,
  blocked_pages  text[] not null default '{}',  -- ['all'] | ['write','chat', ...]
  created_by     bigint references users(id),
  created_at     timestamptz default now()
);
```

### 2-2. 헤더 공지 배너 컴포넌트

- **위치**: `src/components/common/header/NoticeBanner.tsx` (신규)
- 현재 시각 기준 `starts_at <= now() <= ends_at`인 공지만 표시
- 복수 공지: 슬라이더 방식 (`1/2`, `<`, `>`)
- 서버 컴포넌트에서 `system_notices` 조회 후 클라이언트 컴포넌트로 전달
- `DeletionPendingBanner` / `SuspendedBanner` 기존 패턴 재사용

### 2-3. 점검 차단 미들웨어

- **위치**: `src/middleware.ts` (기존 파일 수정)
- 점검 윈도우 활성 여부 + `blocked_pages` 확인 후 `/maintenance` 페이지로 리다이렉트
- Upstash Redis(기존 사용 중)에 현재 점검 설정 캐시 (TTL 60초) → DB 조회 비용 최소화

### 2-4. 공지 등록 시 전체 알림

- 공지 등록 서버 액션에서 `common_notifications` bulk insert
- `receiver_id` 기준으로 전체 유저 배치 삽입 (기존 알림 패턴 재사용)

### 2-5. 어드민 페이지

- `/admin/notices` — 공지 목록 / 등록 / 삭제
- `/admin/maintenance` — 점검 일정 설정 / 차단 페이지 선택

## 3. 콘텐츠 관리

### 3-1. DB 테이블

```sql
-- 금칙어 룰
create table profanity_rules (
  id          bigint generated always as identity primary key,
  scope       text not null,   -- 'chat' | 'post' | 'nickname'
  words       text[] not null, -- ['시발','병신']
  created_by  bigint references users(id),
  created_at  timestamptz default now(),
  updated_at  timestamptz default now()
);

-- 스팸 감지 설정 (싱글턴 — 항상 id=1 행을 upsert)
create table spam_config (
  id                    bigint primary key default 1,
  chat_window_sec       int default 3,     -- N초 내
  chat_max_count        int default 5,     -- N번 이상
  chat_cooldown_sec     int default 10,    -- N초 쿨다운
  max_urls_per_post     int default 3,
  profanity_strike_max  int default 3,
  report_strike_max     int default 5,
  auto_sanction_enabled boolean default false,
  updated_at            timestamptz default now()
);

-- 자동 제재 메시지 템플릿
create table sanction_message_templates (
  id          bigint generated always as identity primary key,
  trigger     text not null,   -- 'profanity' | 'spam' | 'report'
  title       text not null,
  body        text not null,
  created_by  bigint references users(id),
  created_at  timestamptz default now()
);
```

### 3-2. 금칙어 필터링

- 금칙어 등록 서버 액션: `profanity_rules` insert/update
- Redis에 scope별 금칙어 배열 캐시 (TTL 30초) → 채팅/게시글 write 시 검사
- 금칙어 등록 시 기존 게시물 검색: `posts` 테이블 full-text search(`title || content`)로 일치 목록 반환 → 어드민 UI에 표시

### 3-3. 스팸 감지 설정

- 채팅 도배 감지는 기존 그룹 채팅 서버(`src/services/go/groupChat.ts`) 수정
- `spam_config` 행을 Redis에 캐시 → 채팅 메시지 전송 핸들러에서 참조
- 어드민 UI: 슬라이더/숫자 입력으로 설정값 수정

### 3-4. 어드민 페이지

- `/admin/profanity` — 금칙어 룰 목록 / 등록 / 수정 / 삭제 + 영향받는 기존 게시물 표시
- `/admin/spam-config` — 스팸 감지 민감도 설정 + 자동 제재 토글 + 메시지 템플릿 관리

## 4. 통계 & 모니터링

### 4-1. 실시간 접속자 수

- 방식: Next.js Route Handler(`/api/admin/visitors`) — 브라우저 렌더링 기준
- 구현: 유저 레이아웃 마운트 시 `/api/admin/visitors/ping` POST, 언마운트 시 DELETE
- Redis SET(`visitors:{userId}`, TTL 60s)로 고유 접속자 집계
- 어드민 대시보드에서 30초 폴링으로 접속자 수 표시

### 4-2. DAU / MAU

- 기존 RPC `get_active_users_count(days)` 재사용
- DAU: `get_active_users_count(1)`, MAU: `get_active_users_count(30)`
- 신규 RPC 추가: `get_dau_series(year, month)` — 월별 일간 활성 유저 시계열
- 그래프: 기존 `signupValues` 차트 컴포넌트 확장, 연/월/일 탭 전환

### 4-3. 오류 로그 (Sentry 연동)

- `npm install @sentry/nextjs`
- `sentry.server.config.ts` / `sentry.client.config.ts` 설정
- 어드민 통계 페이지에 Sentry 대시보드 iframe embed 또는 링크 제공
- 에러 로그는 Sentry 외부 서비스 의존 — 별도 DB 테이블 불필요

### 4-4. 어드민 페이지

- `/admin` (기존 대시보드) — DAU/MAU 차트, 실시간 접속자 추가
- `/admin/monitoring` — Sentry 연동 링크 / 주요 오류 요약 (Sentry API 사용)

## 6. 감사 로그

### 6-1. DB 테이블

```sql
create table audit_logs (
  id          bigint generated always as identity primary key,
  actor_id    bigint references users(id),  -- 행위자 (어드민/슈퍼어드민)
  actor_role  text not null,               -- 'admin' | 'super_admin'
  action      text not null,               -- 'resolve_report' | 'delete_post' | ...
  target_type text,                        -- 'user' | 'post' | 'report' | 'notice' | ...
  target_id   bigint,
  detail      jsonb,                       -- 추가 컨텍스트 (이전값/이후값 등)
  created_at  timestamptz default now()
);
```

### 6-2. 로깅 방식

- 기존 모든 서버 액션(신고 처리, 제재, 게시글 삭제 등)에 `insertAuditLog()` 헬퍼 호출 추가
- `src/services/admin/auditLog.ts` (신규) — `insertAuditLog(actor, action, target?)` 함수

### 6-3. 접근 제어

- RLS 정책: `audit_logs`는 `role IN ('super_admin')` 인 경우만 SELECT 허용
- 레이아웃 레벨: 슈퍼어드민만 감사 로그 메뉴 노출
- `/admin/audit-logs` 진입 시 `requireSuperAdmin()` 서버 검증

> 결과적으로 라우트는 `/admin/notices`, `/admin/maintenance` 등 기존 패턴과 동일하게 유지된다.

## 9. 검증 방법

1. **공지 배너**: 공지 등록 후 일반 유저 헤더에 노출 확인, 라이프사이클 종료 후 자동 숨김 확인
2. **점검 차단**: 점검 윈도우 설정 후 지정 페이지 접근 시 `/maintenance` 리다이렉트 확인
3. **금칙어 필터**: 금칙어 등록 후 채팅/게시글 작성 시 차단 확인
4. **스팸 감지**: `spam_config` 변경 후 채팅 도배 테스트
5. **권한 체계**: 어드민 계정으로 슈퍼어드민 전용 메뉴 접근 불가 확인
6. **감사 로그**: 신고 처리 후 `audit_logs` 행 생성 확인
7. **DAU/MAU**: `get_dau_series` RPC 호출 결과 그래프 렌더링 확인
