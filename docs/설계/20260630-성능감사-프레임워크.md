## 1. 핵심 성능 지표 기준값

### Core Web Vitals (Google 권장)

| 지표 | 설명 | Good (목표) | Needs Improvement | Poor (위험) |
|------|------|------------|-------------------|------------|
| **LCP** | Largest Contentful Paint — 주요 콘텐츠 로드 시간 | ≤ 2.5s | 2.5s ~ 4s | > 4s |
| **INP** | Interaction to Next Paint — 사용자 입력 응답성 | ≤ 200ms | 200ms ~ 500ms | > 500ms |
| **CLS** | Cumulative Layout Shift — 레이아웃 이동 누적 | ≤ 0.1 | 0.1 ~ 0.25 | > 0.25 |
| **FCP** | First Contentful Paint — 첫 콘텐츠 표시 시간 | ≤ 1.8s | 1.8s ~ 3s | > 3s |
| **TTFB** | Time to First Byte — 서버 응답 시간 | ≤ 800ms | 800ms ~ 1.8s | > 1.8s |

> FID는 INP로 대체되었음 (2024년 기준). INP를 우선 측정.

### API / DB 응답 속도 기준

| 유형 | Good (목표) | Needs Improvement | Poor (위험) |
|------|------------|-------------------|------------|
| 목록 조회 (Supabase SELECT) | ≤ 200ms | 200ms ~ 500ms | > 500ms |
| 단건 조회 (상세 페이지) | ≤ 100ms | 100ms ~ 300ms | > 300ms |
| 쓰기 (INSERT / UPDATE) | ≤ 300ms | 300ms ~ 800ms | > 800ms |
| 이미지 업로드 (Storage) | ≤ 1s | 1s ~ 3s | > 3s |
| AI 챗봇 응답 (Cerebras) | ≤ 2s | 2s ~ 5s | > 5s |

## 2. 측정 도구 및 방법

### 2-1. 개발 중 (로컬)

#### Web Vitals 콘솔 리포터
- `WebVitalsReporter` 컴포넌트가 `app/layout.tsx`에 포함되어 있음
- 개발 환경(`NODE_ENV=development`)에서만 동작
- 브라우저 DevTools 콘솔에서 실시간 확인 가능

```
[CWV] LCP        1234ms  ✅ good
[CWV] CLS       0.023    ✅ good
[CWV] INP        180ms   ✅ good
[CWV] FCP        850ms   ✅ good
[CWV] TTFB       320ms   ✅ good
```

#### 번들 사이즈 분석
```bash
npm run analyze
```
- 빌드 완료 후 브라우저에서 번들 시각화 자동 열림
- `client.html` / `server.html` 두 뷰 제공
- 주의: 빌드 시간이 길어짐 (로컬 전용)

#### API 응답 시간 확인
- Chrome DevTools > Network 탭 > Fetch/XHR 필터
- `Timing` 항목에서 `Waiting for server response` 확인
- Next.js dev 서버 터미널에서 server component 렌더 시간 확인 가능

### 2-2. 운영 환경

#### Sentry Performance
- URL: Sentry 대시보드 > Performance 탭
- `tracesSampleRate: 1` 설정으로 100% 트랜잭션 수집 중
- LCP, FCP, TTFB, INP, CLS 자동 수집됨
- 페이지별 P75 / P95 응답 시간 확인 가능

#### Sentry 확인 항목
1. Performance > Web Vitals 탭 → 페이지별 점수 현황
2. Performance > Transactions 탭 → API 응답 느린 순 정렬
3. Issues > Performance 탭 → 자동 감지된 성능 이슈

### 2-3. Google Search Console
- URL: search.google.com/search-console
- Core Web Vitals 리포트 (필드 데이터, 실제 사용자 기준)
- 배포 후 ~28일 데이터 누적 필요

## 4. 주요 점검 항목

### 이미지
- [ ] `<img>` 태그 직접 사용 없이 `next/image` 사용 중인지 확인
- [ ] 목록 카드 이미지에 `priority` 없이 `loading="lazy"` 기본 동작 확인
- [ ] 히어로/LCP 이미지에 `priority` prop 추가되어 있는지 확인
- [ ] Supabase Storage 이미지 URL에 `width`, `quality` 파라미터 적용 여부

### JavaScript
- [ ] `npm run analyze` 후 500KB 이상 단일 청크 없는지 확인
- [ ] 동적 import (`import()`) 로 무거운 라이브러리(지도, 차트 등) 지연 로드 확인
- [ ] `"use client"` 범위를 필요한 최소 컴포넌트로 제한하고 있는지 확인

### 레이아웃 (CLS)
- [ ] 이미지 컴포넌트에 `width` / `height` 또는 `aspect-ratio` 지정 여부
- [ ] 폰트 로드 전후 레이아웃 이동 없는지 확인 (Geist 폰트는 `next/font` 사용 중 ✅)
- [ ] 공지사항 배너 표시/숨김 시 레이아웃 이동 없는지 확인

### Supabase 쿼리
- [ ] 목록 조회 시 필요한 컬럼만 SELECT하고 있는지 확인 (`*` 남용 없음)
- [ ] 자주 사용되는 필터 컬럼(barangay, region, category)에 인덱스 있는지 확인
- [ ] N+1 쿼리 없는지 확인 (Supabase join/embed 활용)
- [ ] Server Component에서 병렬 fetch 사용 중인지 확인 (`Promise.all`)

### 캐싱
- [ ] Next.js `fetch` 캐시 전략 확인 (정적 데이터는 `cache: 'force-cache'`)
- [ ] 자주 안 바뀌는 API (공지사항, 약관 등)에 revalidate 시간 설정 여부
- [ ] Notion 캐싱 (`terms-notion-caching.md` 설계 참고)

## 5. 성능 회귀 방지 가이드

### PR 머지 전 체크

```
□ 새 이미지 추가 시 next/image 사용
□ 새 페이지 추가 시 주요 이미지에 priority 여부 검토
□ 무거운 라이브러리 추가 시 npm run analyze 실행
□ Supabase 쿼리 추가 시 Promise.all 병렬화 고려
□ "use client" 경계를 최소화했는지 확인
```

### 성능 이슈 우선순위

| 우선순위 | 기준 |
|---------|------|
| P0 (즉시 수정) | LCP > 4s, CLS > 0.25, INP > 500ms |
| P1 (이번 스프린트) | LCP 2.5~4s, API 응답 > 500ms |
| P2 (백로그) | 번들 사이즈 최적화, 캐시 전략 개선 |
