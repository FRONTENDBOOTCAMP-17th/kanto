## 1. 프로젝트 개요

### 1-1. 목적

"칸토 go!"는 칸토(필리핀 거주 한인 대상 커뮤니티 플랫폼) 안에 새로 추가하는 **위치 기반 일회성 번개모임** 기능입니다.

- 사용자가 지도 위에서 주변에서 열리는 번개모임(맛집, 운동, 스터디 등)을 핀으로 발견하고 참여할 수 있다.
- 모임은 1회성이며 시작/종료 시각이 있고, 종료되면 지도에서 자동으로 사라진다(별도 cron 없이 쿼리 시점 필터로 처리).
- 그룹 채팅 등 부가 기능은 v2로 미루고, v1은 "발견 → 참여 → 어드민 모더레이션"까지만 다룬다.

### 1-2. 기술 스택

| 영역 | 기술 |
|---|---|
| 프레임워크 | Next.js 16.2.6 (App Router), React 19.2.4 |
| 언어 | TypeScript 5 |
| 스타일 | Tailwind CSS v4 |
| 상태관리 | Zustand 5 (`useAuthStore` 등) |
| i18n | next-intl 4 (`messages/ko.json`, `en.json`, `fil.json`) |
| DB/백엔드 | Supabase (Postgres + RLS + Realtime), `supabase gen types typescript`로 타입 자동생성 |
| 지도 | `@vis.gl/react-google-maps` 1.8.3 (Google Maps JS API + Geocoding API) |
| 인증 | Supabase Auth, 세션 사용자는 `getSessionUser()` (서비스 레이어) |

### 1-3. Supabase 클라이언트 3종 (이 프로젝트 고유 컨벤션)

| 클라이언트 | 위치 | 용도 |
|---|---|---|
| `createClient()` | `@/utils/supabase/server` | 서버 액션/서버 컴포넌트, RLS 적용됨 (현재 로그인 사용자 권한으로 동작) |
| `createAdminClient()` | `@/utils/supabase/admin` | 서버 전용, RLS 우회 — 어드민 조회/변이, 알림 insert 등에 사용 |
| `supabase` (싱글턴) | `@/lib/supabase` | 클라이언트 컴포넌트에서 직접 사용 (예: Realtime 구독) |

> ⚠️ **`@/utils/supabase/client`는 이 프로젝트에 존재하지 않는다.** 디자이너가 넘긴 코드가 종종 이 경로를 가정하고 있었는데, 매번 `@/lib/supabase`(클라이언트 컴포넌트) 또는 `@/utils/supabase/server`(서버 액션)로 고쳐야 했다.

### 2-2. 파일별 상세 역할

#### `supabase/migrations/20260623000000_kanto_go_meetups.sql`
- `meetups` 테이블: `post_id`(PK, `posts.id` FK, bigint) 1:1 확장 — 기존 게시글 시스템(`posts`)에 `meetup` 타입 게시글로 얹는 구조. `topic`(text), `start_at`/`end_at`(timestamptz), `location_lat/lng/address/detail`, `description`, `max_participants`(기본 6).
- `meetup_participants` 테이블: `meetup_post_id` + `user_id` 복합 유니크, `status`는 `joined`/`cancelled`.
- RLS: 조회는 전체 공개, 변이(insert/update/delete)는 게시글 작성자 본인만 (`my_user_id()` 헬퍼 사용, 기존 `user_blocks` 정책 패턴과 동일).
- `ALTER PUBLICATION supabase_realtime ADD TABLE ...` 로 두 테이블 모두 Realtime 발행 등록.
- **Supabase 대시보드 SQL Editor로 직접 실행**해서 적용함 (CLI `supabase db push`는 과거 마이그레이션 히스토리가 대시보드 작업으로 어긋나 있어 회피). 적용 후 `npm run gen:types`로 `src/type/supabase.ts`에 반영 확인됨.

#### `supabase/migrations/20260623010000_kanto_go_fix_participants_fkey.sql` (신규, 런타임 버그 수정)
- 원본 마이그레이션에서 `meetup_participants.meetup_post_id`가 `posts(id)`를 참조하도록 정의되어 있었음(데이터 정합성 자체는 문제 없음 — `meetups.post_id`가 `posts.id`의 1:1 FK라서). 그런데 PostgREST는 3개 테이블을 거치는 간접 관계를 추론하지 못해서, `meetups` 조회에 `meetup_participants`를 중첩 select 하는 모든 쿼리(`getActiveMeetups`, `adminGetMeetups`)가 `PGRST200`("Could not find a relationship between 'meetups' and 'meetup_participants'") 런타임 에러로 실패함.
- 결과적으로 모임이 **DB(`meetups` 테이블)에는 정상 생성**되지만 `/go` 지도와 `/admin/go` 어드민 목록 양쪽 모두에서 전혀 보이지 않는 증상으로 나타남(읽기 경로만 깨짐, 쓰기는 정상).
- FK를 `posts(id)` → `meetups(post_id)`로 교체해서 PostgREST가 직접 관계를 인식하도록 수정. 대시보드 SQL Editor로 적용, 적용 후 스키마 캐시 자동 갱신 확인.

#### `src/constants/meetupTopics.ts`
- 주제 7종(`food, sports, study, hobby, trip, social, other`)과 핀 색상(`pinColor`)을 단일 소스로 정의.
- `MeetupTopicKey` 타입, `TOPIC_META`(뱃지용 bg/color/border 포함 — `bg`는 `pinColor`에 알파값을 더한 연한 배경), `TOPIC_OPTIONS`(필터 칩용 `{value, label, color, border}`)를 파생.
- ⚠️ 디자이너 원본 zip의 키는 `culture`/`travel`이었으나, 이 프로젝트에서는 `hobby`/`trip`으로 명명했다(기획서는 라벨/색상만 확정했고 키 문자열은 미정이었음). 코드 전체가 이 파일을 단일 소스로 참조하므로 내부적으로는 완전히 일관됨.

#### `src/type/go.ts`
- `Meetup`: `Tables<"meetups">`를 베이스로 `posts`/`users` 조인 결과(`title`, `host_id`, `host_name`, `status`, `participant_count`)를 합성한 뷰 타입.
- `MeetupParticipant`: 참여자 상세(상세 패널용, `display_name`/`avatar_url` 포함).
- `CreateMeetupInput`: 생성 모달 폼 입력 (날짜+시간 분리, `createMeetup`에서 ISO로 조합).
- `AdminMeetup`: `Meetup`을 확장해 어드민 전용 필드(`host_initial`, 시간 기반 `status`, `participants`, `reports`, `created_at`) 추가.

#### `src/services/go/go.ts` ("use server")
사용자 API — RLS 적용된 `createClient()` 사용:
- `getActiveMeetups()`: `end_at > now()` AND `posts.status = 'active'` 필터로 진행 중 모임만 조회 (자동 만료 처리, cron 불필요).
- `getMeetupDetail(postId)`: 모임 1건 + `joined` 상태 참여자 목록.
- `createMeetup(input)`: `posts` 행 생성(`post_type: "meetup"`) → `meetups` 행 생성, 생성된 `post_id` 반환.
- `joinMeetup(meetupPostId)`: 참여자 insert + 주최자 본인이 아니면 `common_notifications`에 알림 insert(어드민 클라이언트로 RLS 우회).
- `cancelJoin(meetupPostId)`: 모임 시작 전에만 취소 가능(시작 후 취소 시도 시 에러).
- `hostEndMeetup(postId)` (신규): 주최자 본인이 자기 모임을 일찍 종료. RLS 세션 클라이언트(`createClient()`) 사용 — 어드민 강제종료(`adminForceEndMeetup`)와 달리 RLS를 우회하지 않음. `posts.user_id === sessionUser.id` 명시적 확인을 추가했는데, 이유는 Supabase `.update()`가 RLS에 의해 대상 행이 걸러져도 에러 없이 0건 영향으로 "성공" 처리되기 때문 — 명시적 체크 없이는 비호스트가 호출 시 혼란스러운 무동작 성공이 발생함(실제 권한 경계는 여전히 RLS).

어드민 API — RLS 우회 `createAdminClient()` 사용:
- `adminGetMeetups()`: 전체 모임 + 참여자 + (post별 `common_reports` 신고 건수 별도 집계 후 매핑) + 시간 기반 상태(`active`/`upcoming`/`ended`) 계산.
- `adminForceEndMeetup(postId)`: `posts.status = 'inactive'`로 변경(핀이 `getActiveMeetups` 쿼리에서 자동 제외됨).

> ⚠️ **런타임 버그 수정**: `posts` 테이블에 `users`로 가는 FK가 2개(`posts_user_id_fkey`, `posts_handled_by_fkey`)라서, `posts` 안에 `users(name)`을 중첩 select하는 모든 쿼리가 `PGRST201`(관계 모호함) 에러로 실패했음. `getActiveMeetups`/`getMeetupDetail`/`adminGetMeetups` 3곳 모두 `users(name)` → `users!posts_user_id_fkey(name)`으로 명시적으로 고쳐서 해결.

#### `src/hooks/go/useLiveMeetups.ts` ("use client")
- 마운트 시 `getActiveMeetups()` 1회 호출 + `topicFilter`로 클라이언트 필터링.
- `@/lib/supabase` 싱글턴으로 `meetups`/`meetup_participants` 테이블 Realtime 구독 → 변경 시 재조회(`load()` 재호출). 다른 사용자가 모임을 생성/참여해도 새로고침 없이 지도에 반영됨.
- `{ meetups, allMeetups, loading, error, refetch }` 반환 (`allMeetups`는 필터 적용 전 전체 목록, 현재 페이지에서는 미사용이지만 향후 카운트 등에 쓸 수 있음).

#### `src/hooks/go/useLocationPicker.ts` ("use client")
- 지도 클릭 이벤트(`MapMouseEvent`)에서 좌표를 받아 Google Geocoding API로 역지오코딩.
- `establishment`/`point_of_interest` 우선으로 주소를 고르고, 없으면 첫 결과 또는 좌표 문자열로 폴백.

#### `src/components/go/MeetupPin.tsx` ("use client")
- `AdvancedMarker`로 지도 위 핀 렌더링, 주제별 `pinColor` 적용. `mapId`가 있어야 `AdvancedMarker`가 렌더되므로 부모(`/go` 페이지)에서 `mapId`를 반드시 전달해야 함.

#### `src/components/go/TopicFilterChips.tsx` ("use client")
- 상단 오버레이의 가로 스크롤 주제 필터 칩. "전체" + `TOPIC_OPTIONS` 7종.

#### `src/components/go/MeetupCreateModal.tsx` ("use client")
- 미니 지도(줌 컨트롤 포함, `mapId` env로 분리) 클릭 → `useLocationPicker`로 주소 자동표시 → 제목/주제/날짜/시작·종료시간/한줄소개/정원 입력 → `createMeetup()` 호출 → `onCreated(postId: number)` 콜백.
- 필수값 검증에 `description`(한줄소개) 누락 체크 추가됨 — 디자이너 원본은 이 검증이 빠져 있어 DB `NOT NULL` 제약과 충돌할 수 있었음.
- **런타임 버그 수정**: 미니지도 마커에 `AdvancedMarker`의 기본 앵커(`BOTTOM`, `['50%','100%']`)와 중복되는 `transform: translate(-50%,-100%)`를 수동으로 추가하고 있었음 — SVG `viewBox`가 이미 바텀 중앙 기준으로 그려져 있어 이중 보정이 발생, 클릭 위치보다 핀이 좌상단으로 14px/36px 어긋나 보이는 증상이었음. 수동 transform 제거로 해결.

#### `src/components/go/MeetupDetailPanel.tsx` ("use client")
- 핀 클릭 시 우측에서 슬라이드 인. 일시/장소/주최자/참여자 아바타/정원 프로그레스바(70%↑ 주황, 90%↑ 빨강)/소개/참여·취소 버튼.
- **신고 버튼(Siren 아이콘) 연결됨** — 기존 `ReportModal`/`@/services/report`(`checkReported`, `submitReport`, `target_type: "post"`)를 재사용. 마운트 시 `checkReported`로 기존 신고 여부 확인 후 모달에 `initialReported`로 전달.
- `currentUserId`를 props로 받아 "내가 이미 참여했는지" 판단(참여 버튼 토글: 참여하기 ↔ 참여 취소).
- **호스트 분기 추가 (신규)**: `isHost = currentUserId === meetup.host_id`로 판단해, 본인 모임을 열었을 때는 하단 액션 영역이 참여 버튼이 아니라 "모임 종료하기" 버튼으로 바뀜. 클릭 시 어드민 강제종료(`/admin/go`)와 동일한 2단계 확인 UX(주황 `AlertTriangle` 경고 박스, "취소"/"종료 확인")를 톤만 다르게(중립 슬레이트 vs 어드민 레드) 재사용. 확인 시 `hostEndMeetup()` 호출 → 토스트 → 패널 닫힘 → Realtime으로 핀이 지도에서 자동 제거됨. 이미 종료된 모임(`status === "inactive"`)을 다시 열면 "종료된 모임입니다" 정적 메시지만 표시. 비호스트 화면은 변경 없음.

### 3-5. Realtime 동기화 버그 2건 수정 (생성/종료/참여/참여취소가 새로고침 없이 반영되도록)

실제 사용 테스트 중 받은 피드백:
1. "모임 종료하기" 후 핀이 다른 사용자 화면에서 즉시 사라지지 않음(새로고침해야 사라짐). 생성은 즉시 보였는데 종료만 안 됨.
2. 참여하기를 누르면 개설자 화면에서 참여자 인원수가 실시간으로 늘어나지 않음(1→2로 안 바뀌고 새로고침해야 반영). 단, 참여자 아바타/알림 자체는 실시간으로 보임.

**원인 분석**
- `useLiveMeetups`(`src/hooks/go/useLiveMeetups.ts`)는 `meetups`/`meetup_participants` 두 테이블만 Realtime 구독함. 그런데 `hostEndMeetup`/`adminForceEndMeetup`은 `posts.status`만 `inactive`로 바꾸고, `posts` 테이블은 `supabase_realtime` publication에 등록돼 있지 않음(`20260623000000_kanto_go_meetups.sql`에서 `meetups`/`meetup_participants`만 등록함) → 종료 이벤트를 클라이언트가 영원히 못 받음.
- `src/app/(user)/go/page.tsx`에서 핀 클릭 시 `selectedMeetup` state에 그 시점의 `Meetup` 객체를 그대로 스냅샷으로 저장하고 있었음. 이후 `useLiveMeetups`가 실시간으로 새 목록을 받아와도 `selectedMeetup`은 클릭 당시 데이터로 고정된 채 갱신되지 않아, 상세 패널(`MeetupDetailPanel`)에 전달되는 `participant_count`가 절대 바뀌지 않음.
- ⚠️ **1차 수정 시도였다가 되돌린 변경**: `posts` 테이블 UPDATE를 직접 구독하도록 `useLiveMeetups`에 `.on(...)`을 추가했었으나, `posts`가 publication에 없는 테이블이라 해당 바인딩이 채널 구독 핸드셰이크 자체를 깨뜨려 같은 채널에 묶여 있던 `meetups`/`meetup_participants` 이벤트(생성/참여)까지 전부 안 오게 되는 회귀가 발생함. 즉시 원복함 — **publication에 없는 테이블을 같은 channel에서 구독하면 그 channel 전체가 깨질 수 있다는 점**을 여기서 확인함.

**최종 수정**
- `src/hooks/go/useLiveMeetups.ts`: 구독 대상은 원래대로 `meetups`/`meetup_participants` 두 테이블만 유지(주석으로 `posts`를 구독하면 안 되는 이유 명시).
- `src/services/go/go.ts`: `hostEndMeetup`/`adminForceEndMeetup`이 `posts.status`를 바꾼 직후, publication에 등록된 `meetups` 행에 대해 `post_id = post_id`인 no-op `update`를 한 번 더 호출 → 값은 그대로지만 실제 UPDATE 문이 실행되어 WAL에 기록되고, 이미 구독 중인 채널이 이 이벤트를 받아 `getActiveMeetups()`를 재조회함 → 종료된 모임은 `posts.status='active'` 필터에서 자동 제외되어 핀이 새로고침 없이 사라짐.
- `src/app/(user)/go/page.tsx`: 선택 상태를 `Meetup` 객체 대신 `post_id`(`selectedMeetupId`)만 저장하도록 변경하고, 실제로 패널에 넘기는 `selectedMeetup`은 매 렌더마다 `meetups.find(m => m.post_id === selectedMeetupId)`로 실시간 목록에서 다시 찾아오도록 변경. 참여/참여취소로 `meetup_participants`가 바뀌어 목록이 재조회될 때마다 선택된 모임도 같이 최신화되어 참여자 수가 즉시 반영되고, 모임이 종료/삭제되어 목록에서 빠지면 패널도 자동으로 닫힘.
- `npx tsc --noEmit` 관련 파일 0 errors 확인. 브라우저 2탭(개설자/참여자)으로 직접 클릭 테스트는 사용자가 진행 후 "완료" 확인함(2026-06-23).

2. "번개모임 만들기" → 모달 미니지도에서 줌 후 핀 클릭 → 주소 자동표시 → 한줄소개 등 입력 → 저장 → 메인 지도에 주제색 핀 등장. — ✅ 확인됨 (마커 오프셋 버그 수정 후)
3. 다른 계정으로 같은 페이지를 열어 모임 생성/참여 시 새로고침 없이 핀·정원이 갱신되는지(Realtime). — ✅ 확인됨 (3-5절 참고. 생성/종료/참여/참여취소 4개 액션 모두 새로고침 없이 반영)
4. 핀 클릭 → 우측 패널에 제목·주제·일시·장소·주최자/참여자 아바타·정원 프로그레스·참여하기 버튼 정상 표시. — ✅ 확인됨
5. 다른 계정으로 참여 → 참여자 추가 + 주최자에게 알림 도착 + 정원 도달 시 참여 버튼 비활성 + 90%↑ 마감임박 색상 변경. — ⬜ 미확인
6. 모임 시작 전 참여 취소 → 자리 반환됨. 종료시각이 지난 모임은 핀이 사라짐(새로고침 없이도 다음 polling/realtime 트리거 시). — ⬜ 미확인
7. 상세 패널의 신고(Siren) 버튼 → `ReportModal`이 열리고 신고 제출이 정상 동작. — ⬜ 미확인
8. `/admin/go` → 통계 카드/필터/테이블 정상 표시, 행 클릭 시 Drawer, 강제종료 2단계 확인 후 모임이 실제로 종료되고(`posts.status = 'inactive'`) 지도에서 핀이 사라지는지. — ✅ 모임이 어드민 목록에 보이는 것까지는 확인됨, 강제종료 동작 자체는 미확인
9. (신규) 본인 모임 패널에서 "모임 종료하기" 2단계 확인 → 종료 후 토스트/패널 닫힘/지도에서 핀 제거, 그리고 `posts` owner UPDATE RLS가 실제로 이 동작을 막지 않는지. — ✅ 확인됨 (3-5절 Realtime 수정 후 핀 제거까지 확인됨)

### 4-3. 코드상 알아두면 좋은 미완성/의도적 보류 사항

- **검색바**: `/go` 페이지 상단 검색 입력창은 UI만 존재하고 실제 검색 로직은 없음(플레이스홀더만 표시). 기획서에도 검색 기능은 명시되어 있지 않았으므로 범위 밖으로 판단, 별도 요청 시 구현 필요.
- **`useLiveMeetups`의 `allMeetups`**: 필터 적용 전 전체 목록을 반환하지만 현재 `/go` 페이지에서는 사용하지 않음(예: 전체 모임 수 통계에 쓸 수 있음).
- **신고 알림**: 신고 제출 시 어드민에게 별도 알림이 가지는 않음(기존 `ReportModal`/`common_reports` 흐름 그대로 재사용 — 어드민은 `/admin/go` Drawer 또는 `/admin/reports`에서 직접 확인).
- **`/go` 지도가 한 화면에 안 들어오는 스크롤 불편함**: 사용자 요청으로 **의도적으로 미해결 상태**(3-3절 참고) — 재작업 요청 전까지 손대지 않음.
- **모임 수정하기(편집)**: 호스트가 자신의 모임 제목/시간/장소 등을 직접 수정하는 기능은 아직 없음(이번엔 "종료하기"만 구현, 3-4절 참고). 필요 시 `MeetupCreateModal`에 `initialData` prop을 추가해 생성/수정 겸용으로 만들고 `useLocationPicker`에 초기 위치 시드를 지원하는 방향으로 후속 작업 가능.

### 4-4. Phase 2 (v2) — 착수 전, 참고용

기획서에 명시된 v2 범위(현재 작업 범위 밖, 코드 없음). **그룹 채팅은 참여(Join) 시 자동 생성되도록 기획되어 있으나 아직 코드가 전혀 없음** — 2026-06-23에 "참여하면 채팅방이 생기느냐"는 질문이 있었고, 답은 "아니다, v2 범위"였음:
1. `meetup_chat_rooms` / `meetup_chat_messages` 테이블 신규 — 그룹 채팅.
2. 기존 1:1 채팅 Realtime 패턴(`src/hooks/chat/useChatRoomRealtime.ts`)을 그룹용으로 변형, 참여(Join) 시 채팅방 자동 생성.

4. (검토 단계) `/go/[id]` 형태의 모임 상세 페이지 — 콘텐츠 확장 시 고려.
