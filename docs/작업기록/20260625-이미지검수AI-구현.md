## 1. 프로젝트 개요

### 목적

Kanto는 게시글 작성(중고거래 `usedgoods` / 부동산 `rental` / 구인구직 `job`) 시 사용자가 이미지를 업로드한다. 기존 흐름은 **클라이언트 → Supabase Storage 직접 업로드**였고, 검증은 `accept="image/*"`(클라이언트 한정)와 개수 제한(최대 10장)뿐이었다. **이미지 내용에 대한 검증이 전혀 없어** 음란·폭력·혐오·불법 거래물 이미지가 그대로 공개될 수 있는 문제가 있었다.

이 기능은 **이미지를 선택하는 즉시 AI 비전 모델로 내용을 분석해, 부적합 이미지를 업로드 전에 차단**한다.

### 기술 스택 (이 기능에서 사용)

- **검수 AI**: Groq `meta-llama/llama-4-scout-17b-16e-instruct` (비전 가능 모델, 단일 제공자)
- **백엔드**: Next.js API Route (`src/app/api/moderate-image/route.ts`) — 별도 마이크로서비스 없이 Kanto 앱에 내장
- **레이트리밋**: Upstash Redis (`@upstash/redis`) — 기존 챗봇 라우트와 동일한 패턴 재사용
- **i18n**: next-intl (`ko`/`en`/`fil` 3개 로케일)
- **이미지 업로드/저장**: 기존 Supabase Storage 업로드 흐름은 그대로 유지(검수는 그 앞단에 추가됐을 뿐)

### 핵심 설계 결정 (왜 이렇게 됐는가 — 매우 중요, 시행착오 포함)

이 결정들은 사용자와의 대화를 통해 여러 차례 바뀌었다. 최종 상태만 보면 이해가 안 될 수 있어 변경 과정을 남긴다.

1. **독립 마이크로서비스 ❌ → Kanto API 라우트 내장 ✅**
   최초 설계 문서는 "타 사이트에서도 재사용 가능한 독립 HTTP 마이크로서비스(Hono+Docker)"를 제안했다. 하지만 이 레포는 모노레포가 아닌 **단일 Next.js 앱**이라 별도 배포 인프라가 과한 부담이었다. → MVP는 Kanto 라우트에 직접 내장하기로 결정. 추후 추출이 쉽도록 검수 코어 로직은 `src/lib/moderation/`에 모듈로 분리해뒀다.

2. **fail-open ❌ → fail-closed ✅**
   설계 문서는 "AI 실패 시 그냥 통과(fail-open)"였다. 사용자는 "미검수 이미지가 공개되는 게 싫다"는 이유로 거부했다. 다만 검수가 **이미지 선택 시점·1장 단위**로 일어나기 때문에, fail-closed로 가도 "폼 전체가 막히는 게 아니라 그 이미지 1장만 재시도 가능하게 거부"되는 구조라 비용이 낮다고 판단 → **fail-closed로 확정**.

3. **폴백 제공자: OpenAI omni-moderation ❌ → Groq 멀티모달 ✅**
   처음엔 "1차 Gemini, 2차 OpenAI omni-moderation"으로 설계했었다(서로 다른 실패 유형을 가져야 폴백 의미가 있다는 논리). 하지만:
   - OpenAI는 무료 플랜에서 **지출 한도($0)를 거는 기능 자체를 막아둠**("You must be on a paid plan to set project spending limits") → "카드는 등록하되 과금은 0으로 보장"이라는 전제가 깨짐.
   - 사용자가 **과금 가능성 자체를 원천 차단**하고 싶어함 → OpenAI 옵션 폐기.
   - 대안으로 Groq 콘솔에서 비전 모델 가용성을 실제 API 호출로 확인 → `meta-llama/llama-4-scout-17b-16e-instruct`가 이미지 입력 + `response_format: json_object` 구조화 출력을 모두 지원함을 검증. 기존 챗봇이 쓰던 `GROQ_API_KEY`를 그대로 재사용 가능(신규 가입/결제 불필요) → **Groq로 교체**.

4. **1차 Gemini + 2차 Groq ❌ → Groq 단일 ✅ (최종)**
   Groq로 교체한 뒤에도 한동안 "1차 Gemini(`gemini-2.5-flash`) + 2차 Groq 폴백" 구조로 운영했다. 그런데 실측 테스트 중 **Gemini 무료 티어가 `gemini-2.5-flash` 모델 기준 프로젝트당 하루 20회**로 매우 제한적이라는 걸 발견했고, 더 큰 문제로 **이 한도가 기존 챗봇의 `GEMINI_API_KEY`와 같은 키/프로젝트를 공유**해서 챗봇과 이미지 검수가 같은 할당량을 나눠 쓰는 상황이었다.
   - 해결 시도: 검수 전용 `GEMINI_MODERATION_API_KEY`를 새로 발급해 분리. 그런데 발급 과정에서 실수로 유료 등급(Tier 1·선불) 프로젝트에 키가 묶여 "선결제 크레딧 소진" 에러 발생 → 무료 등급 프로젝트(`gen-lang-client-...`)에 새로 키를 발급해 해결.
   - 이 과정에서 **Groq의 레이트리밋이 모델 단위로 완전히 독립적**이라는 걸 실측으로 확인했다(`llama-4-scout` 호출 2번 시 999→998로 감소, 동시에 `llama-3.3-70b-versatile`(챗봇 모델)는 영향 없이 999 그대로 — 같은 API 키를 써도 모델별 풀이 분리됨, 분당 1,000회 수준으로 매우 넉넉함).
   - 결론: **Gemini 없이 Groq 하나만으로도 처음부터 충분했다.** Gemini를 굳이 추가해서 얻는 "제공자 다양성"의 이점은 검증된 적이 없었던 반면(둘 다 "범용 LLM에 프롬프트로 점수를 뽑아내는" 방식이라 정확도 우위가 없음), 키 관리·할당량·결제 등급 confusion 같은 실질적 비용은 컸다. → **Gemini 관련 코드/키 전부 제거, Groq 단일 제공자로 최종 단순화.**

### 최종 흐름

```
[작성 폼] 이미지 선택(파일 input onChange)
  └ 파일별로 즉시 캔버스 다운스케일(≤1024px) → base64
        │
        ▼
POST /api/moderate-image   (Kanto 서버, 클라가 외부 AI를 직접 호출하지 않음)
  · Upstash 레이트리밋 (IP별 분당 30회)
  · Groq llama-4-scout 비전 모델 호출 (JSON 강제: adult/violence/hate/illegal 0~1 점수)
  · 정책 엔진: 임계값 비교 → allow / block
  · 실패(타임아웃 8초 또는 에러, 최대 3회 재시도 후) → fail-closed
        │
        ▼ 판정 결과 JSON
[클라이언트]
  · allowed=true  → 미리보기에 추가 (기존 Supabase 업로드 흐름은 폼 제출 시 그대로 동작)
  · allowed=false → 미리보기에 추가 안 함 + 하단 Toast로 일반 안내(구체 카테고리는 노출 안 함, 3초 후 자동 닫힘)
```

## 2. 폴더 구조 및 파일별 역할

### 신규 생성 파일

```
src/lib/moderation/
├── types.ts          # 공용 타입: ModerationCategory, CategoryScores, ModerationProvider, ModerationResult
├── policy.ts          # 정책 엔진: 카테고리별 임계값(adult 0.7 / violence 0.8 / hate 0.7 / illegal 0.7) 비교 → allow/block 판정
├── moderate.ts        # 오케스트레이션: Groq 호출(8초 타임아웃) → 실패 시 fail-closed 반환
└── providers/
    └── groq.ts        # 유일한 제공자. llama-4-scout 비전 모델 호출 + 503/429 재시도(최대 3회, 1.5s*n 백오프)

src/app/api/moderate-image/
└── route.ts           # POST 엔드포인트. Upstash 레이트리밋(IP별 분당 30회) + moderateImage() 호출 + 응답 조립

src/lib/moderateImage.ts  # 클라이언트 헬퍼: File → canvas 다운스케일(≤1024px, JPEG 0.85) → base64 →
                          # /api/moderate-image 호출 → { allowed, reason } 반환. 네트워크 실패도 fail-closed 처리
```

### 건드리지 않은 것 (의도적으로 그대로 둠)

- Supabase Storage 업로드 루프(각 폼의 `handleSubmit` 내부) — 검수는 그 앞단(파일 선택 시점)에만 끼워 넣었고, 업로드 로직 자체는 무수정.
- `src/app/(user)/profile/_components/profileAvatar.tsx` — `useImageUpload([], 1)`을 쓰지만 이번 통합 범위(작성 폼 3종)에는 포함하지 않음. 훅이 하위호환되게 바뀌었으므로 자동으로 검수가 적용되긴 하지만 별도로 검증하지는 않았음.

## 3. 완성된 기능 / 작업 현황

### ✅ 완료

- [x] 이미지 선택 시점(작성 완료 시점 아님) 1장 단위 검수, 중고거래·렌탈·구인 3개 폼 모두 적용
- [x] Groq `llama-4-scout` 비전 모델로 adult/violence/hate/illegal 4개 카테고리 0~1 점수 산출
- [x] 정책 엔진(`policy.ts`): 임계값 이상이면 차단, 사유(카테고리 한글 라벨) 조립 — 단 사용자에게는 구체 카테고리를 노출하지 않고 일반 메시지만 표시
- [x] fail-closed: Groq 실패/타임아웃(8초) 시 차단 + `degraded:true` + 서버 로그(`console.warn`). 이미지 원본은 어떤 경우에도 저장하지 않음
- [x] Upstash 레이트리밋(IP별 분당 30회, 키 prefix `moderate_image:`)
- [x] 다국어 안내 문구(ko/en/fil)
- [x] 검수 중 로딩 UI(`isChecking`)
- [x] 차단 사유 UX 개선 — `alert()`를 기존 `Toast` 컴포넌트로 교체(3개 폼 모두). 여전히 구체 카테고리는 노출하지 않고 일반 메시지만 표시(우회 악용 방지 목적은 유지)
- [x] **실제 API로 end-to-end 검증 완료**:
  - 정상 이미지 → `allowed:true, provider:"llama-4-scout"`
  - Groq 키 무효화 → `allowed:false, degraded:true, reason:"unavailable"` (fail-closed)
  - 레이트리밋 30회 도달 → `429 {"error":"rate_limit"}`
  - 정책 엔진 단위 테스트: 가짜 고점수(`adult:0.95`) 입력 시 `action:"block"` 정상 동작
  - 실제 사진(풍경·인물) 2장 → 오탐 없이 `allow` 처리 확인
  - Groq 레이트리밋이 챗봇 모델(`llama-3.3-70b-versatile`)과 검수 모델(`llama-4-scout`)에서 독립적으로 집계됨을 실측 확인 → 같은 `GROQ_API_KEY`를 써도 서로 충돌하지 않음
- [x] `npx tsc --noEmit`, `eslint` 통과(변경 파일 기준 에러 0건)

### ⛔ 미구현 / 의도적으로 범위 제외 (후속 과제)

- [ ] **제출 시점 2차 방어** — 현재는 선택 시점에만 검수한다. 개발자도구 등으로 클라이언트를 조작해 검수를 우회한 파일을 강제로 제출하면 현재 구조로는 막지 못한다.
- [ ] **fail-to-review(관리자 검토 큐)** — 지금은 fail-closed가 곧 "차단"이다. 검수 불가 상황을 관리자가 검토하는 별도 큐로 보내는 기능은 없음.
- [ ] **독립 마이크로서비스 추출** — 현재는 Kanto 앱에 내장. 타 사이트 재사용이 필요해지면 `src/lib/moderation/` 모듈을 별도 서비스로 추출 가능하도록 구조는 분리해둠.
- [ ] **임계값 튜닝** — 현재 임계값(`policy.ts`)은 설계 문서의 권장값을 그대로 사용 중이며, 실제 유해 이미지 샘플셋으로 정확도(오탐/미탐)를 검증한 적은 없음. AI(Groq llama-4-scout)가 "범용 LLM을 프롬프트로 검수에 맞춘 것"이라 전용 검수 모델 대비 정확도가 검증되지 않았다는 한계가 있음.
- [ ] **단일 제공자(Groq) 의존 리스크** — 현재 폴백 제공자가 없다. Groq 자체가 장애 나면 곧바로 fail-closed(차단)된다. 의도된 단순화이지만, 향후 트래픽이 커지면 재검토가 필요할 수 있다.

```
GROQ_API_KEY   # 기존 챗봇과 공유. 검수 전용 키 불필요(모델별로 레이트리밋이 분리되어 안전)
UPSTASH_REDIS_REST_URL / UPSTASH_REDIS_REST_TOKEN   # 기존 레이트리밋 인프라 재사용
```

### 핵심 파일 하나씩 열어볼 순서 (코드 리뷰 시)

1. `src/lib/moderation/types.ts` — 타입부터
2. `src/lib/moderation/providers/groq.ts` — 실제 AI 호출
3. `src/lib/moderation/policy.ts` — 차단 판정 로직
4. `src/lib/moderation/moderate.ts` — fail-closed 포함 오케스트레이션
5. `src/app/api/moderate-image/route.ts` — 레이트리밋 + 위 모듈 연결
6. `src/lib/moderateImage.ts` — 클라이언트에서 호출하는 진입점
7. `src/hooks/useImageUpload.ts` 또는 `CreateUsedGoodsForm.tsx` — 실제 사용처

### 동작 확인 방법 (수동)

```bash
# 1. dev 서버 실행 후 안전 이미지로 테스트
curl -X POST http://localhost:3000/api/moderate-image \
  -H "Content-Type: application/json" \
  -d '{"image":"data:image/png;base64,<base64>"}'
# → {"allowed":true,"action":"allow","provider":"llama-4-scout",...}

# 2. 브라우저에서 /usedgoods/create, /rental/create, /job/create 진입 →
#    사진 선택 시 "이미지 확인 중..." 스피너 확인 → 정상 사진은 미리보기에 추가됨
```
