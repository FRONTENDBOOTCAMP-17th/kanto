# 관리자 제재/해제 배너 실시간 반영 버그 — 문제 해결 정리

작성: `THLIMM`
일자: 2026-06-23

## 1. 증상 (관찰된 문제)

admin 권한(DB `users.role = 'admin'`) 유저는 다른 유저를 제재(suspend)하거나 제재를
해제할 수 있다. 그런데 제재 대상(피제재자)의 화면에서 다음 현상이 발생했다.

- admin A가 유저 B를 제재하면 **알림은 즉시** 뜨지만, 상단 빨간 **제재 배너는 즉시 뜨지
  않고 새로고침을 해야** 나타난다.
- 제재를 **해제**한 경우에도 빨간 배너가 즉시 사라지지 않고, 새로고침을 해야 사라진다.
- 더욱이 새로고침이 아니라 브라우저 **뒤로가기 / 앞으로가기** 버튼으로 페이지를 이동하면
  배너 상태가 전혀 갱신되지 않아, admin의 조치가 적용되었는지 즉시 알 수 없다.

즉 "알림은 실시간인데 배너는 실시간이 아니다" + "새로고침은 되는데 뒤로/앞으로가기는
안 된다"는 두 가지 단서가 있었다.

## 2. 원인 분석 (단서 추적)

### 2-1. 배너의 데이터 소스

제재 배너와 모달은 모두 **`useAuthStore`의 `user.suspended_until`** 값 하나만 본다.

- `src/components/common/SuspendedBanner.tsx` — `useAuthStore().user.suspended_until`
  으로 노출 여부 판단
- `src/hooks/useSuspended.ts` — 동일한 값으로 `isSuspended` 계산

그리고 `authStore`는 `src/hooks/useAuthInit.ts`에서 **`supabase.auth.onAuthStateChange`
이벤트가 발생할 때만** `users` 테이블을 다시 읽어 채운다. 이 이벤트는
로그인 / 로그아웃 / 토큰 갱신 / 최초 세션 로드 시에만 발생하며, **DB의 `users` 행이
바뀌는 것과는 무관**하다.

### 2-2. 왜 알림은 즉시 뜨고 배너는 안 뜨는가

제재 액션(`applySanction.ts`)과 해제 액션(`liftSanction.ts`)은 두 가지 쓰기를 한다.

1. `users.suspended_until` 컬럼 UPDATE
2. `common_notifications` 테이블에 `type='suspension'` 알림 INSERT

알림은 `src/hooks/useNotifications.ts`에서 `common_notifications` 테이블의
`postgres_changes`(INSERT/UPDATE)를 **realtime 구독**하기 때문에 즉시 화면에 반영된다.

반면 **`users` 테이블에는 현재 로그인 유저를 대상으로 한 realtime 구독이 전혀 없었다.**
그래서 admin이 `suspended_until`을 바꿔도 피제재자 클라이언트의 `authStore`는 stale
상태로 남아 배너가 갱신되지 않았다.

### 2-3. 왜 새로고침은 되고 뒤로/앞으로가기는 안 되는가

- **새로고침(F5)**: 앱이 완전히 재마운트 → `onAuthStateChange`의 `INITIAL_SESSION`이
  발생 → `users` 행을 새로 fetch → 최신 `suspended_until` 반영 → 배너 갱신.
- **뒤로/앞으로가기**: Next.js 클라이언트 라우팅(또는 bfcache 복원)이라 앱이 재마운트되지
  않고 `onAuthStateChange`도 다시 발생하지 않음 → `authStore`가 stale 그대로 → 배너
  변화 없음.

**결론(근본 원인):** `suspended_until`(users 테이블)에 대한 클라이언트 실시간 동기화가
없다. 배너는 stale한 `authStore`를 그대로 읽고 있었고, 이를 갱신하는 유일한 경로가
`onAuthStateChange`(주로 새로고침 시 발생)뿐이었다.

## 3. 해결 방향 검토

두 가지 접근을 비교했다.

| | (A) `users` 테이블 realtime 구독 | (B) 알림 realtime에 편승해 user refetch |
|---|---|---|
| 정확성 | 모든 users 행 변경 즉시 반영 | suspension 알림 이벤트에 반응 |
| 추가 작업 | `users`를 `supabase_realtime` publication에 추가하는 마이그레이션 필요 | 없음 (기존 동작 인프라 재사용) |
| 리스크 | users 행(이메일/전화/region 등) 변경이 broadcast → RLS·민감정보 노출 범위 확대 | 낮음 |
| 트리거 정합성 | 범용 | 제재/해제가 **항상** suspension 알림을 함께 insert → 1:1 대응 |

**(B)를 선택했다.** 이유:

1. 제재/해제 액션이 항상 suspension 알림을 함께 insert하므로, 우리가 원하는 트리거와
   정확히 1:1로 대응한다 — (B)의 단점인 "간접성"이 이 케이스에선 사실상 없다.
2. 이미 검증되어 동작 중인 알림 realtime 인프라를 재사용하므로 publication/RLS 추가
   설정이 불필요하다.
3. 최소 변경(surgical) — 새 publication·민감정보 broadcast 없이 기존 흐름에 refetch
   로직만 얹는다.

## 4. 구현

`src/hooks/useAuthInit.ts`에 두 번째 `useEffect`를 추가했다. 현재 로그인 유저(`userId`)
앞으로 들어오는 `common_notifications` INSERT를 구독하고, `type === 'suspension'`인
알림이 도착하면 `users` 행을 다시 읽어 `authStore.setUser`로 갱신한다.

```
src/hooks/useAuthInit.ts
├── USER_COLUMNS 상수 추출
│     — onAuthStateChange / suspension-sync 두 곳에서 동일한 컬럼 select 재사용
└── 2번째 useEffect: suspension 알림 → users refetch → setUser
      ├── channel: `suspension-sync:${userId}`
      ├── postgres_changes INSERT, filter: receiver_id=eq.${userId}
      ├── payload.new.type !== 'suspension' 이면 무시
      └── users 행(suspended_until 포함) refetch 후 setUser → 배너 즉시 갱신
```

이렇게 하면 제재 시 `suspended_until`이 채워져 배너가 즉시 뜨고, 해제 시 `null`로
바뀌어 배너가 즉시 사라진다. 새로고침/뒤로가기 여부와 무관하게 동작한다.
