# 콘텐츠 관리 페이지 리팩토링

**날짜:** 2026-06-26  
**대상:** `src/app/(admin)/admin/operation/content/`

---

## 배경

어드민 콘텐츠 관리 페이지(`page.tsx`)가 금칙어 관리와 스팸 관리 기능을 하나의 파일에 모두 담고 있었다. 기능이 추가될수록 파일이 비대해졌고, 관련 없는 상태와 핸들러가 뒤섞이면서 유지보수가 어려워졌다.

---

## 문제점

### 1. 단일 파일에 너무 많은 책임

`page.tsx` 한 파일이 아래 모두를 담당하고 있었다.

- 금칙어 룰 목록 조회 / 등록 / 수정 / 삭제
- 금칙어가 포함된 게시물 검색
- 스팸 감지 설정 조회 / 저장
- 제재 알림 템플릿 조회 / 수정
- 섹션 탭 전환 UI

결과적으로 `useState` 선언만 16개, 핸들러 함수 10개 이상이 한 파일에 존재했다.

### 2. 서버 상태와 UI 상태가 혼재

API에서 가져오는 데이터(서버 상태)와 사용자 인터랙션에 의한 상태(UI 상태)가 구분 없이 `useState`로 관리되고 있었다.

```ts
// 서버 상태 — API 응답 데이터
const [rules, setRules] = useState<ProfanityRule[]>([]);
const [rulesLoading, setRulesLoading] = useState(true);

// UI 상태 — 사용자 인터랙션
const [profTab, setProfTab] = useState<ProfTab>("list");
const [scopes, setScopes] = useState<Scope[]>([]);
```

이 두 가지를 같은 방식으로 관리하면 데이터 페치 로직이 복잡해지고, 등록/수정 후 목록을 다시 불러오는 코드를 직접 작성해야 했다.

### 3. 반복되는 fetch 보일러플레이트

서버 데이터 하나를 가져오려면 매번 같은 패턴을 작성해야 했다.

```ts
// 데이터 하나당 이 패턴이 반복됨
const [data, setData] = useState([]);
const [loading, setLoading] = useState(true);

const loadData = useCallback(async () => {
  try {
    const res = await fetch("/api/...");
    setData(await res.json());
  } finally {
    setLoading(false);
  }
}, []);

useEffect(() => {
  loadData();
}, [loadData]);
```

금칙어 룰, 스팸 설정, 제재 템플릿 세 가지 데이터에 이 패턴이 반복되었다.

### 4. effect 내부 setState 경고

`useEffect` 안에서 `loadRules()`를 호출하면, 함수 내부에서 `setRulesLoading(true)`가 동기적으로 실행되어 리액트 렌더링이 연쇄적으로 발생한다는 경고가 발생했다.

```
Error: Calling setState synchronously within an effect can trigger cascading renders
```

## 해결 방법

### 단계 1 — 컴포넌트 분리

`page.tsx`에서 섹션별로 컴포넌트를 분리했다.

```
page.tsx (780줄)
  ↓
page.tsx (65줄)          — 섹션 탭 + 라우팅만
ProfanitySection.tsx     — 금칙어 관리 전체
SpamSection.tsx          — 스팸 관리 전체
```

분리 기준: **부모가 자기 자신을 위해 사용하지 않는 상태는 자식에게 이전한다.**  
`page.tsx`가 들고 있던 `words`, `scopes`, `profTab` 등은 모두 `ProfanitySection`을 위한 것이었으므로 전부 내려보냈다.

### 단계 2 — TanStack Query 도입

서버 상태(API 데이터)를 `useQuery` / `useMutation`으로 분리했다.

**도입 이유:**

| 문제 | TanStack Query 해결 방식 |
|------|--------------------------|
| fetch 보일러플레이트 반복 | `useQuery` 하나로 로딩/데이터/에러 통합 |
| 등록 후 목록 재fetch 직접 호출 | `invalidateQueries`로 자동 갱신 |
| effect 내 setState 경고 | `useQuery`는 effect 없이 동작 |
| 탭 이동 시 불필요한 재요청 | `staleTime` 설정으로 캐싱 |

**변환 전후 비교 (룰 목록 조회):**

```ts
// 이전 — 5개 선언 필요
const [rules, setRules] = useState([]);
const [rulesLoading, setRulesLoading] = useState(true);
const loadRules = useCallback(async () => { ... }, []);
useEffect(() => { loadRules(); }, [loadRules]);

// 이후 — 1개
const { data: rules = [], isLoading: rulesLoading } = useQuery({
  queryKey: ["profanity-rules"],
  queryFn: async () => {
    const res = await fetch("/api/admin/profanity-rules");
    if (!res.ok) throw new Error();
    return res.json();
  },
});
```

**변환 전후 비교 (등록 후 목록 갱신):**

```ts
// 이전 — 저장 후 직접 재fetch
async function handleProfSubmit() {
  await fetch(url, { method, body });
  await loadRules(); // 직접 호출
  resetProfForm();
  setProfTab("list");
}

// 이후 — onSuccess 콜백
const submitMutation = useMutation({
  mutationFn: async (body) => { await fetch(url, { method, body }); },
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ["profanity-rules"] });
    onSuccess(); // 부모에게 완료 알림
  },
});
```

### 단계 3 — ProfanitySection 내부 추가 분리

`ProfanitySection`도 목록과 폼이라는 두 가지 역할을 동시에 담당하고 있어 추가로 분리했다.

```
ProfanitySection.tsx (280줄)
  ↓
ProfanitySection.tsx (55줄)   — 탭 + editingId 상태만
profanity/
  constants.ts                — SCOPE_OPTIONS
  ProfanityList.tsx           — 목록 조회/삭제
  ProfanityForm.tsx           — 등록/수정 폼 + 게시물 검색
```

**핵심 설계 결정 — `key` prop을 이용한 폼 초기화:**

목록에서 수정 버튼을 누르면 폼이 해당 룰 데이터로 초기화되어야 한다.
`useEffect`로 props 변경을 감지해 state를 초기화하는 대신, `key` prop을 이용해 컴포넌트를 새로 마운트시켰다.

```tsx
// ProfanitySection.tsx
<ProfanityForm
  key={editingId ?? "new"}   // ← 이 값이 바뀌면 ProfanityForm이 완전히 새로 마운트됨
  editingId={editingId}
  initialScopes={initialScopes}
  initialWords={initialWords}
  onSuccess={goToList}
  onCancel={goToList}
/>
```

```ts
// ProfanityForm.tsx — 마운트 시 props로 초기화, 이후 독립적으로 관리
const [scopes, setScopes] = useState<Scope[]>(initialScopes);
const [words, setWords] = useState<string[]>(initialWords);
```

이 방식은 `useEffect`로 props를 감지해 state를 리셋하는 것보다 단순하고, effect 내 setState 경고도 발생하지 않는다.

### 단계 4 — SpamSection 내부 추가 분리

`SpamSection`도 두 카드(설정, 템플릿)가 서로 다른 API와 상태를 가지므로 분리했다.

```
SpamSection.tsx (220줄)
  ↓
SpamSection.tsx (9줄)           — 두 카드를 나열하는 레이아웃만
spam/
  SpamConfigCard.tsx            — 감지 민감도 설정 + 저장
  SanctionTemplateList.tsx      — 제재 알림 템플릿 목록 + 인라인 수정
```

**SpamConfigCard의 서버값·로컬값 분리:**

스팸 설정은 사용자가 값을 수정한 뒤 저장 버튼을 눌러야 반영된다. 수정 중인 값(draft)과 서버에서 온 값(savedConfig)을 분리해 관리했다.

```ts
const { data: savedConfig } = useQuery({ queryKey: ["spam-config"], ... });
const [draft, setDraft] = useState<SpamConfig | null>(null);

// draft가 없으면 서버값을 표시. 저장 성공 시 draft를 null로 초기화 → 서버값으로 복귀.
const config = draft ?? savedConfig ?? DEFAULT_SPAM_CONFIG;
```

**SanctionTemplateList의 인라인 수정:**

템플릿은 목록 안에서 행 단위로 수정된다. `editingId`로 현재 수정 중인 항목을 추적하고, 저장 성공 시 `setQueryData`로 전체 목록을 재요청하지 않고 해당 항목만 캐시에서 교체했다.

```ts
onSuccess: (updated) => {
  queryClient.setQueryData<SanctionTemplate[]>(["sanction-templates"], (prev = []) =>
    prev.map((t) => (t.id === updated.id ? updated : t)),
  );
  setEditingId(null);
},
```

## 결과

### 파일 구조

```
content/
  page.tsx                           (65줄)   ← 780줄에서 축소
  _components/
    ProfanitySection.tsx             (55줄)   ← orchestrator
    profanity/
      constants.ts                   (8줄)
      ProfanityList.tsx              (90줄)
      ProfanityForm.tsx              (175줄)
    SpamSection.tsx                  (9줄)    ← orchestrator
    spam/
      SpamConfigCard.tsx             (145줄)
      SanctionTemplateList.tsx       (110줄)
```

### 상태 관리 구조

```
서버 상태  →  TanStack Query (useQuery / useMutation)
UI 상태    →  useState (탭, 폼 입력값, editingId 등)
```

### 개선된 점

- `useEffect` 내 setState 경고 제거
- 탭 이동 시 30초 이내 재요청 없음 (`staleTime: 30_000`)
- 각 파일의 역할이 명확해져 수정 위치를 즉시 파악 가능
- 폼 상태가 `ProfanityForm` 내부에 완전히 캡슐화됨
- 스팸 설정의 서버값·로컬 draft 분리로 저장 전 수정 상태 관리가 명확해짐
- 템플릿 저장 시 `setQueryData`로 캐시만 교체 (전체 목록 GET 재요청 절약, 낙관적 업데이트와 다름)
