## 2. 채팅방 진입 쿼리 통합 — `api/chat/[id]/route.ts`

**변경 전**

채팅방 진입 시 DB 왕복 3회 발생:
1. `getChatDetail` — chats + users + posts(title, post_type)
2. `posts` 별도 조회 — user_id, is_reserved
3. `used_goods` 별도 조회 — price

**변경 후**

`getChatDetail` 호출을 제거하고 route 내부에 단일 임베드 쿼리로 통합. 왕복 3회 → 1회.

```ts
supabase.from("chats").select(`
  *,
  user1:users!chats_user_id_1_fkey(id, name, avatar_url, created_at),
  user2:users!chats_user_id_2_fkey(id, name, avatar_url, created_at),
  posts(title, post_type, user_id, is_reserved, used_goods(price), rentals(price))
`)
```

- `post_type`에 따라 `used_goods.price` 또는 `rentals.price`를 분기 처리
- `getChatDetail` import 제거

## 3. 읽음 처리 단일화 + RPC 통합

**변경 전**

채팅방 입장 시 읽음 처리가 3곳에서 분산 실행:
1. `useChatMessages.ts` — `messages.is_read` 업데이트 (클라이언트 직접 쿼리)
2. `ChatRoomClient.tsx` 마운트 — `markChatReadAction` (chats.unread만 갱신)
3. `ChatRoomClient.tsx` 언마운트 — 동일 액션 중복 호출

messages와 chats.unread가 따로 처리되어 어긋날 수 있는 구조.

**변경 후**

Supabase RPC `mark_chat_read` 생성 — messages + chats.unread를 트랜잭션 안에서 한 번에 처리.

```sql
create or replace function mark_chat_read(p_chat_id bigint, p_user_id bigint)
returns void language sql security definer as $$
  update messages set is_read = true
    where chat_id = p_chat_id and sender_id <> p_user_id and is_read = false;
  update chats
    set user_id_1_unread = case when user_id_1 = p_user_id then 0 else user_id_1_unread end,
        user_id_2_unread = case when user_id_2 = p_user_id then 0 else user_id_2_unread end
    where id = p_chat_id;
$$;
```

- `markChatReadAction` — 기존 3쿼리(getUser + chat 조회 + unread 업데이트) → `rpc("mark_chat_read")` 1호출로 교체
- `ChatRoomClient.tsx` — 언마운트 중복 호출 제거, 마운트 시 1회만
- `useChatMessages.ts` — messages 직접 업데이트 useEffect 제거, `supabase` import 제거

**수정 파일**
- `src/components/common/chat/chatPanel/room/actions.ts`
- `src/components/common/chat/chatPanel/room/ChatRoomClient.tsx`
- `src/hooks/chat/useChatMessages.ts`

## 4. 메시지 SELECT 경량화 — `services/chat/message.ts`

**변경 전**

모든 메시지에 `transactions(*)` 조인 — 대부분의 메시지가 `null`임에도 매번 join 경로 평가.

**변경 후**

- `MESSAGE_SELECT`에서 `transactions(*)` 조인 제거, 필요한 컬럼만 명시
- `type === "payment"`이고 `transaction_id`가 있는 메시지에 한해 `transactions` 배치 조회 후 Map으로 매핑

```ts
const paymentIds = messages
  .filter((m) => m.type === "payment" && m.transaction_id != null)
  .map((m) => m.transaction_id as number);

if (paymentIds.length > 0) {
  const { data: transactions } = await supabase
    .from("transactions").select("*").in("id", paymentIds);
  // Map으로 messages에 붙임
}
```

## 5. Realtime 채팅 목록 구독 필터 — `useChatListRealtime.ts`

**변경 전**

- 필터 없이 `chats` 테이블 전체 변경 구독 → 내 채팅 아닌 이벤트도 클라이언트에 전달됨
- INSERT 시 `/api/chat/list` 전체 목록 재조회

**변경 후**

서버 사이드 필터로 2채널 분리 — 내 채팅만 수신.

```ts
// user_id_1인 채팅만
supabase.channel(`chat-list-user1-${currentUserId}`)
  .on("postgres_changes", { ..., filter: `user_id_1=eq.${currentUserId}` }, ...)

// user_id_2인 채팅만
supabase.channel(`chat-list-user2-${currentUserId}`)
  .on("postgres_changes", { ..., filter: `user_id_2=eq.${currentUserId}` }, ...)
```

INSERT 처리 개선:
- 기존: 전체 목록 재조회(`/api/chat/list`)
- 변경: `payload.new.id`로 해당 chat 1건만 조회(user/posts 조인 포함) 후 목록 앞에 추가

**수정 파일**
- `src/hooks/chat/useChatListRealtime.ts`
