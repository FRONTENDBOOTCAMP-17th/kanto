# 결제 보안 강화

## 작업 내용

### 1. Disbursement 중복 발생 방지 (🔴 긴급)
- **문제**: 구매자가 "수령 확인" 버튼을 빠르게 두 번 누르면 `confirmReceiptAction`이 동시 호출 → 두 요청 모두 `status='paid'` 확인 후 disbursement 호출 → 판매자에게 돈이 두 번 지급될 수 있음
- **해결**: `claimTransactionRelease()` 함수 추가
  - PostgreSQL의 atomic `UPDATE ... WHERE status='paid'` 활용
  - 첫 번째 요청만 row update 성공 (1 row), 이후 요청은 0 rows → null 반환 → throw
  - disbursement는 DB lock 획득 성공한 요청만 호출
- **변경 파일**:
  - `src/services/payment/transaction.ts`: `claimTransactionRelease()` 추가
  - `src/components/common/chat/chatPanel/room/paymentActions.ts`: `confirmReceiptAction` 수정

### 2. 중복 결제 요청 서버 검증 추가 (🔴 긴급)
- **문제**: UI에서 `paymentRequestBlocked`로 버튼을 막지만 서버 액션 직접 호출 시 뚫림 → 한 채팅방에 `pending`/`paid` 거래가 여러 개 생성 가능
- **해결**: `createPaymentRequestAction` 시작 시 `hasBlockingTransactionForChat()` 서버 레벨 검증 추가
- **변경 파일**: `src/components/common/chat/chatPanel/room/paymentActions.ts`

### 3. Disbursement 실패 시 released 처리 방지 (이전 커밋)
- released 업데이트 전에 disbursement 먼저 호출하도록 순서 변경
- disbursement 실패 시 throw → 상태 `paid` 유지, 구매자 재시도 가능

### 4. confirmReceiptAction 실행 흐름 (최종)
```
1. 구매자 본인 확인 + status=paid 확인
2. 판매자 계좌 정보 조회
3. claimTransactionRelease() → atomic DB update (status: paid→released)
   └ null 반환 시 → "이미 처리 중" throw (동시 요청 차단)
4. createDisbursement() → Xendit 정산 호출
5. xendit_disbursement_id DB 저장
6. 게시글 sold 처리
7. 시스템 메시지 발송
```
