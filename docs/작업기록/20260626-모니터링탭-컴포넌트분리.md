# 모니터링 탭 컴포넌트 분리

**날짜:** 2026-06-26  
**대상:** `src/app/(admin)/admin/operation/monitoring/_components/`

---

## 배경

`MonitoringTabContent.tsx` 한 파일이 통계·성능·오류 로그 세 탭의 UI와 각 탭에 필요한 상수·유틸 함수를 모두 담고 있어 355줄에 달했다. 탭 하나를 수정하려 해도 파일 전체를 읽어야 했고, 관련 없는 코드가 뒤섞여 유지보수가 어려웠다.

---

## 문제점

### 1. 단일 파일에 세 탭의 UI가 혼재

`MonitoringTabContent.tsx`가 아래를 모두 담당하고 있었다.

- 통계 탭: 가입자 트렌드 차트, 게시물 유형 분포, 지역별 게시물
- 성능 탭: Core Web Vitals 카드, 트랜잭션 상위 10개 테이블
- 오류 로그 탭: Sentry 이슈 목록

### 2. 상수·유틸 함수가 한 파일에 집중

탭마다 전용으로 쓰이는 상수와 유틸 함수가 섞여 있었다.

```ts
// 오류 로그 탭에서만 쓰임
const LEVEL_CONFIG = { ... };
function relativeTime(iso: string) { ... }

// 성능 탭에서만 쓰임
function vitalStatus(...) { ... }
const VITAL_STATUS_STYLE = { ... };

// 통계 탭에서만 쓰임
const CAT_COLORS: Record<string, string> = { ... };
```

---

## 해결 방법

### 탭별 컴포넌트 분리

각 탭을 독립 파일로 분리하고, 해당 탭에서만 쓰이는 상수·유틸은 각 파일 안에 직접 선언했다.

```
MonitoringTabContent.tsx (355줄)
  ↓
MonitoringTabContent.tsx (56줄)    — 탭 버튼 + 탭 상태 관리만
stats/StatsTab.tsx                 — 통계 탭 전체
perf/PerfTab.tsx                   — 성능 탭 전체
errors/ErrorsTab.tsx               — 오류 로그 탭 전체
```

**분리 기준:** 해당 탭에서만 쓰이는 것은 그 탭 파일이 소유한다.

| 컴포넌트 | 소유한 상수·유틸 |
|----------|-----------------|
| `StatsTab` | `CAT_COLORS` |
| `PerfTab` | `vitalStatus()`, `VITAL_STATUS_STYLE` |
| `ErrorsTab` | `LEVEL_CONFIG`, `relativeTime()` |

`MonitoringTabContent`는 `errorCount` 계산(탭 배지용)과 탭 스위칭만 담당한다.


### 확인 사항 — TanStack Query 불필요

`page.tsx`는 Next.js 서버 컴포넌트(async)로 `Promise.all`로 데이터를 한 번에 패치한 뒤 props로 전달한다. 클라이언트에서 추가 패치가 없으므로 TanStack Query 없이 충분하다.

```ts
// page.tsx — 서버에서 fetch 후 props 전달
const [stats, sentry, performance] = await Promise.all([
  getMonitoringStats(),
  getSentryIssues(),
  getSentryPerformance(),
]);
```
