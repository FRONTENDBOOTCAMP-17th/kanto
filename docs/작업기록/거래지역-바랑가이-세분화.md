## 1. 프로젝트 개요

### 1-1. 무엇을 만드는 프로젝트인가
**Kanto(칸토)** 는 **필리핀 거주 한인 커뮤니티**를 대상으로 한 마켓플레이스 웹앱이다.
한국 서비스가 아니라 **마닐라 메트로(Metro Manila) + 팜팡가(Pampanga)** 권역이 기준이다.

주요 카테고리:
- **중고거래(usedgoods)** — 중고 물품 거래
- **임차(rental)** — 숙소/방 임대
- **구인(job)** — 구인/구직
- **커뮤니티(community)** — 게시판
- **칸토 go(go)** — 위치 기반 번개모임 (지도 기반, 별도 기능)

### 1-2. 이번 작업의 목적
기존 "거래지역"은 `trade_location` enum **7개(메트로 구역 수준)** 로만 표현되어 너무 광범위했다.
구매자가 "이 사람이 내 근처에 있구나"를 판단하려면 한국의 **동(洞)** 에 해당하는
필리핀 **바랑가이(Barangay)** 수준까지 좁혀져야 한다.

| 한국 | 필리핀 | 기존 칸토 | 이번 작업 |
|------|--------|-----------|-----------|
| 시/도 | City/Municipality | — | (자동 도출) |
| 구 | 메트로 구역 | **`trade_location` enum** | 유지(1단계) |
| **동** | **Barangay** | 없음 | **추가(2단계)** |

**핵심 긴장과 해법**: 더 세분화하되 **건물 단위로 노출되면 안 된다**
(고가품 거래 시 "예약중" 표시 → 위치 특정 → 범죄 위험).
→ **프라이버시 클램프**: 위치는 지도/검색으로 고르되, **공개·저장은 바랑가이+시 수준까지만**.
정확한 만남 위치는 채팅으로 확정. 상세페이지 지도는 **정확한 핀이 아니라 대략적 원형 영역**만 표시.

### 1-3. 기술 스택
- **프레임워크**: Next.js 16.2.6 (App Router, 서버/클라이언트 컴포넌트), React, TypeScript
- **DB/인증**: Supabase (`@supabase/ssr`, `@supabase/supabase-js`) — 단일 **원격** 프로젝트
- **국제화**: `next-intl` (로케일 3종: `ko`, `en`, `fil` — `messages/*.json`)
- **지도**: `@vis.gl/react-google-maps` (Google Maps Platform + **Google Places Autocomplete**)
  - env: `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`, `NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID`
- **스타일**: Tailwind CSS, `class-variance-authority`, `clsx`, lucide-react 아이콘
- **기타**: Upstash(ratelimit/redis), Xendit(결제), 이미지 모더레이션(AI)

> 칸토 go에 Google Maps/Places가 **이미 연동**되어 있어, 이번 작업은 그 인프라를 재사용했다.

## 3. 데이터 모델 & 동작 흐름

### 3-1. 신규 컬럼 (used_goods / rentals 공통)
```
location_barangay  text              -- 예: "Fort Bonifacio" (없으면 NULL → 시 단위 폴백)
location_city      text              -- 예: "Taguig"
location_lat       double precision  -- 소수점 2자리로 반올림된 값만 저장(≈1.1km)
location_lng       double precision
```
> 기존 `location_type`(중고) / `location`(임차) 광역 enum, `location_custom`/`location_detail`
> 자유텍스트는 **그대로 유지**(하위호환 + 1단계 필터용). 신규 글은 자유텍스트를 NULL로 저장.

### 3-2. 글쓰기 → 저장 흐름 (프라이버시 클램프)
```
사용자가 PlaceAutocomplete에서 장소 검색·선택
  → Place.addressComponents에서 extractBarangayCity()로 바랑가이/시/주 추출
  → 폼의 buildLocationFields()에서:
      location_type(광역) = cityToTradeLocation(city, province)   // 자동 도출, 미매칭 시 "그 외 지역"
      location_barangay   = 추출한 바랑가이
      location_city       = 추출한 시
      location_lat/lng    = roundCoord(정확좌표)                   // ★ 소수점 2자리 반올림(클램프)
  → INSERT. ★ 전체 도로명 주소(formattedAddress)는 저장하지 않음.
```
**보안 포인트**: 정확 좌표를 저장하면 화면에서 흐리게 보여줘도 클라이언트가 API로 원본을 읽을 수 있음.
그래서 **저장 단계에서** 반올림 → 정확한 핀 복원 불가.

### 3-3. 2단계 필터 옵션 소스
- 바랑가이는 자유 검색이라 고정 목록이 없음 → **하드코딩 대신 동적 생성**.
- `getUsedGoodsBarangays()`/`getRentalBarangays()`가 **활성 매물이 있는 바랑가이만** 광역별로 묶어
  `Record<광역, 바랑가이[]>` 형태로 반환 → SearchBar 2단계 드롭다운 옵션.
- 광역(1단계) 선택 시에만 2단계 드롭다운 노출, 광역 변경 시 바랑가이 선택 초기화.

### 3-4. 표시 우선순위 (카드/상세)
```
location_barangay 또는 location_city 있으면 → formatBarangayLabel() ("Brgy. xxx, City")
없으면(레거시 글) → 기존 location_type/location 광역 텍스트 (지도는 숨김)
```

## 5. 앞으로 남은 작업

### 5-1. 실데이터 검증 (가장 중요, 미완료)
로컬에서 앱(`npm run dev`)을 띄워 **실제 마닐라 주소**로 글을 올려 확인:
1. **바랑가이 추출 신뢰도** — Google 필리핀 데이터에서 바랑가이가 `sublocality_level_1` /
   `administrative_area_level_3` / `neighborhood` 중 어디로 오는지 불일치 가능. **누락 케이스**에서
   시 단위 폴백이 잘 되는지 확인. (필요 시 `extractBarangayCity` 우선순위 보강)
2. **클램프 확인** — 저장된 `location_lat/lng`가 소수점 2자리로 반올림됐는지, 전체 주소 미저장인지
   DB 행으로 직접 확인.
3. **원형 반경 튜닝** — 현재 400m. "동네는 식별되되 건물은 특정 불가" 수준인지 보고 조정
   (`ApproxAreaMap`의 `radius` 기본값, 좌표 반올림 자릿수는 `roundCoord`).
4. **광역 자동 도출** — `cityToTradeLocation`의 키워드 매칭이 실제 city 값과 맞는지
   (특히 Pampanga 권역: Angeles/San Fernando/Mabalacat/Clark 등).

### 5-2. 구인(job) 카테고리
이번엔 의도적으로 제외(근무지라 성격이 다르고 확신 부족). 동일 패턴 적용 가능:
- `jobs` 테이블에 동일 컬럼 추가(마이그레이션) + 타입
- `CreateJobForm`(글쓰기), 상세, 카드, `JobFilters`에 `barangaysByLocation` 연결
- `services/job/job.ts`에 `barangay` 필터 + `getJobBarangays()` 추가

### 5-3. 기타 고려
- **임차 반경**: 사용자는 임차도 동일 클램프에 동의했으나, 숙소는 위치가 정확할수록 유용한 성격이라
  추후 임차만 반경/정밀도를 별도 조정하는 옵션 검토 가능.
- **타입 재생성**: 이후 스키마 변경 시 `npm run gen:types`로 `supabase.ts` 재생성
  (이번엔 SQL 에디터로 적용해 수동 반영했으므로, 다음 변경 때 일관성 점검).
- **다른 노출 지점**: 메인 페이지 인기목록/검색결과 등 다른 카드 렌더 위치는 이번 범위에 미포함.
  필요 시 동일하게 바랑가이 우선 표시로 통일.
- **마이그레이션 히스토리**: SQL 에디터로 직접 적용 → CLI 히스토리 테이블에 미기록.
  팀이 `supabase db push`로 관리한다면 히스토리 정합성 한 번 점검 권장(멱등이라 재적용은 안전).

- Supabase는 단일 **원격 공유 프로젝트**. 마이그레이션은 수동 적용(SQL 에디터 또는 `supabase db push`).

## 7. 후속 수정 (2026-06-28)

실데이터 검증 중 발견한 3가지 결함을 수정. 표시 컴포넌트/서비스/필터는 변경 없이,
**저장값과 선택 UX만 보강**해 카드·상세·2단계 필터가 자동으로 정상화되도록 함.

### 7-1. 지명 언어 영어 통일

**문제**: 자동완성 요청이 `language: "ko"`라, 구글에 한글 현지화명이 있는 지명만 한글로 와서
`location_barangay`/`location_city`에 한글·영어가 혼재 저장됨(예: 시는 `시티 오브 마닐라`,
바랑가이는 `Ermita`). 저장값이 곧 카드 라벨이자 검색바 2단계 필터 옵션이라 필터에도 한글이 노출됨.

**결정**: 지명은 고유명사 → **영어를 정본으로 저장·표시·필터 통일**. `Brgy.`·`전체지역` 등 UI 라벨은
기존 로케일 i18n 유지, 지명만 영어 고정.

**변경** (`src/components/go/PlaceAutocomplete.tsx`):
- 자동완성 후보 요청 `language: "ko"` → `"en"`.
- 선택 장소 상세 조회를 `prediction.toPlace()` 대신 `new placesLib.Place({ id, requestedLanguage: "en", requestedRegion: "ph" })`로 명시 생성 → `addressComponents`(바랑가이/시) 항상 영어 보장.
- 기존 한글 혼재 테스트글은 재선택 저장/삭제 후 재작성으로 정리(테스트 데이터라 마이그레이션 불필요).

### 7-2. 수정 화면 "현재 설정된 거래지역" 표시

**문제**: 글쓰기에선 선택 시 하단 연두색 바(전체 주소)로 선택 상태가 보이나, 수정 진입 시엔 `picked`가
`null`이고 프라이버시상 전체 주소를 저장하지 않아 연두색 바가 안 뜸. 오히려 "아직 위치가 선택되지
않았습니다"가 떠서 기존 거래지역이 풀린 것으로 오해.

**변경**:
- `PlaceAutocomplete.tsx`: optional prop `fallbackLabel` 추가 → 새 선택 없으면 `현재 설정된 거래지역: <공개 라벨>` 연두색 바 표시(미선택 문구 대체). go 모임 생성은 prop 미전달이라 기존 동작 유지.
- `src/components/common/LocationPicker.tsx`: `fallbackLabel` 전달 + 안내문 라벨 중복 제거(공개 라벨은 새 선택 시에만, 프라이버시 노트는 위치 있으면 항상).

### 7-3. 시 성분 없는 장소(랜드마크 등) 최선 폴백

**문제**: Manila Bay 같은 자연지물·랜드마크는 주소 컴포넌트에 `locality`/`sublocality`가 없어
`barangay=null, city=null` → 광역 "그 외 지역", 표시 로직의 레거시 자유텍스트도 신규 글엔 `null`이라
**카드/상세 라벨이 빈 값**이 됨.

**결정**: 최선 폴백 허용(엄격 차단 X). 좌표는 이미 클램프되어 프라이버시 안전.

**변경**: 표시 로직은 `location_city`만 차 있으면 usedgoods·rental 모두 정상화되므로 **저장 시 city 폴백 체인** 적용.
- `src/type/go.ts`: `PickedLocation`에 `displayName` 추가.
- `PlaceAutocomplete.tsx`: `onSelect`에 `displayName: place.displayName ?? null` 포함.
- `CreateUsedGoodsForm.tsx` / `RentalCreateForm.tsx`의 `buildLocationFields`(새 선택 분기):
  `location_city = picked.city ?? picked.province ?? picked.displayName ?? picked.address ?? null`.
  광역 매핑(`cityToTradeLocation`)·좌표 클램프는 그대로 — 폴백은 표시용 `location_city`에만.
- 결과: Manila Bay → `location_city="Manila Bay"`, 카드/상세에 표시, 광역은 "그 외 지역" 그룹.
  2단계 바랑가이 필터는 `location_barangay`(여전히 null) 기준이라 영향 없음.

**검증**: `tsc --noEmit` + 변경 파일 ESLint 통과. 실데이터(npm run dev) 동작 확인은 별도 진행.
