# 채팅 유저 신고/차단 기능 — 작업 정리

브랜치: `feat/user-report-block`
작성: `THLIMM`

## 1. 프로젝트 개요

**kanto**는 필리핀 생활 정보를 연결하는 마켓 플랫폼이다. 중고거래
(usedgoods), 구인구직(jobs), 방렌트(rental), 1:1 채팅, 안전결제
(Xendit 연동) 등의 기능을 제공한다.

**기술 스택**

- **프레임워크**: Next.js 16(App Router) + React 19
- **언어**: TypeScript
- **스타일**: Tailwind CSS 4 (CSS 모듈/styled-components 미사용, 전부 유틸리티 클래스)
- **상태관리**: Zustand (`useAuthStore` 등)
- **백엔드/DB**: Supabase (Postgres + Auth + Realtime), `@supabase/ssr`로 서버/브라우저
  클라이언트 분리, RLS(Row Level Security) 적용
- **국제화**: `next-intl` (ko/en/ta 3개 로케일, `messages/*.json`)
- **결제**: Xendit (안전결제 에스크로)
- **UI 컴포넌트**: shadcn 기반 `components/ui/*` (Select, Button, Textarea 등)

### 핵심 데이터 모델

- **`common_reports`**: 게시글/유저 공용 신고 테이블. `target_type`이 `"post"` 또는
  `"user"`, `target_id`는 게시글 id 또는 유저 id. 신규 테이블 없이 기존 테이블을
  재사용했다.
- **`user_blocks`**(신규): `blocker_id`(차단한 유저) / `blocked_id`(차단당한 유저) /
  `created_at`. `(blocker_id, blocked_id)` 유니크 제약으로 중복 차단 방지. RLS는
  `blocker_id = my_user_id()`인 행만 SELECT/INSERT/DELETE 가능 — 즉 **차단당한 쪽은
  자기 세션으로 이 테이블을 조회할 수 없다.** 그래서 차단 여부를 검사하는 로직은
  반드시 `createAdminClient()`(RLS 우회)를 사용해야 한다.

## 3. 지금까지 완성된 기능/작업

### 3-1. 채팅 내 유저 신고

- 기존 게시글 신고(`InteractionButtons` → `ReportModal` → `services/report.ts` →
  `common_reports`) 플로우를 그대로 재사용.
- `ReportModal`에 `categories`/`targetType` prop을 추가해 일반화. 게시글 신고는
  기존 6종 카테고리 + `target_type: "post"`로 동작 그대로 유지.
- 채팅방 헤더의 "신고하기" 버튼에 유저 신고용 카테고리 5종
  (`욕설/비방`, `성희롱/성적 불쾌감`, `사기/금전 요구`, `도배/광고/스팸`,
  `부적절한 행위`)과 `target_type: "user"`로 `ReportModal`을 연결.
- ko/en/ta 3개 로케일에 신규 카테고리 번역 키 추가.

### 3-2. 채팅 유저 차단

- 차단 관련 DB가 전혀 없던 상태였으므로 `user_blocks` 테이블/RLS를 신규 마이그레이션
  으로 추가.
- 채팅방 헤더 "차단하기" 클릭 → 확인 모달("OO님을 차단하시겠어요?") → 확인 시
  `blockUserAction`이 `user_blocks`에 기록하고, 기존 `leaveChatAction`과 동일하게
  해당 채팅방에서 나가기 처리.
- 동일 상대 재차단 시 `UNIQUE` 제약으로 중복 row가 쌓이지 않음(upsert 처리).

### 3-3. 차단된 유저 간 메시지 송수신 차단

처음 구현 시 두 가지 버그가 있었고, 순서대로 수정했다.

1. **RLS로 인해 차단 검증이 항상 실패하던 버그**: `isBlockedPair`가 일반 서버
   클라이언트(요청자 세션 기준 RLS 적용)로 `user_blocks`를 조회했는데, RLS 정책상
   `blocker_id = my_user_id()`인 행만 보이기 때문에 **차단당한 쪽이 보낼 때는
   차단 row 자체가 보이지 않아** 검증이 무력화됐다. → `isBlockedPair`를
   `createAdminClient()`(RLS 우회)로 변경해 해결.
2. **메시지가 잠깐 보였다가 사라지는 UX 버그**: 기존엔 메시지를 먼저 낙관적으로
   화면에 그린 뒤 서버 액션 내부에서 차단 여부를 검사해, 차단 시 잠깐 표시되었다가
   삭제되는 깜빡임이 있었다. → `actions.ts`에 `checkBlockedAction`을 추가하고,
   `ChatRoomClient.tsx`의 `handleSend`에서 **낙관적 메시지를 추가하기 전에** 먼저
   차단 여부를 확인하도록 순서를 변경. 차단 상태면 메시지를 화면에 그리지 않고
   즉시 `Toast`로 안내만 표시.

최종적으로 `sendMessageAction`(기존 채팅방)과 `createChatAndSendAction`(신규 채팅방
생성 + 첫 메시지) 양쪽 모두에서, 그리고 **양방향**(차단한 사람 ↔ 차단당한 사람
모두) 메시지 전송이 막힌다.

### 검증 완료 항목

- `npx tsc --noEmit` 통과 (프로젝트에 기존부터 있던 `nodemailer` 타입 누락 에러는
  이번 작업과 무관하며 그대로 남아 있음).
- `npx eslint`로 변경 파일 전체 통과.
- `messages/*.json` 3개 파일 모두 JSON 파싱 정상.
- DB 마이그레이션은 사용자가 직접 Supabase에 적용 후 `npm run gen:types`로 타입
  재생성 완료(원격 프로젝트 직접 조작은 에이전트가 수행하지 않음).

### 남아있는 범위 외 작업 (미구현)

- 차단 목록 조회 UI(`ProfileBlockedSection.tsx`)는 기존에 정적 placeholder로만
  존재하며, 이번 작업에서 `user_blocks` 데이터와 연동하지 않았다.
- 차단 시 게시글/프로필 등 채팅 외 영역에서 상대를 숨기는 기능은 범위에 포함하지
  않았다(채팅 메시지 송수신 차단까지만 구현).
