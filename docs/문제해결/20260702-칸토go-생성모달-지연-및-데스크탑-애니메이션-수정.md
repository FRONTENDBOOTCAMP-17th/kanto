## 이슈 1. 모바일 생성 모달 지연 (~0.4초+)

### 진단 과정

React `<Profiler>`로 `MeetupCreateModal` 서브트리를 감싸 마운트 비용을 측정했다.

| 단계 | phase | 소요 시간 |
|---|---|---|
| 최초 마운트 | mount | 70~95ms |
| 연쇄 리렌더 1 | nested-update | ~300ms |
| 연쇄 리렌더 2 | nested-update | ~200ms |
| 연쇄 리렌더 3 | nested-update | ~160ms |

클릭 후 첫 페인트까지 **총 600~700ms(dev 기준)** 블로킹. `nested-update`는 layout effect에서 발생한 동기 setState가 페인트 전에 리렌더를 유발했다는 의미다.

용의자 배제 실험:
- 중첩 `APIProvider` 제거 → 변화 없음
- `PlaceAutocomplete` 렌더 자체를 제거 → 변화 없음
- **`ResponsiveSelect`를 조건부 마운트로 변경 → 연쇄 리렌더 완전 소멸** ✅

### 원인

`src/components/ui/responsive-select.tsx`가 **모바일 버튼과 데스크탑 Radix Select를 둘 다 마운트하고 CSS(`md:hidden` / `hidden md:flex`)로만 숨기는 구조**였다. 생성 모달에는 `ResponsiveSelect`가 7개(월/일/시작 시·분/종료 시·분/최대 인원) 들어가므로, 모바일에서 쓰지도 않는 Radix Select 7개가 전부 마운트되며 각각의 layout effect가 연쇄 리렌더를 일으켰다.

### 수정

`matchMedia` 기반 `useIsMobile` 훅을 추가하고 화면 크기에 맞는 한쪽만 마운트하도록 변경.

```tsx
// responsive-select.tsx
function useIsMobile() {
  return useSyncExternalStore(
    subscribeMobile,                                      // matchMedia change 구독
    () => window.matchMedia("(max-width: 767px)").matches, // 클라이언트 스냅샷
    () => false,                                          // 서버 스냅샷 (SSR/하이드레이션 안전)
  );
}

// 렌더: 양쪽 마운트 + CSS 숨김 → 조건부 마운트
{isMobile ? <button .../> : <Select ...>...</Select>}
```

`useSyncExternalStore`를 쓴 이유: SSR 시 서버 스냅샷(`false`)으로 렌더해 하이드레이션 불일치가 없고, 하이드레이션 직후 실제 값으로 동기화된다. 클라이언트에서만 마운트되는 모달 경로에서는 첫 렌더부터 정확한 값을 쓴다.

### 결과

| | 수정 전 | 수정 후 |
|---|---|---|
| 마운트 | 70~95ms | **17ms** |
| 연쇄 리렌더 (nested-update) | 3회, 합계 ~660ms | **0회** |
| 클릭 → 모달 표시 (로그인 상태 실측) | 400~700ms | **~50ms** |

같은 컴포넌트를 쓰는 구인 작성 페이지(`CreateJobFormPage1`)도 동일하게 개선된다.

### 함께 반영한 개선

- `MeetupCreateModal` 내부의 중복 중첩 `APIProvider` 제거 — `GoClient` 최상위에 이미 동일 키의 `APIProvider`가 있어 컨텍스트가 전달됨
- `MeetupPin`/`ClusterPin`을 `React.memo`로 감싸고, `GoClient`의 `handlePinClick`을 `useCallback`으로 고정 + 인라인 화살표 함수 제거 — 모달 열기 등 지도와 무관한 상태 변경 시 마커 리렌더 차단

## 이슈 2. 데스크탑 모달이 좌상단에 떴다가 중앙으로 점프

### 원인: Tailwind v4의 translate 속성과 keyframe transform의 이중 적용

모달 패널은 `left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2`로 중앙 정렬한다. **Tailwind v4에서 `-translate-*` 유틸리티는 `transform`이 아니라 별도의 `translate` CSS 속성**(`translate: -50% -50%`)을 생성한다.

그런데 `modalPopIn` 키프레임이 `transform: translate(-50%, -50%) scale(...)`을 포함하고 있어, 애니메이션 0.3초 동안 두 속성이 **합성되어 이동량이 -100%, -100%로 이중 적용**됐다. 그래서 모달이 좌상단 사분면(목록 패널의 + 버튼 근처)에 나타났다가, 애니메이션 종료로 transform이 사라지는 순간 중앙으로 점프했다.

실측(1280×800, 애니메이션 중간 시점):

```
translate 속성:  -50% -50%                            (Tailwind 유틸리티, 상시)
transform:       matrix(0.975, 0, 0, 0.975, -280, -360)  (키프레임, -280px = 폭의 -50%)
→ 모달 중심 (352, 40) — 중앙 (632, 400)에서 한참 벗어남
```

### 수정

키프레임에서 translate를 제거하고 scale/opacity만 애니메이션. 센터링은 상시 적용되는 `translate` 속성이 유지한다.

```css
/* globals.css — 수정 전 */
@keyframes modalPopIn {
  from { opacity: 0; transform: translate(-50%, -50%) scale(.95); }
  to   { opacity: 1; transform: translate(-50%, -50%) scale(1); }
}

/* 수정 후 */
@keyframes modalPopIn {
  from { opacity: 0; transform: scale(.95); }
  to   { opacity: 1; transform: scale(1); }
}
/* modalPopOut 동일하게 수정 */
```

수정 후 실측: 애니메이션 중간 모달 중심 = 최종 위치와 동일(중앙 고정), scale만 변화. 모바일 `slideInUp`/`slideOutDown`은 `max-md:translate-x-0 max-md:translate-y-0`(translate 0)과 합성되므로 영향 없음.

### 교훈

> Tailwind **v4**부터 `translate-*`, `rotate-*`, `scale-*` 유틸리티는 `transform`이 아닌 개별 CSS 속성(`translate`/`rotate`/`scale`)을 쓴다. 이 유틸리티로 위치를 잡은 요소에 `transform`을 애니메이션하는 keyframe을 적용하면 **두 값이 합성되어 이중 적용**된다. keyframe에는 유틸리티가 담당하지 않는 속성만 넣을 것.
