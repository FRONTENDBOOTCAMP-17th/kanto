# Kanto 코드 리뷰 — 2026-06-23

오늘은 평소처럼 화면 단위로 보는 대신, **API가 실제로 쓰는 데이터베이스 테이블(ERD)을 한 번 정리**하고 그 위에서 **느려질 만한 쿼리**를 집중적으로 봤습니다. 팀에서 채팅이 느리다는 이야기가 있었다고 해서, 채팅 쪽은 실제 Supabase에 같은 쿼리를 직접 날려 응답 시간까지 재봤습니다. 평소 리뷰처럼 정확성·보안·가독성도 함께 봤고요.

> 참고: 이 조의 `supabase/migrations/`에는 `chats`/`messages`/`posts`/`users` 같은 핵심 테이블의 최초 `CREATE TABLE`이 없고, 콘솔에서 직접 만든 뒤 일부 변경(ALTER·트리거·신규 테이블)만 마이그레이션으로 남아 있습니다. 그래서 "이 컬럼에 인덱스가 있다/없다"는 마이그레이션·자동생성 타입(`src/type/supabase.ts`) 기준으로만 적었습니다. 실제 인덱스는 Supabase 대시보드의 Database → Indexes 또는 `select * from pg_indexes where schemaname='public';` 한 번으로 확정할 수 있으니, 아래 인덱스 관련 지적은 그걸로 한 번 확인해 주세요.

## 1. API가 쓰는 테이블 정리 (ERD)

코드의 모든 `.from('...')`과 마이그레이션을 훑어 실제 사용 중인 테이블을 정리했습니다.

**채팅 도메인**
- `chats` — `id`, `user_id_1`, `user_id_2`(FK→users), `post_id`(FK→posts), `last_message_at`, `last_message_content`, 그리고 비정규화 컬럼 `user_id_1_unread`/`user_id_2_unread`, `user_id_1_left`/`user_id_2_left`, `user_id_1_active`/`user_id_2_active`
- `messages` — `id`, `chat_id`(FK→chats), `sender_id`(FK→users), `content`, `is_read`, `type`(text/system/payment), `post_id`·`transaction_id`(nullable FK), `created_at`
- `transactions` — `id`, `chat_id`, `post_id`, `buyer_id`, `seller_id`, `amount`, `status`, 결제(xendit) 컬럼들

**게시글 도메인**
- `posts` — 공통 부모. `id`, `user_id`, `post_type`(used_goods/rental/jobs), `status`, `title`, `view_count`, `like_count`, `is_reserved`, `is_sold`, `created_at`
- `used_goods` / `rentals` / `jobs` — 각각 `post_id`로 posts와 1:1, 도메인 컬럼(price, room_type, category 등)을 가짐

**그 외**
- `users`, `common_notifications`, `common_likes`, `common_reports`, `user_sanctions`(`report_id` FK), `user_blocks`(`blocker_id`/`blocked_id`, UNIQUE)

**관계의 핵심 한 가지**: 게시글이 `posts`(공통) + `used_goods`/`rentals`/`jobs`(자식)로 나뉘어 있어서, 목록·상세를 보여줄 때 항상 부모·자식을 조인해야 합니다. 이 구조 자체는 합리적인데, 아래 3번에서 보듯 **필터를 자식 컬럼으로 거는 순간 쿼리가 까다로워지는** 원인이기도 합니다.

## 2. 채팅이 왜 느린가 — 실측 포함 (최우선)

결론부터 말하면, 지금 데이터 양(메시지 842행, 가장 큰 방이 252행)에서는 **쿼리 한 방의 CPU 비용보다 "한 화면을 그리려고 서버에 몇 번 왕복하느냐"가 체감 속도를 좌우**합니다. 실제로 이 조의 Supabase에 같은 쿼리를 날려보니 PostgREST 왕복 한 번이 평균 110~120ms 나오더군요(측정 위치 → Supabase 네트워크 지연 포함). 그러니 직렬 왕복이 한 번 늘 때마다 체감 지연도 그만큼 더 붙는 구조라 생각합니다.

### [필수] 채팅방 하나 열 때 직렬 왕복이 4~5번 쌓입니다 — `src/app/api/chat/[id]/route.ts:29-51`
채팅방에 들어가면 대략 이 순서로 실행됩니다: `auth.getUser()` → `users` 조회 → (`getChatDetail` + `getMessageList` 병렬) → `posts` 조회 → `used_goods` 조회. 앞의 둘과 마지막 둘이 **직렬**입니다. 제가 실제로 재보니 `users`·`posts`·`used_goods` 세 번만 더해도 약 351ms가 순수 왕복으로 쌓였습니다.

문제는, `posts`와 `used_goods`가 사실 `getChatDetail`이 이미 가져오는 `chats → posts(...)` 임베드에 끼워서 한 번에 받을 수 있는 정보라는 점입니다. 그런데 별도 쿼리로 떼어 놓다 보니 왕복이 2번 더 늘었네요. PostgREST의 임베드(`select=*, posts(*, used_goods(*))`)로 합치면 이 두 번을 0번으로 줄일 수 있습니다.

### [필수] 입장할 때 안읽음 처리가 두 군데서 중복으로 일어납니다 — `src/hooks/useChatMessages.ts:30-40` + `ChatRoomClient.tsx:175-181`
방에 들어가면 (1) `useChatMessages`가 `messages`의 `is_read=true`를 다건 UPDATE하고, 동시에 (2) `ChatRoomClient`가 마운트/언마운트 **양쪽**에서 `markChatReadAction`을 호출합니다. 그런데 `markChatReadAction`은 호출될 때마다 내부에서 `auth.getUser` → `users` → `chats` 조회 → `chats` UPDATE를 또 합니다(`actions.ts:147-173`, 4왕복). 한 번 입장에 markChatRead가 2번 = 8왕복이 더 붙는 셈이고, 거기에 messages UPDATE와 chats UPDATE가 같은 "안읽음=0" 결과를 **중복으로** 만들고 있습니다.

읽음 처리는 한 경로로 모으는 게 좋습니다. 마운트 시 한 번만 호출하고(언마운트 호출 제거), 가능하면 messages의 is_read 갱신과 chats의 unread=0을 하나의 RPC로 묶으면 왕복도 줄고 둘 사이 불일치 구간도 사라집니다.

### [제안] 메시지 50건을 가져올 때 항상 무거운 select를 씁니다 — `src/services/chat/message.ts:4-30`
지금은 `MESSAGE_SELECT = '*, sender:users(...), transaction:transactions(*)'`로 메시지 전 컬럼 + 작성자 + 트랜잭션을 매번 가져옵니다. 제가 252행짜리 방(chat 1)에서 50건을 재보니 이 무거운 select가 평균 128ms, 컬럼을 명시한 가벼운 select가 116ms였습니다. 지금은 차이가 크지 않지만(데이터가 적어서), 거의 모든 메시지가 `transaction_id IS NULL`인데도 `transactions(*)` 조인 경로를 매번 평가하는 건 메시지가 쌓일수록 손해입니다. 필요한 컬럼만 적고(`id,content,sender_id,is_read,created_at,type`), transaction은 payment 타입 메시지에서만 따로 끌어오는 편이 깔끔합니다.

### [제안] 메시지 정렬·커서가 인덱스에 기대고 있습니다 — `message.ts:18-24`
메시지 조회는 `chat_id` 필터 + `created_at` 역순 정렬 + `limit(50)` + `before` 커서(`lt('created_at')`) 구조입니다. 페이지네이션 설계로는 좋습니다. 다만 **`messages(chat_id, created_at)` 복합 인덱스가 있어야** 이 설계가 제값을 합니다. 인덱스가 없으면 Postgres는 chat_id로 거른 행을 다 모은 뒤 created_at로 메모리 정렬(Sort)을 하고 나서 50개를 잘라내거든요. 그러면 방에 메시지가 수천 개일 때 LIMIT 50의 이점이 사라지고 정렬 비용이 선형으로 늘어납니다. 반대로 인덱스가 있으면 인덱스를 역방향으로 50개만 읽고 멈춥니다. 위 입장 시 is_read 다건 UPDATE(`useChatMessages.ts:33`)도 같은 인덱스에 기대므로, 이 복합 인덱스 하나가 채팅 성능 전체에 영향을 줍니다. 꼭 확인해 주세요.

### [제안] 채팅 목록 Realtime 구독에 필터가 없습니다 — `src/hooks/chat/useChatListRealtime.ts:16,31`
`chats` 테이블의 INSERT/UPDATE를 필터 없이 구독하고 있어서, **DB의 모든 사용자 채팅 변경**이 이 클라이언트로 내려온 뒤 JS에서 내 것이 아니면 버리는 구조입니다(`:19-23`). 사용자가 늘수록 무관한 이벤트가 그만큼 쏟아지겠죠. 게다가 INSERT가 오면 무조건 `/api/chat/list` 전체를 다시 부르고(`:25`), UPDATE도 목록에 없는 방이면 또 전체를 다시 부릅니다(`:51`). 새 메시지가 활발해지면 무거운 목록 쿼리(아래)가 반복 호출되는 거고요. Supabase Realtime은 `filter: 'user_id_1=eq.<나>'` 같은 서버 측 필터를 걸 수 있으니, 내 방 변경만 받도록 좁혀 주세요.

### [제안] 채팅 목록 쿼리가 무겁고 일부 필터를 JS에서 합니다 — `src/services/chat/chat.ts:4-28`
`CHAT_SELECT='*, user1(...), user2(...), posts(...)'` + `.or('user_id_1.eq.X,user_id_2.eq.X')` + `last_message_at` 정렬입니다. `or()`는 두 컬럼을 한 인덱스로 못 타서(`user_id_1`, `user_id_2`가 따로면 BitmapOr가 필요) 비용이 더 듭니다. 그리고 떠난 방을 거르는 `user_id_N_left` 필터를 **DB가 아니라 JS에서** 하고 있어서(`:24-27`), 떠난 방까지 다 받아온 뒤 버립니다. 떠난 방 필터는 쿼리 조건으로 내리고, `last_message_at` 정렬 인덱스도 함께 확인해 주세요. 제가 이 목록 쿼리를 직접 재보니 114ms였는데, 데이터가 적은 지금도 무거운 select라 그렇습니다.

## 3. 그 외 느려질 만한 쿼리

### [필수] 메인 인기글이 3개 도메인 전체를 limit 없이 가져옵니다 — `src/services/main/main.ts:5-12`
`getPopularList`가 중고·렌탈·구인 목록 서비스를 **필터·limit 없이** 모두 호출하고, 각 서비스는 `status='active'` 전체를 `*, 자식(*), users(...)`로 가져옵니다(`usedGoods.ts:23-29` 등). 지금은 글이 적어 괜찮지만, 게시글이 쌓이면 메인 페이지 한 번 여는 데 세 테이블을 통째로 끌어오게 됩니다. 메인에서 쓸 만큼만(예: 각 6개) `order + limit`으로 DB에서 잘라 받아 주세요.

### [제안] 목록 필터를 전부 가져온 뒤 JS에서 거릅니다 — `usedGoods.ts:47-56`, `rental.ts:63-71`, `job.ts:38-47`
카테고리·지역·방 타입 같은 필터를 전체 조회 후 JS `filter()`로 적용하고 있습니다. 거를 행을 네트워크로 다 보낸 뒤 버리는 셈이라 데이터가 늘수록 손해입니다. 자식 테이블 컬럼이라 까다롭긴 한데, `자식!inner(...)` 임베드 + `.eq()`로 DB까지 내릴 수 있습니다.

### [제안] 검색이 `ilike('%term%')`라 인덱스를 못 탑니다 — `usedGoods.ts:38-40` 등
앞에 `%`가 붙은 패턴은 B-tree 인덱스를 쓸 수 없어 항상 전체 스캔입니다. B-tree는 "왼쪽부터 정렬된 값"으로 찾는데 앞에 와일드카드가 있으면 시작점을 못 잡거든요. 지금 규모에선 괜찮지만, 검색을 본격적으로 쓸 거면 `pg_trgm` GIN 인덱스나 전문검색(tsvector)을 검토해 주세요.

## 4. ERD로 본 비즈니스 관점 점검

### [제안] 거래 상태가 세 군데로 흩어져 있습니다 — `posts.is_reserved` + `posts.is_sold` + `transactions.status`
예약/판매완료 상태가 `posts.is_reserved`, `posts.is_sold`, 그리고 `transactions.status`로 세 곳에 나뉘어 있습니다. `toggleReserveAction`은 `posts`만 바꾸고 transactions와 연동되지 않아서(`room/toggleReserveAction.ts`), 시간이 지나면 "거래는 완료인데 글은 예약중" 같은 어긋남이 생기기 쉽습니다. 가능하면 거래 상태 하나(예: `posts.trade_status` enum)로 모으고 나머지는 그것을 따라가게 하는 편이 안전합니다.

### [제안] 차단(block)이 채팅 목록·기존 대화에는 반영되지 않습니다 — `actions.ts:65,113`
차단 검사가 **메시지 전송 시점**에만 들어갑니다(`isBlockedPair`). 그래서 차단을 해도 채팅 목록(`chat.ts`)에는 그 사람과의 방이 그대로 남고, 기존 메시지도 다 보이는 상태입니다. "차단했는데 대화가 남아 있다"는 건 사용자 입장에선 어색하니, 목록 조회와 메시지 조회에도 차단 관계를 반영할지 팀에서 한번 정해 보면 좋겠습니다. 더해서 `user_blocks`에 `blocked_id` 단독 인덱스가 없어(`20260618120000_user_blocks.sql`) "나를 차단한 사람" 방향 조회는 인덱스를 못 탑니다.

### [제안] 외래키 ON DELETE 정책이 안 보입니다 — 고아 레코드 주의
마이그레이션에 `chats`/`messages`/`posts`/`users` 사이의 ON DELETE가 없다 보니, 사용자가 탈퇴하거나 글이 지워질 때 채팅방·메시지가 어떻게 정리되는지 정의가 불분명합니다. `messages.sender_id`는 NOT NULL이라 작성자가 사라지면 삭제가 막히거나 고아 레코드가 남게 되고요. 탈퇴/삭제 시나리오를 한 번 정리해서 FK 정책(CASCADE / SET NULL)을 정해 두면 좋겠습니다.

## 6. 팀 공통

- **인덱스 점검을 한 번 하고 가면 좋겠습니다.** 오늘 [제안]의 상당수가 "인덱스가 있으면 괜찮고 없으면 느려진다"입니다. `select * from pg_indexes where schemaname='public';` 한 번으로 `messages(chat_id, created_at)`, `chats.last_message_at`, `common_notifications(receiver_id, created_at)`, `user_blocks.blocked_id`가 있는지 확인하고, 없으면 추가해 주세요. 이건 코드 변경 없이 효과가 큽니다.
- **"한 화면 = 몇 왕복?"을 습관으로 들이면 좋겠습니다.** 채팅방 진입처럼 여러 쿼리가 직렬로 붙는 곳은, PostgREST 임베드로 합칠 수 있는지부터 봅니다. 왕복 한 번이 100ms대라는 걸 오늘 실측으로 확인했으니, 왕복 수를 줄이는 게 가장 직접적인 속도 개선이라 생각합니다.
- 어제까지의 시연 블로커(관리자 인가 등)는 오늘 주제와 별개라 따로 다루지 않았습니다. 그쪽은 지난 회차에서 반영된 걸로 보고 있고, 다음 회차에 다시 화면으로 확인하겠습니다.
