## 2. 다시 안 깨지게 — 머지 전 tsc/build 게이트 (근본 해법)

빌드가 여섯 라운드째 깨지는 건 PR이 tsc를 통과하지 않은 채 develop에 들어오기 때문입니다. 사람이 매번 기억해서 막을 순 없으니 자동으로 막읍시다. `.github/workflows/ci.yml` 한 파일이면 됩니다.

```yaml
name: CI
on:
  pull_request:
    branches: [develop]
jobs:
  typecheck-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npx tsc --noEmit       # 타입 통과 못 하면 여기서 실패
      - run: npm run build          # 프로덕션 빌드까지 확인
```

그다음 GitHub 저장소 **Settings → Branches → develop 보호 규칙**에서 "Require status checks to pass before merging"을 켜고 위 `typecheck-build`를 필수로 지정하세요. 이러면 tsc/build가 깨진 PR은 머지 버튼 자체가 비활성화됩니다. 이게 켜지는 순간 1장 같은 에러는 develop에 못 들어옵니다.

> CI가 macOS 로컬보다 더 잘 잡습니다. 로컬(mac)은 통과해도 Linux(CI·배포)에선 대소문자·경로 차이로 깨질 수 있거든요.

추가로, 5차에 걸어둔 "`@/lib/supabaseServer` import 금지" ESLint 룰이 새 파일(rental create 등)을 못 잡았습니다. 룰 설정의 적용 범위를 확인하세요.

```js
// eslint.config 에서 no-restricted-imports 가 src 전체에 걸리는지 확인
// files 패턴이 특정 폴더로 좁혀져 있으면 새 라우트가 빠져나갑니다
```

CI에 `npx eslint .` 한 줄을 더 넣으면 이런 import 위반도 머지 전에 잡힙니다.

## 3. redis 로그인 라우트 — 지연 초기화 + 만료 시간

### 3-1. [제안] env 없을 때 우아하게

지금은 `api/login/route.ts:6-9`에서 모듈 로드 시점에 `new Redis({ url: process.env.X! })`를 실행합니다. env가 없으면 라우트를 import하는 순간 죽어요. 함수 안에서 만들고, 키가 없으면 차단만 건너뛰도록 하면 키 없는 환경에서도 로그인 자체는 동작합니다.

```ts
function getRedis() {
  const url = process.env.UPSTASH_REDIS_REST_URL;
  const token = process.env.UPSTASH_REDIS_REST_TOKEN;
  if (!url || !token) return null;       // 키 없으면 차단 비활성
  return new Redis({ url, token });
```

### 3-2. [사소] 만료 시간 의도 맞추기

커밋 메시지엔 "5분"인데 코드는 `redis.expire(key, 60 * 15)` = 15분입니다(`:37`). 그리고 실패할 때마다 `expire`를 다시 불러서 차단 창이 매번 갱신(슬라이딩)돼요. 첫 실패 시각부터 고정된 창을 원한다면 이렇게 씁니다.

```ts
// 첫 실패에만 TTL을 걸고, 이후엔 incr만
const n = await redis.incr(key);
if (n === 1) await redis.expire(key, 60 * 5);  // 의도가 5분이면 5, 15분이면 15
```

## 4. users(anon) RLS — 콘솔에서 확인 후 닫기

문서 두 곳(`docs/20260611-리뷰_개선_할당.md:34` ✅완료 / `docs/20260611-리뷰_개선_김도혁.md:244` ❌미반영)이 서로 다른 말을 합니다. 먼저 실제 상태를 확인해서 문서를 일치시켜 주세요.

### 4-1. 지금 노출되는지 직접 확인

anon(publishable) 키로 users를 그냥 읽어봅니다. 남의 행이 읽히면 구멍입니다.

```bash
curl "https://<프로젝트>.supabase.co/rest/v1/users?select=id,name,phone,auth_id&limit=5" \
  -H "apikey: <ANON_KEY>" \
  -H "Authorization: Bearer <ANON_KEY>"
# 다른 사람의 phone/auth_id 가 나오면 = 막혀야 함
```

### 4-2. 콘솔에서 정책 확인·적용

Supabase Dashboard → Authentication → Policies → `users`에서 RLS가 켜져 있고 SELECT 정책이 "본인 행만"인지 봅니다. 민감 컬럼(phone, auth_id)은 컬럼 단위로 막을 수 있어요.

```sql
-- 본인 행만 읽기
create policy "users select own"
on users for select to authenticated
using (auth_id = (select auth.uid()));

-- anon 의 민감 컬럼 차단이 필요하면 컬럼 권한으로
revoke select (phone, auth_id) on users from anon;
```

> 어드민 대시보드는 service-role(`createAdminClient`)로 RLS를 의도적으로 우회하니, RLS를 닫아도 어드민 통계는 영향받지 않습니다(역할 게이트 뒤에서만 우회하니까요). 그래서 anon SELECT는 안심하고 좁혀도 됩니다.

적용했으면 어떤 정책인지 한 줄만 알려주세요. 다음 회차에 anon 키로 라이브 확인하고 이 항목을 닫겠습니다.
