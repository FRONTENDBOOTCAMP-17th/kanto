## 1. 관리자 서버 액션에 권한 검사 붙이기 (`go.ts`)

### 왜 필요한가

Next.js의 `"use server"` 함수는 화면(레이아웃)을 거치지 않고도 호출할 수 있는 독립 엔드포인트라서, `(admin)/layout.tsx`에서 `role === "admin"`을 확인했더라도 그건 "관리자 화면을 그릴지"만 막아 줄 뿐 서버 액션 자체를 막아 주지는 못합니다. 지금은 로그인만 한 일반 회원이 `adminForceEndMeetup(postId)`를 직접 불러 아무 글이나 비활성화할 수 있는 상태라서, **민감한 액션은 함수 안에서 직접 권한을 다시 확인**해 줘야 합니다.

### 1단계 — `requireAdmin()` 헬퍼 한 번 만들기

지금 코드베이스엔 `requireAdmin`이 없으니, 사용자 조회 로직이 모여 있는 `src/services/user/user.ts`(또는 admin util 옆)에 하나 만들어 둡시다.

```ts
// src/services/user/user.ts (또는 적당한 서버 유틸)
import { getSessionUser } from "@/services/user/user";

/**
 * 로그인 + role === 'admin' 을 보장한다.
 * 관리자 전용 서버 액션 맨 앞에서 호출한다.
 */
export async function requireAdmin() {
  const sessionUser = await getSessionUser();
  if (!sessionUser) throw new Error("UNAUTHORIZED");
  if (sessionUser.role !== "admin") throw new Error("FORBIDDEN");
  return sessionUser;
}
```

> `getSessionUser()`가 이미 `role`을 포함해 반환한다면 위처럼 바로 쓰면 됩니다. 혹시 role이 빠져 있다면 `getSessionUser` 내부 select에 `role`을 더하거나, 여기서 `users` 테이블을 한 번 더 조회해 확인하면 됩니다.

### 2단계 — 관리자 함수 맨 앞에 한 줄씩 추가

```ts
// go.ts:270
```

### 3단계 — 같은 패턴의 다른 관리자 액션도 함께

10차 때 짚었던 `togglePostStatus`·`applySanction` 같은 기존 관리자 액션에도 같은 `await requireAdmin();` 한 줄을 맨 앞에 붙여 주세요. 헬퍼를 한 번 만들어 두면, 앞으로 관리자 액션을 새로 추가할 때마다 첫 줄에 붙이는 습관만 들이면 되니 부담도 적습니다.

## 2. 비로그인 `users` 노출 닫기 (이메일·admin role)

### 왜 필요한가

anon(publishable) 키만 있어도 `GET /rest/v1/users`가 회원 이메일과 `role`까지 그대로 돌려줍니다. 이메일은 그 자체로 개인정보인 데다, `role`이 보이면 "누구를 노려야 관리자 권한을 얻을 수 있는지"까지 공격자에게 알려 주는 셈이라 더 위험합니다.

### 정석 — 공개용 뷰를 분리한다

망고지수나 랭킹처럼 비로그인 사용자에게도 보여줘야 하는 정보가 있다면, `users` 테이블을 통째로 여는 대신 **필요한 컬럼만 담은 뷰**를 만들어 그쪽만 공개하는 게 정석입니다.

```sql
-- 공개해도 되는 컬럼만: 닉네임/점수/등급 등 (email·role·auth_id·phone 제외)
create view public.user_public as
  select id, name, kts_score, kts_grade
  from public.users
  where deleted_at is null;

-- 뷰는 anon 읽기 허용, 원본 테이블은 anon 차단
grant select on public.user_public to anon, authenticated;
```

그러고 나서 `users` 본체의 SELECT 정책을 다시 좁혀 줍니다. 이를테면 로그인 사용자는 공개 표시에 필요한 행만, 더 좁히려면 본인 행만 읽도록 막는 식이죠.

```sql
-- 현재 anon까지 통과시키는 SELECT 정책을 찾아서 교체
-- (Supabase 대시보드 > Authentication > Policies > users 에서 확인)
drop policy if exists "<현재 열려 있는 users select 정책 이름>" on public.users;

create policy "users select - 로그인 사용자만" on public.users
  for select
  to authenticated
  using (true);   -- 공개 표시가 필요하면 authenticated 까지만, 더 좁히려면 본인/관련자 조건
```

클라이언트에서 닉네임·점수를 보여주던 쿼리는 `from("users")`를 `from("user_public")`으로 바꿔 주기만 하면 됩니다.

### 확인 방법

고친 뒤에 같은 호출을 한 번 더 던져 보세요. 0행이나 권한 오류가 돌아오면 제대로 닫힌 겁니다.

```bash
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/users?select=email,role&limit=1" \
  -H "apikey: $ANON_KEY" -H "Authorization: Bearer $ANON_KEY"
# → []  이면 OK (이메일/role 더 이상 노출 안 됨)
```

## 3. 곁들임 — 가볍게

- **`/go` 빌드 깨짐**: `@vis.gl/react-google-maps`가 `package.json`엔 들어 있는데 설치가 안 된 채로 작업하면 `/go`가 빌드 에러로 죽습니다. 작업 전에 `npm install`을 한 번 돌려 주세요. CI에서 `npm ci && npm run build`가 돌고 있다면 이런 건 머지 전에 알아서 걸러집니다.
- **Google Maps 팝업**: 지도에 "can't load Google Maps correctly"가 뜨는 건 키가 비었거나, 키의 허용 도메인에 실행 도메인이 빠져 있을 때입니다. Google Cloud 콘솔에서 키의 HTTP 리퍼러 제한에 배포 도메인을(필요하면 localhost까지) 넣어 주세요.
- **`/mypage` 404**: 망고지수는 `/profile`에 있으니, `/mypage`로 걸어 둔 링크가 남아 있다면 `/profile`로 바꿔 주세요.

우선순위는 1번 권한, 그다음 2번 이메일 노출입니다. 둘 다 시연이나 배포 전에 닫고 가면 좋겠네요. 화이팅입니다.
