## 3. [필수] RLS(Row Level Security) 점검

**(1) 문제**: publishable 키는 공개돼도 되는 키라, DB를 지키는 건 전적으로 RLS 정책뿐인데 그게 켜져 있는지 미확인입니다.

**(2) 왜 고쳐야 하나**: 키를 안 숨겨도 되는 이유가 바로 "RLS가 막아주니까"입니다. RLS가 꺼져 있으면 브라우저 콘솔만 열어도 남의 메시지를 읽거나 남의 이름으로 글·찜을 넣을 수 있습니다. 특히 찜 추가에서 `user_id`를 클라이언트가 직접 보내기 때문에(`useLikes.ts:98`), RLS가 없으면 남의 id로도 찜이 들어갑니다.

**(3) 해결 단계**
1. Supabase 대시보드에서 다음 테이블의 RLS가 ON인지 확인합니다: `messages`, `chats`, `common_likes`, `posts`, `used_goods`, `users`.
2. 각 테이블마다 읽기(select)는 "볼 수 있는 사람만", 쓰기(insert/update/delete)는 "본인 것만" 허용하는지 정책을 확인합니다.
3. 로그아웃 상태에서 콘솔에 `supabase.from('messages').select('*')`를 날려보고 빈 배열이 오면 정상입니다.

**(4) 예시 정책**

`common_likes` — 본인 id로만 추가/삭제:
```sql
alter table common_likes enable row level security;

create policy "본인 찜만 추가"
on common_likes for insert to authenticated
with check (
  user_id = (select id from users where auth_id = auth.uid())
);

create policy "본인 찜만 삭제"
on common_likes for delete to authenticated
using (
  user_id = (select id from users where auth_id = auth.uid())
);
```

`messages` — 내가 속한 채팅방만 읽기:
```sql
create policy "내 채팅방 메시지만 읽기"
on messages for select to authenticated
using (
  chat_id in (
    select id from chats
    where user_id_1 = (select id from users where auth_id = auth.uid())
       or user_id_2 = (select id from users where auth_id = auth.uid())
  )
);
```

> 더 튼튼하게 하려면 찜·메시지 같은 쓰기 작업을 서버 액션으로 옮기는 방법도 있습니다(5번, 그리고 `services/likes.ts`의 `addLike`/`removeLike` 활용). 서버에서 `getUser()`로 신원을 확정하면 클라이언트가 보낸 `user_id`를 믿지 않아도 됩니다.

## 5. [필수] 낙관적 업데이트의 짝꿍 — 찜 실패 시 되돌리기

**(1) 문제**: 찜 토글이 화면을 먼저 바꾸고 DB 결과(error)를 확인하지 않습니다(`useLikes.ts:89-100`).

**(2) 왜 고쳐야 하나**: 낙관적 업데이트(클릭 즉시 화면 반영)는 빠른 느낌을 주는 좋은 기법이지만, 실패하면 화면을 원래대로 되돌리는 처리가 반드시 짝으로 있어야 합니다. 지금은 DB 저장이 실패해도(네트워크 오류, RLS 거부 등) 화면엔 "찜됨"으로 남아서, 새로고침하면 풀려버리는 불일치가 생깁니다. 참고로 채팅 메시지 전송(`ChatRoomClient.tsx`)은 이미 `catch`에서 임시 메시지를 지우고 있습니다. 그 패턴을 찜에도 똑같이 적용하는 것입니다.

**(3) 해결 단계**
1. 클릭 시 현재 상태(`wasLiked`)를 먼저 저장합니다.
2. 화면을 낙관적으로 바꿉니다.
3. DB 결과의 `error`를 확인하고, 실패하면 화면을 원복합니다.

**(4) Before / After**


## 6. [필수] 메시지 교체를 `tempId`로 정확히 하기

**(1) 문제**: 내가 보낸 임시 메시지를 실제 메시지로 바꿀 때 "내용이 같은 것"을 찾아 교체합니다(`useChatRoomRealtime.ts:48-60`).

**(2) 왜 고쳐야 하나**: "ㅇㅇ", "넵" 같은 같은 말을 연달아 두 번 보내면, 어떤 임시 메시지를 교체해야 하는지 내용만으로는 구분이 안 됩니다. 그러면 엉뚱한 걸 교체하거나 중복으로 남을 수 있습니다. 임시 메시지에 고유한 `tempId`를 달아두고 그 id로 정확히 매칭하면 깔끔합니다.

**(3) 해결 단계**
1. 임시 메시지를 만들 때 `tempId`(= 그때의 `Date.now()`)를 함께 넣습니다.
2. 서버 응답이 오면 그 `tempId`를 가진 메시지를 실제 id로 즉시 교체합니다.
3. Realtime INSERT 수신 시엔 "같은 실제 id가 이미 있으면 무시"만 하면 됩니다(지금의 `prev.some(...)` 로직 유지).

**(4) Before / After**

```ts
// 타입에 임시 식별자 허용
type OptimisticMessage = MessageWithSender & { tempId?: number };
```

```tsx
// Before — ChatRoomClient.tsx (요약): id에 Date.now()만 넣고
const optimistic = { id: Date.now(), /* ... */ };
setMessages((prev) => [...prev, optimistic]);
await sendMessageAction({ chatId, postId, content });
```

```tsx
// After
const tempId = Date.now();
const optimistic: OptimisticMessage = { /* ... */, id: tempId, tempId };
setMessages((prev) => [...prev, optimistic]);

const saved = await sendMessageAction({ chatId, postId, content });
// 서버가 돌려준 실제 id로 tempId 기준 교체
setMessages((prev) =>
  prev.map((m) => (m.tempId === tempId ? { ...m, id: saved.id, tempId: undefined } : m)),
);
```

```ts
// useChatRoomRealtime.ts — 내용 매칭(idx 찾기) 블록은 지우고, 중복 방지만 유지
setMessages((prev) => {
  if (prev.some((m) => m.id === newMsg.id)) return prev; // 이미 교체됐으면 무시
  return [...prev, msgWithSender];
});
```

> `Date.now()`를 임시 id로 쓰는 결정 자체는 문서에 근거가 있으니 그대로 둬도 됩니다. 여기서 바꾸는 건 "교체할 대상을 찾는 방법"뿐입니다.

## 8. [제안] setInterval 정리하기 (도배 방지 훅)

**(1) 문제**: 쿨다운 카운트다운용 `setInterval`을 만들지만 정리(cleanup)하지 않습니다(`useSpamPrevention.ts:21`).

**(2) 왜 고쳐야 하나**: 쿨다운 도중에 컴포넌트가 사라지면 인터벌이 살아남아, 이미 없는 컴포넌트의 state를 바꾸려다 경고가 납니다. interval id를 ref에 보관하고 cleanup에서 꺼주면 깔끔합니다.

**(3) 해결 단계**
1. `intervalRef`를 만들어 interval id를 저장합니다.
2. 언마운트 시 `clearInterval`하는 cleanup을 추가합니다.
3. 새 쿨다운 시작 전 기존 인터벌이 있으면 먼저 정리합니다.

**(4) After**

```ts
import { useEffect, useRef, useState } from "react";

export function useSpamPrevention() {
  const [isCooldown, setIsCooldown] = useState(false);
  const [cooldownSeconds, setCooldownSeconds] = useState(10);
  const sendTimestamps = useRef<number[]>([]);
  const intervalRef = useRef<ReturnType<typeof setInterval> | null>(null);

  useEffect(() => {
    return () => { if (intervalRef.current) clearInterval(intervalRef.current); };
  }, []);

  const recordSend = (): boolean => {
    const now = Date.now();
    sendTimestamps.current.push(now);
    sendTimestamps.current = sendTimestamps.current.filter((t) => now - t < 3000);

    if (sendTimestamps.current.length >= 5) {
      setIsCooldown(true);
      setCooldownSeconds(10);
      if (intervalRef.current) clearInterval(intervalRef.current);
      intervalRef.current = setInterval(() => {
        setCooldownSeconds((prev) => {
          if (prev <= 1) {
            clearInterval(intervalRef.current!);
            setIsCooldown(false);
            return 0;
          }
          return prev - 1;
        });
      }, 1000);
      return true;
    }
    return false;
  };

  return { isCooldown, cooldownSeconds, recordSend };
}
```

## 10. [제안] 서버 클라이언트 유틸 하나로 통일

## 13. [제안] users 동기화 방식 확정 + 문서화

**(1) 문제**: 우리 앱은 곳곳에서 `public.users`의 `id`를 `auth_id`로 조회하는데, 회원가입/소셜로그인 시 그 행이 만들어지는 로직이 코드에 안 보입니다.

**(2) 왜 고쳐야 하나**: `public.users`에 행이 없으면 "로그인은 됐는데 채팅/글쓰기에서 `/login`으로 튕기는" 버그가 납니다. 보통 DB 트리거로 자동 생성합니다.

**(3) 해결 단계**
1. 이미 트리거로 처리 중이면 `docs`에 "users 동기화는 DB 트리거로 처리" 한 줄만 남깁니다.
2. 아니라면 아래 트리거를 추가합니다.

**(4) 예시 (SQL)**

```sql
create or replace function handle_new_user()
returns trigger language plpgsql security definer as $$
begin
  insert into public.users (auth_id, email, name, role)
  values (
    new.id,
    new.email,
    coalesce(new.raw_user_meta_data->>'name', new.raw_user_meta_data->>'full_name', '익명'),
    'user'
  );
  return new;
end; $$;

create trigger on_auth_user_created
after insert on auth.users
for each row execute function handle_new_user();
```
