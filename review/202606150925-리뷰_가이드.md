## ① 빌드 3건 복구 — 가장 먼저, 한 번에

세 건 다 고친 뒤 마지막에 `npx tsc --noEmit`가 0이 나오는지 꼭 확인하고 가세요.

### 1-가. `api/user/route.ts` — 삭제된 모듈 import 교체

지금 코드는 이미 지워진 `@/lib/supabaseServer`를 부르고 있어요. 새로 단일화된 클라이언트로 바꾸면 됩니다.

```ts
// src/app/api/user/route.ts
// 바꾸기 전
import { createSupabaseServerClient } from "@/lib/supabaseServer";
// ...
const supabase = await createSupabaseServerClient();   // 8줄, 50줄 두 곳

// 바꾼 후
import { createClient } from "@/utils/supabase/server";
// ...
const supabase = await createClient();                 // 8줄, 50줄 두 곳 모두
```

왜 중요한가. 함수가 들어 있던 모듈이 통째로 사라지면, 그걸 부르던 곳은 무조건 같이 바뀌어야 합니다. import 한 줄과 호출 두 곳, 총 세 군데만 맞추면 끝이에요. `createClient`도 `await`가 필요한 비동기 함수라 호출 형태는 그대로 두면 됩니다.

### 1-나. SearchBar와 Filters props 통일

`SearchBar`가 받는 props와 `JobFilters`/`RentalFilters`/`UsedGoodsFilters`가 넘기는 props가 다릅니다. 둘 중 한쪽으로 맞춰야 하는데, **Filters가 이미 넘기고 있는 모양(`searchInput`/`onSearchChange`/`onSearchSubmit`/`locationFilter`/`onLocationChange`)에 SearchBar를 맞추는 쪽**을 추천합니다. 화면 세 곳이 이미 그 모양으로 쓰고 있으니 수정 범위가 작아요.

```tsx
// src/components/common/SearchBar.tsx
type SearchBarProps = {
  searchInput: string;
  onSearchChange: (value: string) => void;
  onSearchSubmit: (e: React.FormEvent) => void;
  locationFilter?: string;
  onLocationChange?: (value: string) => void;
  showLocation?: boolean;
};

export default function SearchBar({
  searchInput,
  onSearchChange,
  onSearchSubmit,
  locationFilter,
  onLocationChange,
  showLocation = false,
}: SearchBarProps) {
  return (
    <form onSubmit={onSearchSubmit}>
      <input
        value={searchInput}
        onChange={(e) => onSearchChange(e.target.value)}  // e 타입은 자동 추론
      />
      {showLocation && (
        <select
          value={locationFilter}
          onChange={(e) => onLocationChange?.(e.target.value)}
        >
          {/* ... */}
        </select>
      )}
    </form>
  );
}
```

왜 중요한가. TS7006(implicit any) 6건은 콜백 파라미터 `e`·`v`에 타입이 안 붙어서 났습니다. props 타입을 위처럼 명시하면 `onChange={(e) => ...}`의 `e`는 React가 자동으로 추론해 줘서 따로 타입을 안 적어도 돼요. 타입을 한 곳(props)에만 적으면 나머지는 따라옵니다.

### 1-다. `usedgoods/[id]/page.tsx` 상세 select 확장

users 타입에 `alert_*` 컬럼이 새로 생겨서, 상세 쿼리가 가져오는 데이터 모양과 안 맞습니다. 방법은 두 가지예요.

방법 A — select에 빠진 컬럼을 추가(가장 단순):

```ts
// src/app/(user)/usedgoods/[id]/page.tsx:63 근처
.select(`
  *,
  users (
    id, nickname, profile_image,
    alert_enabled, alert_keywords   // 새로 생긴 컬럼들 추가
  )
`)
```

방법 B — 이 화면에서 실제로 쓰는 필드만 타입으로 좁히기(상세 페이지가 alert 정보를 안 쓴다면 이쪽이 더 깔끔):

```ts
type DetailUser = Pick<UserRow, "id" | "nickname" | "profile_image">;
```

왜 중요한가. 공유 테이블(users)에 컬럼을 더하면, 그 테이블을 select하는 모든 곳의 결과 타입이 바뀝니다. 화면이 그 컬럼을 쓰든 안 쓰든 타입은 맞춰야 하고요. 상세에서 알림 정보가 필요 없다면 방법 B로 "이 화면은 이만큼만 본다"를 분명히 하는 게 읽기에도 좋습니다.

## ② useAlertSettings 실패 롤백

지금은 화면을 먼저 켜고(`setX`) 나서 DB에 저장(`await`)합니다. 저장이 실패하면 화면은 켜졌는데 DB는 꺼진 채로 어긋나죠. 4차에 RentalCard·useJobLike에서 잡았던 패턴을 그대로 가져오면 됩니다.

```ts
const toggleAlert = async (next: boolean) => {
  const prev = enabled;       // 1) 이전 값을 기억
  setEnabled(next);           // 2) 낙관적으로 먼저 반영
  const { error } = await saveAlertSetting(next);
  if (error) {
    setEnabled(prev);         // 3) 실패하면 원래대로 되돌림
    // 토스트로 "저장에 실패했어요" 알려주면 더 좋습니다
  }
};
```

왜 중요한가. 낙관적 업데이트는 반응이 빨라 좋지만, 실패 경로가 없으면 화면과 DB가 따로 놉니다. "기억 → 반영 → 실패 시 원복" 이 세 박자가 한 세트라 생각하면 돼요.

## ③ users RLS — Supabase 콘솔에서 확인하고 적용

문서에 정책을 적어둔 건 좋습니다. 이제 실제로 적용됐는지 확인할 차례예요.

**1) 지금 anon 키로 뚫리는지 확인** (터미널에서, `<URL>`·`<ANON_KEY>`는 본인 값):

```bash
curl "https://<프로젝트>.supabase.co/rest/v1/users?select=email,phone&limit=5" \
  -H "apikey: <ANON_KEY>"
```

남의 email·phone이 배열로 돌아오면 아직 구멍입니다. `[]`나 권한 에러가 나오면 막힌 거고요.

**2) 콘솔에서 정책 적용** (Supabase → SQL Editor):

```sql
-- users 테이블에 RLS 켜기
alter table public.users enable row level security;

-- 로그인한 본인 행만 읽기
create policy "users_select_own"
  on public.users for select
  using (auth.uid() = auth_id);   -- 컬럼명이 auth_id가 맞는지 확인

-- 본인 행만 수정
create policy "users_update_own"
  on public.users for update
  using (auth.uid() = auth_id);
```

**3) 적용 뒤 1)을 다시 돌려** `[]`가 나오는지 확인합니다.

왜 중요한가. 코드의 `.eq("id", user.id)`는 "정상 경로에서 본인 것만 본다"는 약속일 뿐, anon 키로 REST를 직접 때리면 그 약속을 우회합니다. 실제 차단은 **DB 레벨 RLS 정책**만 해요. 실사용자 데이터가 들어가는 순간 남의 이메일·전화번호가 그대로 읽히는 사고라, 시연 전에 꼭 닫고 가야 합니다.

> 정책을 켜면 다른 화면이 깨질까 걱정될 수 있는데, 위 정책은 "본인 행"은 그대로 열어두니 로그인 사용자의 자기 프로필 조회/수정은 정상 동작합니다. 깨진다면 어디서 누구 데이터를 읽는지 같이 보겠습니다.

## ④ 구조를 바꿀 때 팀에 알리기 + 간단한 CI

이번 빌드 3건은 전부 "한쪽이 바꿨는데 다른 쪽이 못 따라간" 경우였습니다. 두 가지 습관으로 막을 수 있어요.

**1) 구조 변경은 PR 설명이나 짧은 공유로 남기기.** 아래 셋 중 하나라도 하면 "공유 대상"이라 생각하면 됩니다.

- 모듈/파일을 **삭제**하거나 옮길 때 (예: `supabaseServer` 삭제)
- 공통 컴포넌트의 **props API**를 바꿀 때 (예: SearchBar)
- 공유 테이블에 **컬럼을 더하거나 뺄 때** (예: users `alert_*`)

PR 본문에 "이제 서버 클라이언트는 `@/utils/supabase/server`의 `createClient`만 씁니다" 한 줄이면 충분해요. 새 패턴이 생길 때는 그걸 쓸 사람과 5분만 같이 화면을 보고 가면 머지 충돌이 거의 안 납니다.

**2) 머지 전 빌드 게이트.** GitHub Actions에 두 줄만 걸어도 됩니다. `.github/workflows/ci.yml`:

```yaml
name: CI
on:
  pull_request:
    branches: [develop]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx tsc --noEmit   # 타입 에러 잡기
      - run: npm run build      # 프로덕션 빌드 확인
```

왜 중요한가. 이번 세 건은 누군가 머지 전에 `tsc`를 한 번만 돌렸어도 다 잡혔습니다. CI는 그 "한 번"을 자동으로 강제해 줘요. 그리고 macOS 로컬에서 통과해도 Linux(`ubuntu-latest`)에서 깨지는 경우가 있어서, 배포 환경과 같은 곳에서 미리 확인하는 효과도 있습니다. develop이 항상 빌드되는 상태로 유지되면 시연·배포 날 사고가 안 나고요.
