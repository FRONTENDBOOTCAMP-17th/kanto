## 1. [필수] 상세 페이지가 비로그인에서 깨지는 문제

### 무슨 일이 일어났나
`users` RLS를 닫으면서 anon(로그아웃)은 `users` 행을 하나도 못 읽게 됐습니다. 상세 페이지는 작성자 정보를 `posts(*, users(*))` 조인으로 가져오는데, anon에선 그 `users`가 **null** 로 들어오니 `data.posts.users.created_at` 같은 접근에서 터지는 겁니다.

### 해결 — 두 가지를 같이 합니다

**(A) DB: 공개해도 되는 컬럼만 anon에 열어주기 (근본 해결)**

지금 정책은 "내 행만 읽기"라 너무 닫혀 있습니다. 작성자의 이름·아바타·가입일은 어차피 화면에 보여주는 공개 정보고, 민감한 건 email·phone뿐이니 그것만 가리면 됩니다. 제일 깔끔한 건 **공개용 뷰**를 하나 만드는 방법이라 생각합니다.

```sql
-- 공개해도 되는 컬럼만 담은 뷰
create or replace view public_user_profiles as
select id, name, avatar_url, created_at, auth_id, role
from users;

-- 뷰는 RLS를 호출자 권한으로 타므로, 안전하게 security_invoker 끄고 공개 읽기 허용
alter view public_user_profiles set (security_invoker = off);
grant select on public_user_profiles to anon, authenticated;
```

그리고 조인을 `users(...)` 대신 이 뷰로 바꿔주면 됩니다. Supabase는 FK가 있으면 뷰도 embed로 조인할 수 있습니다. 만약 안 되면 작성자 정보를 별도 쿼리로 가져와 합치는 식으로 가면 됩니다.

> email·phone은 여전히 `users` 본 테이블에 있고 anon은 못 읽으니, 6차에 닫은 보안은 그대로 유지됩니다.

**(B) 코드: 작성자가 없어도 안 터지게 방어 (응급 + 안전망)**

뷰를 적용하기 전이든 적용한 뒤든, 작성자 정보가 비어 있을 때 화면이 죽지 않게 옵셔널 체이닝과 폴백을 넣어둡니다.

```tsx
// rental/[id]/page.tsx, job/[id]/page.tsx
<VerifyAuthor
  authorAuthId={rental.posts.users?.auth_id}   // ?. 추가
  ...
/>

// UsedGoodsDetail.tsx
const user = data.posts.users;                  // null 가능
const accession = user?.created_at ? new Date(user.created_at) : null;
// 이름/아바타도 user?.name, user?.avatar_url 로
```

`VerifyAuthor`가 `authorAuthId`가 undefined일 때 수정/삭제 버튼을 감추도록 처리하면 비로그인에서도 안전하게 렌더됩니다. 편집 버튼은 원래 작성자에게만 보이는 거라 이렇게 가도 자연스럽습니다.

## 2. [필수] 찜하기가 403으로 안 되는 문제

### 무슨 일이 일어났나
`common_likes`에 INSERT하면 거기 걸린 **알림 생성 트리거**가 `common_notifications`에 INSERT를 시도합니다. 그런데 그 알림은 글쓴이에게 가는 거라, 찜한 사람의 권한으로는 RLS를 못 넘습니다. 그래서 403이 나고 찜 전체가 롤백되는 겁니다.

### 해결 — 트리거 함수를 `SECURITY DEFINER`로

알림을 만드는 트리거 함수가 **함수 소유자(보통 postgres) 권한**으로 실행되게 하면 RLS를 우회해 안전하게 알림을 꽂아줍니다. 시스템이 만드는 알림이라 이렇게 가는 게 맞다고 봅니다.

```sql
-- 기존 트리거 함수 이름에 맞춰 수정하세요 (예시)
create or replace function notify_on_like()
returns trigger
language plpgsql
security definer            -- 핵심: 소유자 권한으로 실행
set search_path = public
as $$
begin
  insert into common_notifications (user_id, type, target_id, target_type, actor_id)
  values (
    (select user_id from posts where id = new.target_id),  -- 받는 사람: 글쓴이
    'like', new.target_id, 'post', new.user_id             -- 보낸 사람: 찜한 사람
  );
  return new;
end;
$$;
```

> `security definer` 함수는 보안상 `set search_path`를 반드시 고정해 두세요. 적용한 뒤 로그아웃→로그인 계정으로 찜을 눌러 하트가 유지되는지 확인해 보면 됩니다.

## 4. [제안] 머지 전 빌드 게이트 (반복 깨짐의 근본 해결)

`.github/workflows/ci.yml`을 하나 만들어 PR마다 타입체크와 빌드를 돌리게 해둡니다.

```yaml
name: CI
on:
  pull_request:
    branches: [develop, main]
jobs:
  typecheck-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build
```

그다음 GitHub repo Settings → Branches → develop 보호 규칙에서 이 체크를 **Required**로 걸면, 타입에러나 빌드깨짐이 있는 PR은 머지 자체가 막힙니다. 지금처럼 매 회차 빌드가 깨지는 일은 자연스럽게 사라집니다.

      ticking = false;

  return () => window.removeEventListener("scroll", onScroll);
}, []);
