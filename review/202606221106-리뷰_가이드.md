# Kanto 10차 리뷰 — 해결 가이드 (2026-06-22)

리뷰에서 짚은 항목들을 어떻게 고치면 되는지 단계별로 정리했습니다. 시연이 코앞이니 1번(서버 액션 인가)부터 보시면 되고, 2·3번은 여유가 되면 손대고 나머지는 시연 끝난 뒤로 미뤄도 괜찮습니다.

---

## 1. [필수] 관리자 서버 액션에 인가 추가 (`requireAdmin` 헬퍼)

### 왜 필요한가

페이지 가드(`(admin)/layout.tsx`)는 어디까지나 사용자가 어드민 페이지를 화면에 그릴 때만 동작합니다. 그런데 `"use server"` 액션은 페이지가 아니라 **따로 떨어진 HTTP 엔드포인트**라, 누군가 이 액션을 직접 호출하면 레이아웃을 한 번도 거치지 않고 액션 본문이 바로 실행돼 버립니다. 그래서 액션 안에서 service-role(`createAdminClient`)로 데이터를 바꾸기 전에 **그 액션 스스로** 호출자가 관리자인지 확인해야 한다는 거라 생각합니다.

### 1단계 — 공용 헬퍼 하나 만들기

레이아웃이 하던 검사와 똑같은 내용을 액션에서도 부를 수 있게 함수 하나로 빼냅니다.

```ts
// src/utils/auth/requireAdmin.ts
import { createClient } from "@/utils/supabase/server";
import { createAdminClient } from "@/utils/supabase/admin";

/**
 * 현재 로그인 사용자가 관리자인지 확인하고, 관리자면 users.id 를 돌려준다.
 * 관리자가 아니면 예외를 던져 액션 실행을 막는다. (handled_by 기록에 그대로 쓰면 됨)
 */
export async function requireAdmin(): Promise<number> {
  const supabase = await createClient();
  const {
    data: { user },
  } = await supabase.auth.getUser();
  if (!user) throw new Error("UNAUTHORIZED");

  const admin = createAdminClient();
  const { data } = await admin
    .from("users")
    .select("id, role")
    .eq("auth_id", user.id)
    .single();

  if (!data || data.role !== "admin") throw new Error("FORBIDDEN");
  return data.id; // adminId — 기존에 handled_by/admin_id 에 넣던 값
}
```

### 2단계 — 각 액션 맨 앞에서 호출

기존 액션들은 로그인한 사용자의 id를 조회해 `handled_by`에 넣는 코드를 이미 갖고 있는데, 그 부분을 `requireAdmin()` 한 줄로 바꿔주면 **인가 확인과 adminId 조회가 한 번에** 끝납니다.

```ts
// 예) admin/users/_actions/liftSanction.ts
"use server";
import { createAdminClient } from "@/utils/supabase/admin";
import { requireAdmin } from "@/utils/auth/requireAdmin";

export async function liftSanction(userId: number): Promise<void> {
  const adminId = await requireAdmin();   // ← 관리자 아니면 여기서 멈춤
  const admin = createAdminClient();

  await admin.from("users").update({ suspended_until: null }).eq("id", userId);
  await admin.from("user_sanctions").insert({
    user_id: userId,
    admin_id: adminId,        // 기존에 조회하던 값을 그대로 사용
    sanction_type: "lifted",
    expires_at: null,
  });
  // ... 알림 insert 동일
}
```

같은 식으로 맨 앞 한 줄만 바꾸면 되는 곳:

- `togglePostStatus` — `const adminId = await requireAdmin();` 후 `handled_by: adminId`
- `applySanction` — `adminId`/`adminName`을 따로 조회하던 부분을 `requireAdmin()`으로(이름이 필요하면 헬퍼가 `name`도 함께 돌려주게 살짝 늘려도 됩니다)
- `resolveReport` / `dismissReport` / `updateReportResolution` — 안에 있는 `getCurrentAdminId()`를 `requireAdmin()`으로 교체(이 함수가 지금은 role을 안 보니, 교체하면 한 번에 정리됩니다)
- `getPostReports` — 맨 앞에 `await requireAdmin();` 한 줄(읽기 전용이지만 같은 원칙)

### 3단계 — 실패를 사용자에게 자연스럽게

`requireAdmin()`이 던지는 예외는 클라이언트에서 잡아 토스트로 안내해도 되고, 정상 흐름에선 어차피 관리자만 버튼을 보니 크게 신경 쓸 일은 없습니다. 중요한 건 어쨌든 막혀 있다는 사실 그 자체라 생각합니다.

### 확인 방법

고친 뒤에 일반 회원(role=`user`)으로 로그인한 상태에서 액션이 제대로 거부되는지 보면 됩니다. 가장 간단하게는 헬퍼에 임시 `console.log`를 하나 두고 user로 동작을 시도해 `FORBIDDEN`이 찍히는지 확인하거나, 리뷰의 K30처럼 user 상태에서 어드민 동작 경로가 막히는지 직접 눌러보면 됩니다.

## 2. [제안] `blockUserAction` — 참여자 검증

차단할 때 요청자가 정말 그 채팅방의 참여자인지부터 확인하는 게 좋습니다. 지금은 `isUser1 = chat.user_id_1 === userData.id` 한 줄로만 갈라놓다 보니, 참여자가 아닌 방 id를 넘겨도 엉뚱하게 `user_id_2_left = true`가 찍혀버릴 수 있습니다.

```ts
const isUser1 = chat.user_id_1 === userData.id;
const isUser2 = chat.user_id_2 === userData.id;
if (!isUser1 && !isUser2) {
  throw new Error("이 채팅방의 참여자가 아닙니다.");
}
// 이후 isUser1 분기 그대로
```

## 3. [제안] `favorites/page.tsx` — page 파라미터 가두기

주소창에 비정상 값(`?page=abc`, `0`, 범위 초과)이 들어와도 늘 정상 페이지가 뜨도록, page 값을 1부터 마지막 사이로 한 번 가둬줍니다. 다만 `totalPages`를 계산한 **뒤에** 클램프해야 하니 흐름을 살짝 손봐야 합니다.

```ts
const rawPage = Number(page) || 1;            // NaN 방어
// ... likedIds 로 totalPages 계산 ...
const currentPage = Math.max(1, Math.min(rawPage, totalPages));
const offset = (currentPage - 1) * ITEMS_PER_PAGE;
// slice 는 이 offset 으로
```

`내글보기(myposts)`도 같은 구조라면 이참에 함께 손봐 두면 좋겠습니다.

## 4. [제안 · 9차 이월] `getAdminChatMessages` — `single()` → `maybeSingle()`

없는 채팅방 id로 들어가면 `single()`이 에러를 던지면서 500으로 떨어질 수 있습니다. 이걸 `maybeSingle()`로 바꿔주면 없는 방을 깔끔하게 404로 넘길 수 있는데요, 상세 페이지가 이미 `if (!chat) notFound()`를 기대하고 있어 자연스럽게 맞물립니다.

```ts
const { data: chat } = await admin
  .from("chats")
  .select("...")
  .eq("id", chatId)
  .maybeSingle();   // single() → maybeSingle()
```

## 5. [사소] 커밋 메시지

커밋 메시지는 한 일을 그대로 적어 두면 나중에 그 커밋을 다시 찾을 때 헷갈릴 일이 없습니다.

- 내용과 메시지를 맞추기 (예: 스크롤 체이닝 수정인데 "알림 버그 해결"로 적힌 커밋)

우선순위는 1번(서버 액션 인가) 하나라 생각합니다. 헬퍼 한 줄을 각 액션 맨 앞에 두는 작업이라 양 자체는 많지 않습니다. 이것만 정리해 두면 어드민 기능의 읽기·쓰기 인가가 한 규칙으로 묶여서, 시연에서 관리자 화면을 마음 놓고 보여줄 수 있을 겁니다. 화이팅입니다.
