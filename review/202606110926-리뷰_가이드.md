## 1. [필수·최우선] 빌드 복구 — getLikeList 호출부 맞추기

`getLikeList`가 이제 객체를 반환하니, 받는 쪽을 객체에서 꺼내 쓰면 됩니다.

```ts
// usedgoods/page.tsx
// (before)
const [posts, likedIds] = await Promise.all([
  getUsedGoodsList({ ... }),
  getLikeList("used_goods"),
]);
...
<UsedGoodsList initialPosts={pagedPosts} initialLikedIds={likedIds} />

// (after)
const [posts, likeData] = await Promise.all([
  getUsedGoodsList({ ... }),
  getLikeList("used_goods"),
]);
const { likedIds, currentUserId } = likeData;   // 여기서 꺼낸다
...
<UsedGoodsList
  initialPosts={pagedPosts}
  initialLikedIds={likedIds}
  currentUserId={currentUserId}   // 이왕 받은 김에 클라 재조회 없이 내려주기
/>
```

이렇게 하면 애초에 리팩토링하려던 목적(클라이언트가 세션을 다시 안 묻게)도 완성됩니다. 같은 방식으로 `rental/page.tsx`, `job/page.tsx` 등 **`getLikeList`를 부르는 모든 페이지**를 `grep "getLikeList"`로 찾아 한 번에 맞춰주세요.

> 교훈: 함수의 **반환 모양**을 바꾸는 커밋에서는, 커밋 전에 `npx tsc --noEmit` 한 번. 이번 건 그거 한 번이면 끝이었습니다.

## 2. [필수] usedgoods/[id] 상세를 서버 조회로

목록은 이미 `getUsedGoodsList`로 잘 했으니 상세도 똑같이 갑니다.

```ts
// services/usedGoods/usedGoods.ts 에 추가
import { createSupabaseServerClient } from "@/lib/supabaseServer";

export async function getUsedGoodsDetail(postId: number) {
  const supabase = await createSupabaseServerClient();   // 서버 클라이언트
  const { data, error } = await supabase
    .from("used_goods")
    .select(`*, posts (*, users (*))`)
    .eq("post_id", postId)
    .single();
  if (error) return null;
  return data;
}
```

```ts
// usedgoods/[id]/page.tsx
import { getUsedGoodsDetail } from "@/services/usedGoods/usedGoods";

export default async function UsedGoodsDetailPage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const data = await getUsedGoodsDetail(Number(id));
  if (!data) return <div>상품을 찾을 수 없습니다.</div>;
  return <UsedGoodsDetail data={data} />;
}
```

`posts.status` 리터럴 에러도 서버 조회 타입을 통해 자연히 풀립니다(남으면 `status` 컬럼을 gen types 기준으로 좁혀주세요).

## 3. [필수] users RLS — 민감 컬럼 노출 막기

지금 anon 키로 `users`의 email·phone이 그대로 읽힙니다. publishable 키는 공개되는 키라, **RLS가 유일한 방어선**입니다. Supabase 대시보드 → Authentication/Policies(또는 SQL):

```sql
alter table public.users enable row level security;

-- 본인 행만 읽기
create policy "users read own row"
on public.users for select
to authenticated
using ( auth_id = auth.uid() );
```

공개 프로필(닉네임·아바타)이 목록에 필요하면, **민감 컬럼을 뺀 뷰**를 따로 두고 그 뷰만 anon에 여세요. 절대 `users` 원본 테이블을 anon select 허용으로 두지 마세요.

## 4. [필수] /chats·/jobs 404 — 라우트 상수화

링크가 코드 여러 곳에 흩어져 단/복수가 어긋났습니다. 상수로 모으면 다시는 안 틀립니다.

```ts
// src/constants/routes.ts
export const ROUTES = {
  chat: "/chat",
  job: "/job",
  usedgoods: "/usedgoods",
  rental: "/rental",

```

`Header.tsx:30,313,405`, `Footer.tsx:6`, `UsedGoodsDetail.tsx:178`의 `"/jobs"`·`"/chats"`를 `ROUTES.job`·`ROUTES.chat`으로 교체. 이후 `grep '"/chats"\|"/jobs"' src/`가 0건이어야 합니다.

## 6. [제안] currentUserId 경로 하나로

권장: **서버에서 받아 내려주기**(이미 getLikeList가 반환). 그러면 `useCurrentUserId`(클라에서 또 조회)는 대부분 필요 없어집니다. 서버 컴포넌트 → 클라 컴포넌트로 `currentUserId`를 prop으로 전달하는 방식으로 통일하고, `useCurrentUserId`는 정말 서버 데이터가 없는 순수 클라 화면에서만 쓰세요.

## 7. [제안] useImageUpload cleanup deps 한 가지

언마운트 cleanup은 잘 넣었습니다. 한 가지만 — `useEffect(() => { return () => previews.forEach(revoke) }, [])`처럼 deps가 `[]`면, cleanup이 **첫 렌더 시점의 빈 previews**를 기억(stale closure)해 마지막 미해제분이 남을 수 있습니다. `imagePreviews`를 ref에 같이 담아 cleanup에서 ref.current를 돌리거나, 개별 해제(이미 하고 있음)로 충분하면 그대로 두되 의도를 주석으로 남겨주세요.
