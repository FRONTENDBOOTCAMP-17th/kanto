## 2. 보안 — `users` 테이블 RLS (실측) `[필수]`

anon(publishable) 키는 클라이언트 JS에 그대로 노출되는 공개 키입니다. 그래서 RLS가 유일한 방어선인데, 실제로 그 키로 테이블을 직접 호출해 봤습니다.

```
GET /rest/v1/users?select=*   (apikey = NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY)
→ HTTP 200, 행 그대로 반환
   노출 컬럼: id, phone, name, email, auth_id, avatar_url, provider, role ...
```

즉 **누구든 publishable 키만 있으면(브라우저 개발자도구로 바로 보임) 전체 회원의 이메일·전화번호·auth_id를 긁어갈 수 있습니다.** 개인정보 유출이라 가장 먼저 막아야 합니다.

반면 같은 테스트에서 `chats`·`common_likes`·`messages`는 anon으로 0행(secret 키로는 데이터 존재) — RLS가 잘 막고 있었습니다. `posts`·`jobs`·`rentals`·`used_goods`가 읽히는 건 공개 목록이라 정상입니다.

→ `users`에 "본인 행만 select" 또는 "공개 컬럼만 노출하는 뷰" 정책을 추가하세요. 해결 가이드에 SQL 예시를 넣었습니다.

## 3. 타입 검사 결과 (`npx tsc --noEmit`) `[필수]`

이번 변경에서 타입 에러 3건이 새로 생겼습니다. 빌드/런타임에서 데이터 모양이 어긋날 수 있다는 신호라 그냥 두면 안 됩니다.

```
src/app/(user)/rental/page.tsx(22,21)        RentalWithPost[] 타입 불일치
src/app/(user)/usedgoods/[id]/page.tsx(22,24) UsedGoods.content: string ← null 들어옴
src/services/rental/rental.ts(36,10)         as RentalWithPost[] 캐스팅이 실제 모양과 안 맞음
```

핵심 원인은 **`RentalWithPost` 타입이 두 파일에 따로 정의된 것**입니다.

- `src/type/rental.ts` (19줄)
- `src/type/rental/rental.ts` (35줄)

`RentalList.tsx`는 `@/type/rental`을, `services/rental/rental.ts`는 `@/type/rental/rental`을 import 합니다. 같은 이름인데 모양이 달라서 서로 안 맞는 거죠. 하나로 합쳐야 합니다(가이드 참고).

## 4. 멤버별 리뷰

### 박소유 (soyu) — 구인구직 글쓰기 `CreateJobForm.tsx`

**잘한 점**
- 이미지 업로드 실패 시 방금 만든 `posts` 행을 다시 지우는 보상 로직(83~97줄) — 1차에서 칭찬한 글쓰기 폼 패턴을 그대로 유지했습니다.
- 급여 입력에 `toLocaleString()` 천단위 표시 + 저장 시 `replace(/,/g,"")` — UX가 깔끔합니다.
- `removeImage`에서 `URL.revokeObjectURL` 호출로 메모리 누수 방지. 좋습니다.

**`[필수]` 작성 성공 후 `/jobs`로 보내는데 그 라우트가 없습니다 (404)**
```ts
// CreateJobForm.tsx:134
router.push("/jobs");
```
`src/app/(user)/jobs`는 존재하지 않습니다(있는 건 `job/create`뿐). E2E에서 `/jobs` → **HTTP 404** 확인. 즉 공고를 정상 등록해도 마지막에 404 화면을 만나게 됩니다. 1차의 `/chats`와 똑같은 단·복수 라우트 혼선이에요.

**`[필수]` jobs insert 실패 시 posts 행이 고아로 남습니다**
```ts
// 이미지 실패는 post를 지우는데(잘함), jobs 실패는 안 지움
const { error: jobError } = await supabase.from("jobs").insert({ ... });
if (jobError) {
  alert("공고 등록에 실패했습니다.");
  setIsSubmitting(false);
  return;            // ← post.id 행이 그대로 남음 (jobs 없는 떠돌이 posts)
}
```
이미지 실패는 보상하는데 jobs 실패는 보상하지 않아 일관성이 깨집니다. 이미지 실패 때와 똑같이 `posts.delete()`를 넣어주세요.

**`[제안]`** 18개의 개별 `useState` — 초보 단계에선 명시적이라 읽히긴 합니다. 다음 단계로 `useState` 하나에 객체로 묶거나 `useReducer`를 써보면 제출 핸들러가 훨씬 단순해집니다(가이드에 예시).

### 임태형 (THLIMM) — 방렌트 목록 `RentalList.tsx`

**잘한 점**
- 드롭다운 바깥 클릭 시 닫힘(`handleClickOutside`), 모바일 바텀시트 열릴 때 `body` 스크롤 잠금, 필터 변경 시 페이지 1로 리셋 — 디테일이 좋습니다.
- 이미지 인디케이터 버튼에서 `e.preventDefault()/stopPropagation()`으로 `Link` 이동을 막은 처리, `amenities.filter((a) => a in AMENITY_ICONS)` 방어적 매핑 — 꼼꼼합니다.
- 빈 상태("등록된 매물이 없습니다") 처리까지 되어 있어 E2E에서 깔끔하게 렌더됐습니다.

**`[필수]` 찜 토글에 롤백이 없습니다 (1차 5번 지적의 재발)**
```ts
// RentalList.tsx:175~198
setItems((prev) => prev.map(... 낙관적 토글 ...));
if (target.isLiked) {
  await supabase.from("common_likes").delete()...   // error 무시
} else {
  await supabase.from("common_likes").insert(...)    // error 무시
}
```
박소유님이 `useLikes`에서 고친 "error 시 되돌리기"가 여기엔 없습니다. DB 쓰기가 실패해도 화면은 찜된 것처럼 보여 화면-DB 불일치가 생깁니다. `useLikes`의 수정본을 그대로 가져오면 됩니다. (사실 중고/방렌트 찜 로직이 거의 같으니 **하나의 훅으로 합치는 것**이 근본 해결입니다 — 가이드 참고.)

**`[제안]`** 목록을 `initialPosts` 전체로 받아와 `useMemo` 필터 + `slice` 페이지네이션을 클라이언트에서 합니다. 매물이 적을 땐 문제없지만, 1차에서 채팅을 커서 페이지네이션으로 설계한 것과 비교하면 목록은 전부 메모리로 올리는 구조라 매물이 많아지면 무거워집니다. 지금 당장 바꾸진 않더라도 "전부 가져온다"는 점은 인지해 두세요.

**`[사소]`** 드롭다운 위치를 `getBoundingClientRect()`로 렌더 중에 계산해 `fixed`로 띄웁니다. 드롭다운 연 채로 스크롤하면 위치가 안 따라옵니다. 또 모바일/데스크톱 카테고리 드롭다운 JSX가 거의 같으니 작은 컴포넌트로 빼면 654줄이 한결 가벼워집니다.

## 5. 팀 공통

- **단·복수 라우트 혼선이 반복됩니다** (`/chats`, `/jobs`). 라우트 문자열을 컴포넌트마다 직접 적다 보니 생기는 실수예요. `routes.ts` 같은 상수 파일에 모아 쓰면(`ROUTES.chat(id)`) 오타가 컴파일 단계에서 잡힙니다.
- **클라이언트에서 DB에 직접 쓰는 패턴이 여러 곳**입니다(`useLikes`, `RentalList`, `CreateJobForm`이 `user_id`를 클라가 들고 insert). RLS가 받쳐주면 동작은 하지만, 신뢰 경계가 클라이언트로 내려와 있어 위험합니다. 글쓰기·찜은 점진적으로 **서버 액션**으로 옮기는 걸 권합니다.
- **tsc 에러 3건이 방치**되어 있습니다. `develop` 머지 전에 `tsc --noEmit`를 한 번씩 돌리거나 CI에 넣으면 타입 깨짐이 쌓이지 않습니다.
- 커밋·역할 분담은 1차 대비 고르게 진행됐습니다(soyu 구인 / THLIMM 방렌트 목록 / DoHyuk 상세·타입). 좋은 흐름입니다.
