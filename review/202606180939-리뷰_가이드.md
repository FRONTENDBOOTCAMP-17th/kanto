## 1. `/profile` 런타임 크래시 복구 [필수]

증상: 로그인한 사용자가 `/profile`에 들어가면 `PROVIDER_KEYS is not defined`가 뜨면서 화면이 깨집니다.
원인은 `ProfileCard.tsx`인데, 이미 사라진 이름 두 개(`PROVIDER_KEYS`, `formatSellerInfoCreatedAt`)를 아직 부르고 있어서 그렇습니다.

### 1) `PROVIDER_KEYS` 다시 정의하기

`PROVIDER_KEYS`는 로그인 수단(provider) 키 목록입니다. 37행에서 이렇게 쓰고 있죠.

```tsx
const providerKey = PROVIDER_KEYS.find((p) => p === user.provider) ?? "email";
```

제일 간단한 건 이 키 목록을 상수로 직접 선언하는 겁니다. 컴포넌트 파일 맨 위, import 아래에 이렇게 넣어 주세요.

```tsx
// 지원하는 로그인 수단 키 — i18n 메시지(Profile.providers)의 키와 1:1로 맞춥니다.
const PROVIDER_KEYS = ["email", "google", "kakao"] as const;
type ProviderKey = (typeof PROVIDER_KEYS)[number];
```

이렇게 두면 `p` 매개변수의 implicit any(TS7006)도 덤으로 같이 사라집니다(`p`가 `ProviderKey`로 추론되니까요).
혹시 프로젝트에 provider 상수 파일이 이미 있다면(예: `@/constants/...`) 그걸 import 하는 쪽이 더 낫습니다. 출처가 한 군데로 모이니까요.

> 팁: `Profile.providers` i18n 키와 이 배열이 어긋나면 라벨이 빈칸으로 나옵니다. 두 곳을 같은 키로 맞춰 주세요.

### 2) 죽은 import 제거

18행의 import는 `formatTime`에 이제 없는 이름을 가리키고 있어서 지워야 합니다.

```tsx
// 삭제
import { formatSellerInfoCreatedAt } from "@/utils/formatTime";
```

지운 다음에는 파일 안에서 `formatSellerInfoCreatedAt(...)`를 부르는 데가 남았는지 한 번 훑어보세요.
38행을 보면 가입일은 이미 `new Date(user.created_at)`로 직접 만들고 있으니, 대개는 부르는 곳이 없을 겁니다.
혹시 어딘가에서 부르고 있다면 `formatTime.ts`에 아직 남아 있는 함수(예: `formatTimeAgo`)로 바꾸거나, 필요한 포맷 함수를 새로 export 해 주면 됩니다.

### 3) 확인

```bash

PORT=3100 npm run dev   # 로그인 후 /profile 진입 → 에러 없이 뜨는지, 본인인증 모달도 열리는지
```

## 2. 어드민 사이드바 `badge` 타입 정리 [필수]

증상: `(admin)/layout.tsx(95,53) TS2339: Property 'badge' does not exist`.
원인은 `navItems` 배열에 `badge`가 있는 항목과 없는 항목이 섞여 있다는 점입니다. 그러면 유니언 타입에는 `badge`가 안 들어가는데, 그걸 꺼내려다 막히는 거죠.

해결은 간단합니다. 항목 타입을 하나로 정해 두고 `badge`만 옵셔널로 빼면 됩니다.

```tsx
type NavItem = {
  icon: LucideIcon;       // lucide-react 의 아이콘 타입
  label: string;
  href: string | null;
  badge?: number;         // 있을 수도, 없을 수도
};

const navItems: NavItem[] = [
  { icon: LayoutDashboard, label: "대시보드", href: "/admin" },
  { icon: Flag, label: "신고 내역", href: "/admin/reports", badge: pendingCount },
  // ...
];
```

이렇게 하면 `{ icon: Icon, label, href, badge }`로 구조분해해도 `badge`가 `number | undefined`로 잡혀서 타입이 통과합니다.
쓰는 쪽에서는 `{badge ? <Badge>{badge}</Badge> : null}` 정도로 받으면 됩니다.

## 3. CI 게이트 붙이기 [필수] — 재발을 막는 근본책

`.github/workflows/ci.yml`을 하나 만들어서, PR이 올라올 때마다 타입체크와 빌드를 알아서 돌리게 합니다.

```yaml
name: CI
on:
  pull_request:
    branches: [develop, main]
jobs:
  typecheck-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build
        env:
          # 빌드에 필요한 공개 키만 GitHub Secrets 로 등록해 주입
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY }}
```

그다음 GitHub 저장소 **Settings → Branches → Branch protection rules**에서 `develop`을 보호로 걸고,
"Require status checks to pass before merging"에 방금 만든 `typecheck-build`를 required로 지정하세요.
이렇게 해 두면 tsc나 build가 깨진 PR은 머지 버튼 자체가 안 눌려서, 이번 같은 프로필 크래시가 develop까지 새어 들어오지 못합니다.

> 빌드에 service 키 같은 비밀이 필요하면 GitHub Secrets에만 넣고 yml엔 절대 평문으로 적지 마세요.

## 4. 본인인증 `devCode` 운영 노출 가드 [제안]

`api/profile/verification/route.ts`의 POST 응답을 보면, 이메일이 설정 안 됐을 때 실제 인증번호를 그대로 내려보내는 부분이 있습니다.

```ts
// 지금
devCode: emailResult.isConfigured ? undefined : code,

// 제안 — 개발 환경에서만 노출
devCode:
  emailResult.isConfigured || process.env.NODE_ENV === "production"
    ? undefined
    : code,
```

이렇게 두면 로컬 개발에서는 콘솔이나 응답으로 코드를 받아 편하게 테스트하고, 운영에서는 절대 새어 나가지 않습니다.

## 5. 결제 시스템 메시지 unread 원자 증가 [제안] (여유 있을 때)

`transaction.ts`의 `postSystemMessage`가 unread를 "읽고 +1 해서 다시 쓰는" 식이라, 동시에 메시지가 날아오면 한 번씩 누락될 수 있습니다.
채팅 일반 메시지(`postMessage`)도 똑같은 패턴이라, 나중에 둘을 한꺼번에 RPC로 옮기면 깔끔하게 정리됩니다.

```sql
-- 예시: 원자적 증가 RPC
create or replace function increment_unread(p_chat_id bigint, p_for_user1 boolean)
returns void language sql security definer as $$
  update chats set
    user_id_1_unread = user_id_1_unread + case when p_for_user1 then 1 else 0 end,
    user_id_2_unread = user_id_2_unread + case when p_for_user1 then 0 else 1 end
  where id = p_chat_id;
$$;
```

급한 건 아닙니다. 1~3번을 먼저 끝내고, 결제와 채팅을 같이 손보는 회차에 묶어서 하면 됩니다.

우선 1·2로 지금 깨진 걸 고치고, 3으로 다시는 안 깨지게 막아 두는 순서면 됩니다. 나머지 4·5는 여유 생길 때 천천히 해도 괜찮습니다.
