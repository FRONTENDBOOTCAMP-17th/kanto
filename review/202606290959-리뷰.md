# Kanto 14차 리뷰 — 공개 페이지 SEO 점검 + 조원별 코드 품질

작성일: 2026-06-29

이번 회차는 검색엔진에 노출되는 비로그인(공개) 페이지의 SEO를 집중적으로 봤습니다. 메인·목록·상세 페이지가 서버에서 렌더되는지, 메타데이터·sitemap·robots가 제대로 깔려 있는지를 정적 검증과 샘플 확인 위주로 점검했고, 그와 함께 6/25 이후 조원별로 들어온 코드를 살폈습니다. 미리 한 가지만 밝혀두면, 이번 회차는 라이브 E2E를 새로 돌리지는 않았고 SEO는 정적 검증과 코드 샘플 대조로 확인했습니다.

## 지난 리뷰 반영 현황 (짧게)

- [필수] GO! 관리자 서버액션 인가 누락(`services/go/go.ts` adminGetMeetups/adminForceEndMeetup) — 이번 SEO 회차에서는 별도 재현은 안 했습니다. 다음 회차에 requireAdmin 헬퍼로 모았는지 다시 보겠습니다. 혹시 헬퍼 신설은 끝났을까요?
- [필수] anon `users` 이메일+role 노출 회귀 — 공개 컬럼 뷰 분리 쪽 진행 상황이 궁금합니다. 이번 SEO 점검 중에도 비로그인 노출 표면을 함께 보는 항목이라, 다음에 anon GET으로 다시 확인하겠습니다.
- 채팅 진입 병렬화·메인 인기글 limit·읽음 처리 통합은 지난 회차에 해결로 확인했고, 그 상태가 유지되고 있습니다.

## 이번 리뷰 핵심

- Kanto는 다섯 조 중 SEO 인프라가 가장 단단합니다. sitemap·robots·metadataBase·title.template에 더해 상세 라우트마다 generateMetadata와 JSON-LD, 동적 opengraph-image까지 갖춰져 있어서, 이 부분은 강하게 칭찬할 만합니다.
- 다만 그 위에서 마무리가 덜 된 곳이 몇 군데 있습니다. 메인 페이지가 서버 컴포넌트인데 클라이언트 훅을 쓰고 있어 런타임에서 깨질 위험이 있고, JSON-LD를 끼워 넣는 과정에서 상세 두 곳에 닫히지 않은 `<div>`가 남았습니다. 이 둘이 이번 회차 최우선입니다.

## 공개 페이지 SEO 점검

### 잘 되어 있는 부분 (칭찬)

먼저 잘한 점부터 짚겠습니다. 공개 페이지가 검색에 잡히려면 크롤러가 받는 첫 HTML에 실제 콘텐츠가 들어 있어야 하는데, Kanto는 목록(`/usedgoods`, `/rental`, `/job`)과 상세(`/[type]/[id]`)가 전부 서버 컴포넌트에서 데이터를 받아 자식 클라이언트 컴포넌트에 props로 내려주는 구조라, 크롤러가 빈 셸이 아니라 채워진 HTML을 받습니다. 이게 SEO의 토대인데 이미 잘 잡혀 있습니다.

거기에 더해 상세 라우트마다 `metadata.ts`를 분리해 `generateMetadata`로 제목·설명·OG 이미지·canonical을 채우고(`src/app/(user)/usedgoods/[id]/metadata.ts`), Product·JobPosting·Organization JSON-LD를 심고, 동적 `opengraph-image.tsx`까지 둔 건 다른 조보다 한참 앞서 있는 작업입니다. `metadataBase`와 `title.template`로 제목 일관성을 잡은 것, `sitemap.ts`·`robots.ts`를 둔 것, 구글·네이버 verification 파일까지 챙긴 것도 좋습니다. 이 정도면 SEO 기반은 거의 완성에 가깝다고 봅니다.

### 고칠 부분

**[필수] 메인 페이지가 서버 컴포넌트인데 클라이언트 훅을 호출합니다.**
`src/app/(user)/main/page.tsx:2`에서 `import { useTranslations } from "next-intl";`를 가져와 `:18`의 `MainPage` 함수 본문 최상단에서 `const t = useTranslations("Main");`을 부르고 있는데, 이 파일에는 `'use client'`가 없어 서버 컴포넌트입니다. `useTranslations`는 클라이언트 훅이라 서버 컴포넌트에서 호출하면 런타임 에러로 이어지고, 메인은 사이트의 얼굴이라 여기서 깨지면 크롤러가 에러 페이지를 보게 됩니다. 서버에서는 `next-intl/server`의 `getTranslations`를 `await`로 써야 합니다(가이드에 전환 스니펫을 넣어뒀습니다). 참고로 `rental/page.tsx`·`usedgoods/page.tsx`는 이미 `getTranslations`(`/server`)를 잘 쓰고 있어서 메인만 패턴이 어긋난 상태입니다.

**[사소] 목록 페이지에 중복 import가 남아 있습니다.**
`src/app/(user)/rental/page.tsx:4-5`에 `getRentalList`가 두 번 import 되어 있고(`getRentalList, getRentalBarangays`를 가져온 줄 바로 아래에 `getRentalList`만 다시 가져옴), `src/app/(user)/usedgoods/page.tsx:4-5`도 `getUsedGoodsList`가 동일하게 중복입니다. 동작에는 문제 없지만 머지 과정에서 남은 흔적으로 보이니 한 줄씩 정리하면 깔끔합니다.

**[제안] 상세 metadata fallback에 noindex를 넣어두면 좋겠습니다.**
`src/app/(user)/usedgoods/[id]/metadata.ts:17`의 `if (!data) return { title: "게시글을 찾을 수 없습니다" };`는 데이터가 없을 때의 안전망인데, 실제로는 페이지가 `notFound()`로 404를 내니 대부분 문제가 안 됩니다. 다만 메타데이터 단계에서 색인 거부를 명시해 두면 만에 하나 200으로 새는 경우에도 "찾을 수 없습니다" 페이지가 검색에 잡히지 않습니다. `robots: { index: false }`를 fallback에 함께 넣어두는 걸 권합니다(rental 쪽 metadata.ts도 동일).

**[제안] robots.ts disallow에 로그인 전용 경로가 빠져 있습니다.**
`src/app/robots.ts:11`의 disallow는 `["/admin/", "/api/", "/chat/", "/profile/edit"]`인데, 로그인해야만 의미 있는 `/profile/`·`/favorites/`·`/notifications/`·`/myposts/`·`/create/`·`/payment/`·`/go/`가 빠져 있습니다. 특히 `/go`는 실시간 `'use client'` 페이지라 크롤링 대상이 아니니 막아두는 게 맞습니다. 이런 경로는 크롤러가 시간만 낭비하고 색인 가치가 없으니 disallow에 넣어주면 크롤 예산이 공개 콘텐츠로 몰립니다.

## 조원별 리뷰

### 박소유 (@soyupark1997)

결제 보안 쪽을 단단하게 가져가고 있습니다. 미수금 해제를 원자적으로 claim 하도록 만든 부분, Xendit 연동, 카드 만료·관리자 결제 처리까지 흐름이 촘촘합니다. 유틸 통합도 좋았는데, `formatPrice`·`formatDate`가 여기저기 흩어져 있던 걸 한 곳으로 모아 중복을 걷어낸 건 정확히 가야 할 방향입니다. 챗봇 i18n도 챙겼고요.

**[필수·주의] `claimTransactionRelease`에 함수 차원의 구매자 검증이 없습니다.**
`src/services/payment/transaction.ts:97`의 `claimTransactionRelease(id)`는 인자로 트랜잭션 id만 받아 service-role 클라이언트로 곧장 `status: "released"`로 바꿉니다. 지금은 호출부인 `confirmReceiptAction`이 `buyer_id == userId`를 검증하고 있어서 현재 경로상으로는 안전합니다. 다만 이 함수가 service-role로 RLS를 우회하는 만큼, 호출부 한 곳의 검증에만 안전이 걸려 있는 구조는 다음에 다른 호출부가 생기면 쉽게 새어나갑니다. 함수 자체에 `.eq("buyer_id", userId)` 같은 필터를 한 겹 더 두는 방어적 설계를 권합니다.

**[제안]** 결제 만료 시각을 `PaymentCard`에서 클라이언트 시계로 계산하는 부분이 있는데, 실제 만료 확정은 webhook이 서버에서 처리하니 운영상 문제는 없습니다. 다만 "여기 계산은 표시용이고 만료의 단일 출처는 webhook"이라는 주석을 한 줄 달아두면 다음 사람이 오해하지 않습니다.

**[사소]** `formatTime.ts`에 이제 `formatPrice`까지 들어가 있어서, 파일 이름이 내용을 못 따라갑니다. `format.ts` 정도로 바꾸면 이름이 내용을 정직하게 말해줍니다.

### 김도혁 (@DoHyuk-Centric)

이번 회차 SEO 인프라의 핵심을 맡았습니다. JSON-LD·generateMetadata·sitemap·robots 전반이 김도혁 손에서 나온 작업인데, 위 "잘 되어 있는 부분"에서 칭찬한 토대가 대부분 여기입니다. 글쓰기 폼의 드래그 업로드 UI, 어드민의 TanStack Query·감사 로그·권한·공지 배너까지 범위가 넓고 완성도도 좋습니다.

**[필수] JSON-LD를 끼워 넣다가 닫히지 않은 `<div>`가 두 곳 남았습니다.**
`src/app/(user)/job/[id]/page.tsx`를 보면 `:81`의 `<div className="page-container w-full py-6">` 안에 `<script type="application/ld+json">`를 넣은 뒤 `:86`에서 `<div ... divide-y ...>`를 열고, 바로 `:87`에서 또 `<div className="page-container pb-12">`를 엽니다. 원래 본문이 시작하는 컨테이너 앞에 JSON-LD용 래퍼 div가 한 겹 더 끼어든 모양새라 구조가 어긋나 있습니다. `src/app/(user)/rental/[id]/page.tsx`도 같은 패턴인데, `:72`의 `<script>` 다음 `:76`에서 `<div className="flex items-center justify-between">`를 열고 `:77`에서 또 `<div className="flex items-center justify-between mt-4">`를 엽니다. 해결 방향은 단순합니다. `<script>`는 이미 있는 최상위 div 바로 안에 그대로 두고, 그 뒤로는 원래 본문 구조가 바로 이어지게 해서 여분의 래퍼 div를 없애면 됩니다(가이드에 수정본을 넣었습니다).

**[제안]** `UnifiedBanner`에서 렌더 도중 `setState`로 파생 상태를 만드는 부분이 있는데, 동작은 하지만 왜 그렇게 했는지 주석을 달아두면 좋겠습니다. 렌더 중 setState는 보통 안티패턴이라 다음 사람이 무심코 "고쳐야 할 버그"로 오해할 수 있어서입니다.

**[사소]** `ImageUploadField`의 드래그 업로드 경로에서 `maxCount` 체크를 건너뛸 수 있어 보입니다. 클릭 업로드와 드래그 업로드가 같은 상한을 타도록 맞춰주면 좋겠습니다.

### 이동근 (@dongkeun99)

버그 수정과 폼 검증, UX 개선을 폭넓게 가져갔습니다. 로그아웃 리다이렉트, 비로그인 신고 에러, 가격 자연수 검증, select 필수값을 브라우저 기본 검증으로 전환한 것, 상세 레이아웃 통일과 수정/삭제, 로그인 유도 모달, i18n까지 손이 많이 갔습니다. 로그인 유도 모달 패턴(`go/page.tsx`)은 흐름이 자연스러워서 좋았습니다.

**[필수] `responsive-select.tsx`의 숨김 `<select>`가 빈 onChange로 제어되고 있습니다.**
`src/components/.../responsive-select.tsx:83-90`을 보면, 브라우저 기본 필수 검증을 살리려고 `required`인 숨김 `<select>`를 두고 `value={value}`로 값을 묶었는데, `:90`의 핸들러가 `onChange={() => {}}`입니다. value를 주면서 onChange를 빈 함수로 두면 React가 controlled input 경고를 띄웁니다(값은 바뀌는데 핸들러가 아무 일도 안 하니 React가 모순으로 봅니다). 이 select는 표시용이 아니라 검증 트리거이니, `value` 대신 `defaultValue`로 두는 비제어 방식이나 `readOnly`를 쓰는 쪽이 의도에 맞습니다(가이드에 두 방법 다 적었습니다).

**[제안]** `JOB_DETAIL_SELECT`에 `auth_id`를 추가했는데, 이에 맞춰 `JobDetail` 타입도 같이 갱신됐는지 확인해주세요. select 컬럼과 타입이 어긋나면 런타임에는 멀쩡해 보여도 타입 안전망이 헛돌게 됩니다.

### 임태형 (@THLIMM)

상세 페이지에 지도를 붙이는 작업을 가져갔습니다. `ApproxAreaMapWithProvider`·`LocationPicker`·바랑가이 라벨까지 들어가서 위치 표현이 풍부해졌습니다. AI 이미지 검수 설계 문서도 커밋했는데, 구현 전 단계에서 fail-open(검수 실패 시 통과) 정책을 잡아둔 건 초기 정책으로 합리적입니다.

**[필수·확인] 렌탈 편집 모드에서 location이 null로 저장될 위험이 있습니다.**
`RentalCreateForm`의 `buildLocationFields`를 보면, 편집 모드에서 사용자가 위치를 새로 찍지 않아(`picked = null`) 기존 값(`initialData.location`)도 null인 레거시 글을 저장하면 location이 null로 들어갈 수 있어 보입니다. 새 글이야 위치를 반드시 찍게 강제하면 되지만, 위치 없이 올라가 있던 옛날 글을 편집 저장할 때 이 경로를 타지 않는지 확인해주세요. 이 경우엔 어떻게 동작하나요?

**[제안]** `UsedGoodsDetail`에서 `ApproxAreaMapWithProvider` 블록이 이미지 있을 때와 없을 때(대략 `163-169`행, `234-241`행) 두 군데에 복붙되어 있습니다. 같은 지도 JSX를 변수로 한 번 만들어 두 분기에서 꺼내 쓰면 한 곳만 고치면 되니 유지보수가 편해집니다.

**[사소]** AI 검수 설계 문서를 미리 커밋해 둔 건 좋습니다. 실제로 `/api/moderate-image`를 구현할 때는 rate limit을 꼭 같이 넣어주세요. 외부 모델 호출이 들어가는 라우트라 무방비면 비용·남용에 그대로 노출됩니다.

## 팀 공통

- SEO 인프라를 팀 차원에서 끝까지 밀어붙인 점이 인상적입니다. 다섯 조 중 가장 앞서 있습니다. 이제 남은 건 "심어놓은 게 실제로 깨지지 않는지"인데, 이번에 잡힌 메인 페이지 서버/클라이언트 훅 문제와 상세 orphaned div가 딱 그 마무리 영역입니다.
- 머지 흔적으로 보이는 중복 import(rental/usedgoods 목록)와 닫히지 않은 div가 동시에 나온 걸 보면, JSON-LD·i18n 리팩터가 여러 손을 거치며 머지될 때 빌드/타입 게이트가 한 번 더 걸러줬으면 좋았을 항목들입니다. CI에 `tsc`·`build` 게이트가 이미 있으니, JSX 구조 깨짐과 클라이언트 훅 오용도 build 단계에서 잡히는지 확인해보면 좋겠습니다.
- 작성자가 네 명에게 고루 분포해 있는 점은 좋습니다.

## 이번 회차 Top 3

1. [필수] `main/page.tsx`가 서버 컴포넌트인데 `useTranslations`(클라이언트 훅) 호출 — `getTranslations`(`next-intl/server`)로 전환.
2. [필수] `job/[id]/page.tsx`·`rental/[id]/page.tsx` JSON-LD 삽입 시 닫히지 않은 여분 `<div>` — 래퍼 제거하고 본문 구조 복구.
3. [제안] `robots.ts` 로그인 전용 경로 누락(`/profile/`·`/go/` 등) 보강 + 상세 metadata fallback에 `robots: { index: false }` 추가.
