## 2. 이번 회차에서 가장 급한 것 — 관리자 "서버 액션"에 인가가 빠져 있습니다 [필수]

페이지 읽기는 막혔는데, 관리자 데이터를 **바꾸는** 서버 액션들에는 아직 관리자 확인이 없습니다. 9차에서 `togglePostStatus` 하나를 짚었는데, 이번에 보니 회원 제재·신고 처리 쪽 액션들도 같은 모양입니다. 정리하면 이렇습니다.

- `togglePostStatus` (글 내리기/올리기)
- `applySanction` / `liftSanction` (회원 정지 / 정지 해제)
- `resolveReport` / `dismissReport` / `updateReportResolution` (신고 처리)

이 액션들은 공통적으로 `auth.getUser()`로 **로그인 여부만** 확인하고, 그 사용자의 `users.id`를 "처리자(`handled_by`/`admin_id`)" 기록용으로 조회한 뒤, 곧장 `createAdminClient()`(service-role)로 데이터를 바꿉니다. 정작 **그 사용자의 `role`이 `admin`인지는 한 번도 보지 않습니다.** 예를 들어 정지 해제는 이렇습니다.

```ts
// admin/users/_actions/liftSanction.ts
export async function liftSanction(userId: number): Promise<void> {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();   // 로그인했는지만 확인

  const admin = createAdminClient();                          // 여기서부터 RLS 우회
  let adminId: number | null = null;
  if (user) { /* handled_by 기록용 id 조회 — role 검사 아님 */ }

  await admin.from("users").update({ suspended_until: null }).eq("id", userId);  // 정지 해제 실행
  // ...
}
```

왜 이게 위험하냐면, **서버 액션은 페이지가 아니라 독립된 HTTP 엔드포인트**라서 그렇습니다. `(admin)/layout.tsx`의 가드는 사용자가 `/admin/...` **페이지를 그릴 때** 동작하는 것이지 액션이 호출될 때 그 앞을 막아 주진 않아서, 액션 본문은 레이아웃 렌더와 무관하게 그대로 실행됩니다. 로그인한 일반 회원이 액션을 호출할 수만 있으면 RLS를 우회하는 service-role로 **남을 영구 정지시키거나, 자기 정지를 스스로 풀거나, 임의의 글을 내려 버리는** 일이 코드상 가능하다는 뜻입니다. 읽기 누설보다 오히려 영향이 큰 권한 상승(privilege escalation)에 가깝습니다.

한 가지 솔직히 덧붙이면, 지금은 레이아웃 가드 덕분에 일반 회원이 어드민 페이지를 못 열어서 액션 참조(action id)가 담긴 번들도 평소엔 손에 안 들어옵니다. "당장 클릭 한 번으로 터지는" 상태는 아니라는 거죠. 그래도 action id는 정적 청크(`/_next/static/...`)로도 흘러나올 수 있고 배포 간 값이 잘 안 바뀌어서, 가드만 믿기엔 위태롭습니다. 무엇보다 **"페이지 가드가 막아 주겠지"에 기대 쓰기 인가를 생략하는 구조 자체**가 위험의 근본이라, 시연에서 관리자 기능(정지·신고 처리·글 내리기)을 직접 보여줄 예정이라면 그 전에 막아 두는 게 좋겠습니다.

고치는 법은 9차에서 드린 방향 그대로입니다. `requireAdmin()` 헬퍼 하나를 만들어 **각 서버 액션 맨 앞 한 줄**로 호출하면 됩니다. 레이아웃 가드와 같은 검사(로그인 + `role === "admin"`)를 액션에서도 한 번 더 하는 거죠. 구체적인 코드는 함께 드리는 가이드 1번에 적어 두었습니다.

## 3. 멤버별 리뷰 (9차 이후 신규 코드)

9차 이후 들어온 코드를 작성자별로 봤습니다. 전체적으로 "새 기능을 더 얹는" 단계에서 "있던 걸 단단하게 다듬는" 단계로 넘어온 인상이네요. 잘된 부분을 그대로 짚고 손볼 곳을 함께 적습니다.

### 3-1. 김도혁(@DoHyuk-Centric) — 관리자 패널 개편 (`/admin/posts` 드로어, 신고 내역 링크)

글 관리 페이지를 목록 + 상세 드로어 구조로 개편하고, 신고 내역 상세에서 유저 프로필·원본 게시글로 바로 이동하는 링크를 붙였습니다(E2E K23). 운영자가 "이 글이 왜 신고됐고 누가 올렸나"를 한 화면에서 따라갈 수 있게 동선을 만든 게 좋네요. `getPostReports`에서 신고별 최신 제재를 뽑을 때 `user_sanctions`를 `created_at` 오름차순으로 모아 Map에 덮어쓰며 "가장 최근 행이 현재 결과"가 되게 한 처리도 정확합니다.

- [필수] 글 상태를 바꾸는 `togglePostStatus`에 관리자 확인이 없습니다(2번). 페이지는 이제 막혔지만 액션은 그대로라, 헬퍼 한 줄을 맨 앞에 두면 됩니다.
- [사소] `getPostReports`는 인증 검사 없이 service-role로 신고 데이터를 읽습니다. 지금은 어드민 페이지 안에서만 호출돼서 레이아웃 가드가 가려 주지만, 이것도 "데이터를 다루는 함수는 스스로 권한을 확인한다"는 같은 원칙의 연장선입니다. 2번을 정리할 때 함께 묶어 두면 깔끔합니다.

### 3-2. 임태형(@THLIMM) — 신고-제재 데이터 정합성 (`user_sanctions.report_id` FK)

이번 회차에서 설계적으로 제일 인상적인 작업이라 생각합니다. 그동안 신고(`common_reports`)와 제재(`user_sanctions`)가 느슨하게 흩어져 있던 걸, `user_sanctions.report_id`에 FK를 걸어 "이 제재가 어떤 신고 때문에 내려졌는지"를 데이터 차원에서 연결했습니다(마이그레이션 `20260619000000_link_reports_to_sanctions.sql`, 이어서 중복 컬럼을 걷어내는 `..._drop_report_sanction_columns.sql`). 게다가 `docs/sanction-data-refactoring.md`에 왜 이렇게 바꾸는지, Phase로 나눠 어떻게 진행하는지를 정리해 둔 점이 특히 좋습니다. 스키마를 바꾸기 전에 의도를 글로 남기는 습관은 팀 전체에 그대로 권하고 싶습니다.

- [제안] 9차에서 말씀드린 `blockUserAction` 참여자 검증은 아직 남아 있습니다. 신고/차단 흐름을 이어서 만지실 때, 차단 시 `chat.user_id_1 === userData.id || chat.user_id_2 === userData.id`로 요청자가 그 방 참여자인지 먼저 확인하고 둘 다 아니면 막아 주면 됩니다(가이드 2번).

### 3-3. 이동근(@dongkeun99) — 헤더 로그인 버튼 깜빡임, 어드민 드로어 스크롤, i18n

새로고침할 때 잠깐 로그인 버튼이 보였다 사라지던 깜빡임을 `useAuthInit`에서 깔끔하게 잡았습니다. `onAuthStateChange`에서 **실제 로그아웃(`SIGNED_OUT`)일 때만 유저를 비우고**, 초기 로드 중 일시적으로 세션이 없는 이벤트(INITIAL_SESSION 등)는 서버에서 주입해 둔 유저를 덮어쓰지 않도록 막은 판단이 정확합니다. 왜 깜빡였는지(초기 세션 없음 이벤트가 SSR 상태를 덮어씀)를 정확히 짚고 그 지점만 손댄 점이 좋네요.

- [사소] "알림 버그 해결"로 올라온 커밋(`1046184`)의 실제 변경은 어드민 드로어에 `overscroll-contain`을 더한 것(스크롤이 배경 페이지로 번지는 것 방지)과 알림 RLS 업데이트 정책 마이그레이션입니다. 좋은 수정인데 **커밋 메시지와 내용이 살짝 어긋납니다.** 나중에 이 커밋을 다시 찾을 때 헷갈릴 수 있으니, 메시지는 한 일을 그대로 적어 주면 좋습니다(예: `fix: 어드민 드로어 스크롤 체이닝 방지 + 알림 update 정책`).

### 3-4. 박소유(@soyupark1997) — 페이지네이션(내글보기·찜목록), 에러 페이지, 구인 폼 수정

목록 페이지에 페이지네이션을 붙이면서 `Pagination` 컴포넌트를 데스크톱/모바일 두 형태로 나눠 만든 게 깔끔합니다. `aria-label`("이전 페이지"/"다음 페이지")을 버튼에 붙이고, 끝 페이지·현재 페이지 주변만 숫자로 보여주고 나머지는 `…`로 접는 로직(`getPageNumbers`)도 잘 짜였습니다. 페이지 상태를 URL 쿼리(`?page=`)로 들고 가게 한 것도 좋은 선택이라, 새로고침·뒤로가기에도 페이지가 유지됩니다. 찜목록은 서버에서 전체를 가져와 `slice`로 잘라 내는데, 찜은 한 사람이 가질 수 있는 양이 제한적이라 이 단계에선 허용할 만합니다.

- [제안] `favorites/page.tsx`에서 `currentPage = Number(page ?? 1)`로 쿼리를 그대로 숫자로 바꿉니다. 주소창에 `?page=abc`(NaN)나 `?page=0`, 범위를 벗어난 큰 수가 들어오면 빈 목록이 보일 수 있습니다. `const currentPage = Math.max(1, Math.min(Number(page) || 1, totalPages))`처럼 1~마지막 페이지 사이로 한 번 가둬 주면, 사용자가 주소를 손대도 항상 정상 페이지가 뜹니다(가이드 3번).

## 4. 팀 공통 / 협업

- **인가는 "한 곳에 모으기"의 절반이 완성됐습니다.** 페이지는 레이아웃으로 모았으니, 이제 쓰기(서버 액션)도 같은 `requireAdmin()` 한 줄로 모으면 읽기·쓰기 양쪽이 같은 규칙을 공유하게 됩니다. 9차 때 말씀드린 "기본적으로 막혀 있고 헬퍼를 부르는 형태"가 그걸로 완성되는 셈이고요.
- **스키마 변경을 문서로 남기는 습관(`docs/sanction-data-refactoring.md`)이 정착되고 있어 좋습니다.** 마이그레이션은 되돌리기 어려운 변경이라, 의도를 글로 남겨 두면 나중에 팀원이 "이 컬럼 왜 지웠지?"를 코드만 보고 추측하지 않아도 됩니다.
- CI 게이트(tsc/build) 효과가 이어지고 있습니다. 이번에도 `npx tsc --noEmit` **0건**으로 깨끗했습니다. 다만 이번 인가 누락처럼 타입은 멀쩡한데 동작이 위험한 경우는 게이트가 못 잡으니, 그 빈틈을 코드 리뷰·E2E가 메운다는 점은 팀이 함께 의식해 두면 좋겠습니다.
- 커밋 컨벤션은 대체로 잘 지켜지는데, `chor:`(→`chore:`) 오타와 위에서 짚은 메시지-내용 불일치가 몇 건 보입니다. 사소하지만 "커밋 메시지만 읽어도 무엇을 했는지 안다"는 기준으로 통일하면 좋겠습니다.

**보안 회귀 — 비관리자 어드민 접근 (K30) — 9차 결함 해결 확인**
- role=`user` 계정으로 `/admin/posts`·`/admin/chats`·`/admin/chats/11`·`/admin/users`에 직접 진입 → **네 곳 모두 `/main`으로 리다이렉트**. 9차에 노출됐던 남의 1:1 채팅 전문도 더는 안 보입니다 — `kanto-K30_admin_chats_11.png`.

**보안 실측 — anon(publishable) 키 직접 읽기 (RLS)**
- anon 키로 `users`·`common_reports`·`user_sanctions`·`chats`·`messages`·`common_notifications`를 직접 조회 → **전부 0행**. `posts`만 공개로 열립니다(의도대로). RLS는 단단합니다.

정상 흐름은 사용자·관리자 모두 통과했고, 9차의 핵심 결함(비관리자 어드민 페이지 접근)은 회귀 테스트로 해결을 확인했습니다. 남은 건 4번에서 말씀드린 "쓰기 액션 인가" 한 가지입니다.

> 참고: 옛 리뷰 스펙(`kanto-2nd*.spec.ts`)에서 `/job/create` 헤딩 텍스트를 literal로 검사하던 테스트 2건이 fail로 떴는데, 국제화 이후 selector가 어긋난 리뷰 측 옛 테스트 문제이고 기능 회귀는 아닙니다(데일리 K6에서 `/job/create` 200·렌더 확인).

## 6. 시연까지 우선순위

1. **[필수] 관리자 서버 액션에 인가 추가** — `togglePostStatus`·`applySanction`·`liftSanction`·`resolveReport`·`dismissReport`·`updateReportResolution` 맨 앞에 `requireAdmin()` 한 줄. 레이아웃 가드와 같은 검사를 액션에서도 한 번 더(2번·가이드 1번). **시연 전 최우선.**
2. **[제안] `blockUserAction` 참여자 검증** — 차단 시 요청자가 그 방 참여자인지 먼저 확인(3-2·가이드 2번).
3. **[제안] `favorites` page 파라미터 클램프** — `?page=abc`·범위 초과 시에도 정상 페이지(3-4·가이드 3번).
4. **[제안] `getAdminChatMessages`의 `single()` → `maybeSingle()`** — 없는 채팅방을 404로(9차 이월).
5. **[사소] 커밋 메시지-내용 일치, `chor`→`chore` 통일.**

9차에서 제일 급했던 어드민 페이지 인가를 레이아웃 한 곳으로 모아 깔끔하게 막은 게 이번 회차의 가장 큰 진전이라 생각합니다. 신고-제재 FK 정리처럼 데이터 설계를 단단히 하는 작업도 좋았고요. 남은 건 같은 인가를 "쓰기 액션"까지 한 줄로 확장하는 것 하나뿐이라, 그것만 정리하면 시연에서 관리자 기능까지 자신 있게 보여줄 수 있는 단계라 봅니다. 화이팅입니다.
