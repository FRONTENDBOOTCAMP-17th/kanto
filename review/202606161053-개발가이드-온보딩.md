# Kanto 개발 온보딩 가이드 — 페이지는 이렇게 만들고, 데이터는 이렇게 가져옵니다

새로 합류한 분을 위한 문서입니다. 로그인과 공통 컴포넌트는 이미 갖춰져 있으니, 이 글은 "Kanto에서 화면 하나를 처음부터 끝까지 어떻게 만드는가"에 집중합니다. 기존 조원도 한 번 훑으면 우리 패턴을 다시 정리하는 데 도움이 될 거라 생각합니다.

실제로 동작하는 **중고거래(usedgoods) 흐름**을 모델로 삼겠습니다. 목록 → 카드 → 상세 → 찜하기까지 한 바퀴가 깔끔하게 들어 있어서, 이 한 슬라이스만 이해하면 방렌트(rental)·구인구직(job)도 같은 모양이라 금방 따라옵니다.

---

## 0. 프로젝트 한눈에

스택은 **Next.js 16 App Router + Supabase**입니다. 폴더는 역할별로 이렇게 나뉩니다.

```
src/
  app/
    (user)/            ← 일반 사용자 화면 (라우트 그룹)
      usedgoods/
        page.tsx              ← 목록 페이지 (서버 컴포넌트)
        _components/          ← 이 화면 전용 컴포넌트
          UsedGoodsList.tsx
          UsedGoodsFilters.tsx
        [id]/
          page.tsx            ← 상세 페이지 (동적 경로)
          _components/
            UsedGoodsDetail.tsx
        create/
          page.tsx            ← 글쓰기
    (admin)/           ← 관리자 화면
    api/               ← Route Handler (서버 API)
    auth/
  services/            ← Supabase를 호출하는 함수들 (데이터 읽기/쓰기)
    usedGoods/usedGoods.ts
    likes.ts, likeToggle.ts
  hooks/               ← 클라이언트 상태/로직 (use client 안에서 씀)
  components/
    common/            ← 여러 화면이 같이 쓰는 컴포넌트 (ContentCard, LikeButton …)
    ui/                ← 버튼·카드 같은 기본 UI 조각
  constants/routes.ts  ← 경로 상수
  utils/supabase/server.ts   ← 서버용 Supabase 클라이언트
  lib/supabase.ts            ← 브라우저용 Supabase 클라이언트
  type/                ← 타입 정의 (supabase.ts는 자동 생성)
```

데이터가 흐르는 큰 그림은 한 줄로 요약됩니다.

```
페이지(서버 컴포넌트)  →  services 함수  →  Supabase
        │
        └─ 받아온 데이터를 props로 → 클라이언트 컴포넌트(목록·카드) → 상호작용은 hooks/이벤트 핸들러
```

여기서 두 가지 원칙만 기억하면 됩니다.

1. **읽기는 가능한 한 서버 컴포넌트에서, services 함수를 통해** 한다.
2. **클릭·입력 같은 상호작용은 `"use client"` 컴포넌트에서** 한다.

왜 이렇게 나누냐면, 서버에서 데이터를 미리 받아 HTML로 내려주면 첫 화면이 빠르고, 브라우저로 내려가는 자바스크립트도 줄어들기 때문입니다. 상호작용이 필요한 부분만 클라이언트로 떼어내는 게 App Router의 기본 사고방식이라 생각하면 됩니다.

## 1. 라우트/페이지 만드는 법

App Router는 **폴더 구조가 곧 URL**입니다. 파일 이름에 규칙이 있어요.

| 파일/폴더 | 의미 | 예시 URL |
| --- | --- | --- |
| `usedgoods/page.tsx` | 목록 페이지 | `/usedgoods` |
| `usedgoods/[id]/page.tsx` | 동적 상세 페이지 | `/usedgoods/42` |
| `usedgoods/create/page.tsx` | 글쓰기 | `/usedgoods/create` |
| `_components/` | 이 화면 전용 컴포넌트 모음 | (URL 안 만들어짐) |

`(user)`처럼 **괄호로 묶인 폴더는 "라우트 그룹"** 입니다. URL에는 안 들어가고(즉 `/user/usedgoods`가 아니라 `/usedgoods`), 같은 성격의 화면을 묶어 공통 레이아웃을 입히는 용도예요. `_`로 시작하는 `_components`도 URL이 안 만들어지는 폴더라, "이 화면에서만 쓰는 조각"을 여기에 둡니다. 여러 화면이 공유하는 건 `_components`가 아니라 `src/components/common`으로 갑니다.

### 서버 컴포넌트가 기본, 언제 "use client"?

`page.tsx`는 아무것도 안 붙이면 **서버 컴포넌트**입니다. 그래서 `async` 함수로 만들고 그 안에서 직접 `await`로 데이터를 가져올 수 있어요. 실제 우리 목록 페이지가 그 모양입니다.

```tsx
// src/app/(user)/usedgoods/page.tsx
export default async function UsedGoodsPage({
  searchParams,
}: {
  searchParams: Promise<SearchParams>;
}) {
  const params = await searchParams;
  // ... 서버에서 바로 데이터 fetch
}
```

`"use client"`를 파일 맨 위에 붙이는 건 다음 중 하나가 필요할 때입니다.

- `useState`·`useEffect` 같은 React 훅을 써야 할 때
- `onClick`·`onChange` 같은 이벤트 핸들러가 필요할 때
- 브라우저 API(`window`, `localStorage` 등)를 써야 할 때

찜 버튼처럼 클릭에 반응하는 조각만 클라이언트로 떼어내고, 나머지는 서버에 두는 게 좋습니다. `"use client"`를 페이지 전체에 붙이면 그 아래 트리가 전부 클라이언트가 돼서 이점이 사라지니, **필요한 잎(leaf)에만** 붙인다고 기억해 주세요.

## 2. 데이터 읽기 — services 함수 + 서버 컴포넌트

Supabase 쿼리를 페이지 안에 직접 쓰지 않고, **`src/services/`의 함수로 빼는 게 우리 규칙**입니다. 같은 쿼리를 목록·상세·찜목록 등 여러 곳에서 재사용하고, 나중에 쿼리를 고칠 때 한 곳만 고치면 되니까요.

중고거래 목록 함수를 봅시다.

```ts
// src/services/usedGoods/usedGoods.ts
import { createClient } from "@/utils/supabase/server";
import type { UsedGoodsWithPost } from "@/type/usedGoods";

const USED_GOODS_SELECT = `
  *,
  used_goods(*),
  users(id, name, avatar_url, created_at)
` as const;

export async function getUsedGoodsList(
  filter?: UsedGoodsListFilter,
): Promise<UsedGoodsWithPost[]> {
  const supabase = await createClient();

  let query = supabase
    .from("posts")
    .select(USED_GOODS_SELECT)
    .eq("post_type", "used_goods")
    .eq("status", "active")
    .not("used_goods", "is", null)
    .order("created_at", { ascending: false });

  if (filter?.search) {
    query = query.ilike("title", `%${filter.search}%`);
  }

  const { data, error } = await query;
  if (error) throw new Error(error.message);

  return data as UsedGoodsWithPost[];
}
```

여기서 배워갈 점이 몇 가지 있습니다.

- **서버용 클라이언트는 `@/utils/supabase/server`의 `createClient()`** 를 씁니다. `await`가 붙는 비동기 함수예요(쿠키를 읽어 로그인 세션을 붙이기 때문). 서버/브라우저 클라이언트 차이는 맨 아래에서 한 번 더 정리합니다.
- **`select`는 관계까지 같이 가져옵니다.** `posts`를 기준으로 `used_goods(*)`(중고거래 상세 정보)와 `users(...)`(작성자)를 한 번에 조인해서 받아요. 필요 없는 컬럼까지 `*`로 다 긁기보다, 화면에서 쓰는 컬럼만 적어주면 응답이 가볍습니다.
- **에러는 `if (error) throw`로 즉시 던집니다.** 그래야 호출하는 쪽에서 데이터가 항상 정상이라고 믿고 쓸 수 있어요. 반환 타입(`Promise<UsedGoodsWithPost[]>`)을 명시하는 것도 같은 이유입니다 — 받는 쪽이 모양을 알 수 있으니까요.

그리고 페이지에서 이 함수를 호출합니다.

```tsx
// src/app/(user)/usedgoods/page.tsx
const [posts, { likedIds, currentUserId }] = await Promise.all([
  getUsedGoodsList({
    search: params.search,
    category: params.category,
    location: params.location,
  }),
  getLikeList("used_goods"),
]);
```

서로 의존하지 않는 두 요청(상품 목록 / 내 찜 목록)을 **`Promise.all`로 동시에** 돌리는 점을 눈여겨봐 주세요. 하나 끝나고 다음을 부르면 느려지지만, 같이 돌리면 둘 중 느린 쪽 시간만 걸립니다. 서로 결과를 필요로 하지 않을 때 쓰는 작은 습관입니다.

### 필터/검색은 URL로

검색·카테고리·페이지 번호는 컴포넌트 상태가 아니라 **URL 쿼리스트링(`searchParams`)** 으로 다룹니다.

```tsx
const params = await searchParams;          // { search, category, page ... }
const currentPage = Number(params.page ?? 1);
```

이렇게 하면 검색 결과 URL을 그대로 공유·새로고침해도 같은 화면이 나오고, 뒤로가기도 자연스럽습니다. 필터 UI(`UsedGoodsFilters`)는 입력값을 URL에 반영하고, 페이지는 그 URL을 읽어 다시 그리는 구조라고 보면 됩니다.

## 3. 목록 → 상세 한 바퀴

### 목록: 데이터를 카드로

페이지가 받아온 `posts`를 클라이언트 컴포넌트 `UsedGoodsList`에 props로 넘기고, 거기서 공통 `ContentCard`로 그립니다.

```tsx
// src/app/(user)/usedgoods/_components/UsedGoodsList.tsx
"use client";

export function UsedGoodsList({ initialPosts, initialLikedIds, currentUserId }: Props) {
  const likedSet = new Set(initialLikedIds);

  if (initialPosts.length === 0) {
    return <EmptyState message="등록된 상품이 없습니다" description="첫 번째 판매자가 되어보세요!" />;
  }

  return (
    <div className="grid grid-cols-2 md:grid-cols-3 lg:grid-cols-4 gap-3 sm:gap-4">
      {initialPosts.map((post) => {
        const goods = post.used_goods?.[0];
        return (
          <ContentCard
            key={post.id}
            href={`/usedgoods/${post.id}`}
            images={/* ... */ []}
            title={post.title}
            price={goods?.price ?? 0}
            createdAt={post.created_at}
            likeCount={post.like_count ?? 0}
            initialIsLiked={likedSet.has(post.id)}
            currentUserId={currentUserId}
            postId={post.id}
          />
        );
      })}
    </div>
  );
}
```

포인트.

- 카드 클릭 시 가는 곳은 `href={`/usedgoods/${post.id}`}` — 상세 페이지의 `[id]`로 그대로 이어집니다.
- 데이터가 비었을 때를 `EmptyState`로 처리합니다. **빈 상태·로딩·에러를 빠뜨리지 않는 습관**을 들이면 좋아요. 사용자는 "왜 화면이 텅 비었지?" 하고 헷갈리지 않습니다.
- `goods?.price ?? 0`처럼 옵셔널 체이닝과 기본값을 챙깁니다. 조인 결과가 비어 있을 수 있어서, `undefined`로 화면이 깨지지 않게 막아두는 거예요.

### 상세: `[id]`에서 id 받아 다시 조회

상세 페이지는 URL의 `id`를 받아 그 게시글 하나를 조회합니다.

```tsx
// src/app/(user)/usedgoods/[id]/page.tsx
export default async function UsedGoodsDetailPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const data = await getUsedGoodsItem(Number(id));

  if (!data) {
    return <div>상품을 찾을 수 없습니다.</div>;
  }
  // 작성자/찜 여부 등 추가 조회 후 클라이언트 컴포넌트로 전달
  return <UsedGoodsDetail data={data} /* ... */ />;
}
```

`params`도 `searchParams`처럼 `await`로 풉니다(Next.js 최신 버전 규칙). URL에서 온 `id`는 문자열이라 `Number(id)`로 바꿔 넘기고요. 데이터가 없으면 바로 "찾을 수 없습니다"를 보여줍니다 — 없는 id를 직접 쳐서 들어오는 경우를 막아주는, 빠뜨리기 쉬운 처리입니다.

여기까지가 한 바퀴입니다. **목록 페이지(서버에서 읽기) → 카드(클라이언트로 그리기) → 클릭 → 상세 페이지(id로 다시 읽기).** 방렌트·구인구직도 폴더와 함수 이름만 다를 뿐 같은 모양이라, 한 번 손에 익으면 그대로 복제할 수 있습니다.

## 4. 상호작용(쓰기) — 낙관적 업데이트 + 롤백

읽기는 서버에서 했고, 이제 사용자가 버튼을 눌러 데이터를 바꾸는 쪽입니다. **찜하기(LikeButton)** 가 우리 표준 패턴을 가장 깔끔하게 보여줍니다.

```tsx
// src/components/common/LikeButton.tsx
"use client";

export function LikeButton({ postId, initialIsLiked, currentUserId }: LikeButtonProps) {
  const [isLiked, setIsLiked] = useState(initialIsLiked);

  const handleClick = async (e: React.MouseEvent) => {
    e.preventDefault();
    e.stopPropagation();

    if (currentUserId === null) {
      setShowModal(true);   // 비로그인이면 로그인 유도 모달
      return;
    }

    const wasLiked = isLiked;        // 1) 이전 값을 기억
    setIsLiked(!wasLiked);           // 2) 화면을 먼저 바꿈 (낙관적)

    const { error } = await toggleLike(postId, currentUserId, wasLiked);
    if (error) setIsLiked(wasLiked); // 3) 실패하면 원래대로 되돌림 (롤백)
  };
  // ...
}
```

이 세 박자가 핵심입니다. **기억 → 먼저 반영 → 실패 시 원복.**

- 화면을 먼저 바꾸는 이유부터 봅시다. 서버 응답을 기다렸다가 하트를 칠하면, 네트워크가 조금만 느려도 버튼이 굼떠 보입니다. 그래서 "성공할 거라 낙관하고" 화면을 즉시 바꿔 반응을 빠르게 만듭니다.
- 문제는 저장이 실패할 때입니다. 낙관은 어디까지나 낙관이라, 저장이 실패할 수 있습니다. 그때 화면을 그대로 두면 "하트는 칠해졌는데 DB에는 안 들어간" 상태로 어긋나요. 그래서 실패하면 기억해 둔 이전 값으로 되돌립니다. 이 롤백이 빠지면 화면과 데이터가 따로 노는 버그가 생깁니다.

상호작용의 또 하나 포인트는 **비로그인 분기**입니다. `currentUserId === null`이면 쓰기를 시도하지 않고 로그인 모달을 띄웁니다. 쓰기 동작은 항상 "이 사람이 할 수 있는 동작인가?"를 먼저 확인한다고 생각해 주세요.

> 실제 쓰기 쿼리는 `src/services/likeToggle.ts`의 `toggleLike`가 맡습니다. 여기서는 **브라우저용 클라이언트(`@/lib/supabase`)** 를 씁니다. 사용자의 클릭에 반응하는 코드라 브라우저에서 도니까요. 읽기(서버)와 쓰기(브라우저)에서 쓰는 클라이언트가 다르다는 점, 6장에서 한 번 더 짚겠습니다.

상태가 좀 더 복잡한 상호작용(폼 여러 칸, 이미지 업로드 등)은 로직을 **`src/hooks/`의 커스텀 훅**으로 빼서 컴포넌트를 가볍게 둡니다. 예로 글쓰기 폼은 `src/hooks/usedgoods/useCreateUsedGoodsForm.ts`에 상태와 제출 로직이 모여 있어요. 화면(JSX)과 로직(훅)을 분리하면 둘 다 읽기 쉬워집니다.

## 5. 공통 컴포넌트 · routes 상수 · 커밋 컨벤션

### 공통 컴포넌트는 새로 만들기 전에 먼저 찾아보기

`src/components/common`에 이미 많은 조각이 있습니다. 화면을 만들다 "카드", "빈 상태", "찜 버튼", "이미지(폴백 포함)" 같은 게 필요하면 **새로 만들기 전에 여기부터 봐주세요.**

- `ContentCard` — 목록 카드(이미지·제목·가격·찜 버튼까지 포함)
- `EmptyState` — 데이터 없을 때 안내
- `LikeButton` — 찜하기(낙관적 업데이트 내장)
- `ImageWithFallback` — 이미지 깨질 때 대체

중고거래 카드와 방렌트 카드를 각자 만들면 스타일이 미묘하게 달라지고 고칠 곳이 두 배가 됩니다. 공통 컴포넌트에 props로 차이를 넘기는 쪽이 일관성과 유지보수 둘 다 좋아요.

### 경로는 문자열 대신 ROUTES 상수

경로를 코드 여기저기에 문자열로 박지 말고 `src/constants/routes.ts`를 씁니다.

```ts
// src/constants/routes.ts
export const ROUTES = {
  home: "/main",
  usedgoods: "/usedgoods",
  jobs: "/job",
  rental: "/rental",
  login: "/login",
  favorites: "/favorites",
  // ...
} as const;
```

```tsx
import { ROUTES } from "@/constants/routes";
<Link href={ROUTES.usedgoods}>중고거래</Link>
```

경로가 바뀌어도 상수 한 곳만 고치면 전부 따라오고, 오타로 잘못된 URL을 적는 실수도 줄어듭니다. 새 화면을 추가하면 여기에도 항목을 더해주세요. (지금 코드엔 `/usedgoods/${post.id}` 같은 동적 경로는 템플릿 문자열로 들어간 곳도 있는데, 고정 경로만이라도 `ROUTES`로 통일하는 방향이 좋습니다.)

### 커밋·브랜치 컨벤션

팀이 합의한 규칙입니다. README의 커밋 컨벤션 섹션에 정리돼 있어요.

- **커밋:** `<type>: <설명>` — 예 `feat: 중고거래 목록 페이지 구현`
  - 타입: `feat`(기능) / `fix`(버그) / `refactor`(구조 개선) / `chore`(설정·잡무) / `bug`(버그 추적)
  - 제목 50자 이내, 명령형("추가" O, "추가함" X)
- **브랜치:** `<타입>/<작업내용>` — 예 `feature/usedgoods-list`, `fix/like-rollback`
- **통합 브랜치는 `develop`** 입니다. 보호 브랜치라 직접 push가 막혀 있으니, 작업 브랜치를 따서 **PR로** 머지합니다.

PR을 올릴 땐 `.github/PULL_REQUEST_TEMPLATE.md`가 자동으로 뜹니다. 특히 **"새 패턴 / 리팩토링" 칸** — 새 훅이나 컴포넌트 구조를 도입했다면 여기 한 줄 적어주세요. 팀원이 같은 패턴을 재사용할 수 있게 하는 게 목적입니다.

## 6. 새 기능 추가 체크리스트

같은 패턴으로 새 화면(예: 새 카테고리 목록)을 만들 때 밟을 순서입니다. 위 중고거래 슬라이스를 그대로 본떠 가면 됩니다.

1. **services 함수부터.** `src/services/<도메인>/<도메인>.ts`에 `getXxxList`, `getXxxDetail`을 만든다. 서버 클라이언트(`@/utils/supabase/server`)로 쿼리하고, 반환 타입을 명시하고, 에러는 `throw`.
2. **목록 페이지.** `src/app/(user)/<도메인>/page.tsx`를 `async` 서버 컴포넌트로 만들고, 1의 함수를 호출(여러 요청이면 `Promise.all`). 검색·페이지는 `searchParams`로.
3. **목록/카드 컴포넌트.** `_components/`에 클라이언트 컴포넌트를 만들고, 가능하면 공통 `ContentCard`·`EmptyState`를 재사용. 빈 상태 처리 잊지 않기.
4. **상세 페이지.** `<도메인>/[id]/page.tsx`에서 `params`로 id를 받아 `getXxxDetail` 호출. 데이터 없을 때 처리.
5. **상호작용(쓰기).** 필요하면 `"use client"` 컴포넌트에서 처리. 화면을 바꾸는 쓰기는 **기억 → 낙관적 반영 → 실패 시 롤백** 패턴(LikeButton 참고). 비로그인 분기 챙기기.
6. **경로/공통 정리.** 새 경로는 `ROUTES`에 추가. 공통으로 쓸 만한 조각은 `components/common`으로.
7. **확인하고 PR.** `npx tsc --noEmit`로 타입 에러 0, `npm run dev`로 화면 확인. 브랜치 `<타입>/<작업내용>` → 커밋 `<type>: <설명>` → `develop`으로 PR(템플릿의 "새 패턴" 칸 작성).

## (선택) 서버 클라이언트 vs 브라우저 클라이언트

헷갈리기 쉬운 부분이라 한 번 정리합니다. Supabase 클라이언트가 두 개입니다.

| | 서버용 | 브라우저용 |
| --- | --- | --- |
| 위치 | `@/utils/supabase/server` | `@/lib/supabase` |
| 쓰는 법 | `const supabase = await createClient()` | `import { supabase }` (이미 생성됨) |
| 어디서 | 서버 컴포넌트·services·Route Handler | `"use client"` 컴포넌트 |
| 특징 | 쿠키를 읽어 로그인 세션을 자동으로 붙임 | 브라우저에서 동작 |

규칙은 단순합니다. **서버에서 읽을 땐 `createClient()`(await 필요), 브라우저 클릭에 반응해 쓸 땐 `@/lib/supabase`의 `supabase`.** 둘 다 환경변수(`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`)로 같은 프로젝트를 가리키지만, 세션을 다루는 방식이 달라서 자리를 구분합니다. 서버용을 클라이언트 컴포넌트에서 부르면 쿠키 API가 없어 터지고, 그 반대도 의도와 어긋나니 자리를 지켜주세요.
