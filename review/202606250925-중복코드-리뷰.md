# Kanto 중복 코드 리뷰 (2026-06-25)

이번엔 평소 리뷰와 결을 달리해서, **반복·복붙된 코드**를 한 번 훑어봤습니다. 같이 첨부한 「리팩토링 가이드」가 "언제 빼고 언제 두는지"의 원칙이라면, 이 문서는 그 원칙을 Kanto 코드에 실제로 대입해 본 결과입니다. 핵심부터 말하면, **무조건 다 합치라는 얘기가 아닙니다.** 지금 빼면 명확히 이득인 것과, 아직 둬도 되는 것을 나눠서 정리했습니다.

먼저 칭찬할 부분. 가장 많이 복붙되기 쉬운 **Supabase 클라이언트 생성은 이미 잘 모아두셨습니다.** 서버는 `src/utils/supabase/server.ts`, 브라우저는 `src/lib/supabase.ts` 싱글톤으로 통일해서 다들 import해 쓰고 있어요. 이건 중복이 아니라 모범 사례라 그대로 두시면 됩니다. realtime 훅을 따로 뺀 것도 좋고요. 그래서 아래 지적은 대부분 "인프라는 잘 모았는데, 그 위 라벨·유틸 레이어가 흩어졌다"는 양상입니다.

## 1. [필수] 상수 라벨이 갈라져서 이미 버그가 났습니다 — `POST_TYPE_LABEL`

게시글 카테고리 라벨 맵이 세 곳에 복사돼 있는데, **시간이 지나면서 한쪽만 수정돼 값이 어긋났습니다.**

- `src/app/(admin)/admin/_lib/constants.ts:3` (export, 공용으로 의도된 곳)
- `src/app/(admin)/admin/users/[id]/_components/AdminUsersPosts.tsx:26` (로컬 재정의)
- `src/services/admin/adminPosts.ts:15` (로컬 재정의)

```ts
// _lib/constants.ts, adminPosts.ts 에서는
rental: "방 렌탈",
// AdminUsersPosts.tsx 에서는
rental: "방 렌트",   // ← 같은 카테고리인데 표기가 다릅니다
```

지금 어느 화면을 통해 들어오느냐에 따라 "방 렌탈"이 보이기도 하고 "방 렌트"가 보이기도 합니다. 이게 중복이 버그로 변하는 전형적인 순간이에요. 복사할 땐 분명 똑같았는데, 코드는 시간이 지나면 한쪽만 수정되거든요. 이미 `_lib/constants.ts`에 export 버전이 있으니, 나머지 두 곳은 로컬 정의를 지우고 그걸 import하면 끝납니다. 라벨·상태값 같은 "하나의 약속"은 한 곳에만 둬야 합니다.

같은 양상이 `SANCTION_LABEL`(제재 라벨)에도 있습니다. `reports/_lib/constants.ts:29`(export), `services/admin/adminUsers.ts:4`(여기엔 `lifted` 키가 더 있음), `posts/_components/PostDetailDrawer.tsx:277`(인라인) 세 곳으로 갈라져 있고, `lifted` 포함 여부가 달라요. 이것도 한 곳으로 모으면서 키를 한 번 정리하시면 좋겠습니다.

## 2. [제안] 공용 유틸이 있는데도 `formatDate`를 8군데서 다시 만들었습니다

이미 `src/utils/formatTime.ts`라는 날짜·시간 유틸 모듈이 있는데, admin 컴포넌트들이 거길 안 쓰고 똑같은 `formatDate`/`formatDateTime`을 각자 재정의하고 있습니다.

- `AdminPostsTable.tsx:25`, `AdminPostsCard.tsx:19`, `PostDetailDrawer.tsx:31,38`, `AdminUsersTable.tsx:12`, `UserDetailDrawer.tsx:16,23`, `ReportsClient.tsx:858`

```ts
function formatDate(date: string | null) {
  if (!date) return "-";
  return new Intl.DateTimeFormat("ko-KR", {
    year: "numeric", month: "2-digit", day: "2-digit",
  }).format(new Date(date));
}
```

여섯 파일에 거의 글자 단위로 같은 함수가 들어 있어요. 이건 "추출을 못 해서"가 아니라 `formatTime.ts`가 있다는 걸 발견하지 못해서 생긴 일입니다. `formatTime.ts`에 `formatDate`/`formatDateTime`을 추가하고 8군데를 import로 바꾸면 됩니다. 포매터는 빼는 비용이 거의 0이라 망설일 이유가 없는 1순위입니다.

## 3. [제안] 가격 표기 `₱ ${...toLocaleString()}`가 12곳에 흩어져 있습니다

- `rental/[id]/page.tsx:50`, `usedgoods/[id]/_components/UsedGoodsDetail.tsx:64,139,202`, `job/[id]/page.tsx:51`, `JobInfo.tsx:34`, `JobCard.tsx:90,118`, `PopularJobs.tsx:45` 등

통화기호 `₱` + 공백 + `toLocaleString()` 조합을 컴포넌트마다 손으로 반복하고 있는데, 한 곳은 `null`일 때 "가격 협의"로 처리하고 다른 곳은 안 합니다. `formatPrice` 유틸 하나로 모으면 이 폴백 처리가 일관돼지고, 나중에 통화 단위가 바뀔 때 한 곳만 고치면 됩니다.

```ts
export const formatPrice = (v?: number | null) =>
  v == null ? "가격 협의" : `₱ ${v.toLocaleString()}`;
```

## 4. [제안] 색상/배지 맵도 `_lib/constants.ts`로 모으면 좋겠습니다

`CATEGORY_STYLE`(카테고리별 배경/글자색)이 `AdminPostsTable.tsx:18`, `PostDetailDrawer.tsx:19`에 hex 값까지 똑같이 들어 있고, `POST_STATUS_STYLE`·신고 상태 색도 비슷하게 흩어져 있습니다. 색상은 디자인 토큰 성격이라 한 곳에 모으면 일관성에 도움이 됩니다. 다만 `reports/_lib/constants.ts:13`은 키가 한글 라벨이라 구조가 다르니, 합칠 때 키 통일이 따라옵니다. **`_lib/constants.ts`로 모으는 정도면 충분하고 과한 추상화는 불필요**합니다.

## 5. [제안] 인증번호 모달 두 개가 거의 쌍둥이입니다 — 로직만 훅으로

`login/_components/FindPasswordModal.tsx`와 `profile/_components/IdentityVerificationModal.tsx`가 카운트다운 타이머, ESC 닫기, fetch 에러 처리 골격까지 똑같습니다.

```ts
const INITIAL_SECONDS = 180;
function formatTime(seconds: number) {
  const minutes = Math.floor(seconds / 60);
  const restSeconds = seconds % 60;
  return `${minutes}:${String(restSeconds).padStart(2, "0")}`;
}
```

두 모달은 후속 동작(비번 재설정 vs 본인인증)이 달라서 **UI 껍데기까지 무리해서 합치진 마시고**, 타이머·인증 로직만 `useCountdown`, `useVerificationCode` 훅으로 빼는 정도가 적정선입니다.

```ts
// src/hooks/useCountdown.ts
export function useCountdown(initial: number, active: boolean) {
  const [left, setLeft] = useState(initial);
  useEffect(() => {
    if (!active || left <= 0) return;
    const id = window.setInterval(() => setLeft((p) => Math.max(p - 1, 0)), 1000);
    return () => window.clearInterval(id);
  }, [active, left]);
  return { left, reset: setLeft };
}
```

ESC 닫기(`useEffect` + keydown)는 `ConfirmModal`, `ReportModal`, `JobPreferredModal` 등 5곳에, body 스크롤 잠금은 8곳에 반복됩니다. `useEscapeKey(onClose)`, `useLockBodyScroll()` 같은 작은 훅 두 개로 빼면 깔끔한데, 이미 `useClickOutside.ts`를 만들어 두신 걸 보면 충분히 하실 수 있는 작업입니다.

## 6. [사소] 지금은 그냥 둬도 되는 것들

여기는 **빼지 마시라**는 쪽입니다. Rule of Three를 넘겼어도 둬야 좋은 경우가 있어요.

- **클라이언트 fetch 에러 보일러플레이트** (`FindPasswordModal`, `LoginForm`, `Chatbot` 등): `setLoading(true) → try/await/catch → finally` 골격이 반복되지만, 메시지와 후처리가 호출마다 달라서 억지로 `apiFetch()`로 묶으면 옵션이 늘어 오히려 복잡해집니다. 규칙이 정해지면 그때 얇은 헬퍼로 빼세요.
- **폼 `onChange → setState` 핸들러**: 각 폼의 필드·검증이 달라 전부 하나로 묶을 대상은 아닙니다. 이미 `useCreateJobForm` 등으로 일부 분리하신 건 좋아요. `useState`가 한 폼에 6~8개씩 늘어선 곳만 객체 상태 + 제네릭 `handleChange`로 가볍게 줄이는 정도면 됩니다.
- **DB row 타입의 산발적 재선언**: `src/type/supabase.ts`(생성 타입)가 있는데 일부는 거기서 파생하지 않고 손으로 적었습니다. 지금 당장 버그를 만들진 않으니 낮은 우선순위입니다. "새 타입은 `supabase.ts`의 Row에서 `Pick`하기"를 팀 컨벤션으로 안내하는 정도면 충분합니다.

## 한 줄 정리

런타임 인프라(Supabase 클라이언트·realtime)는 잘 모아두셨는데, **"라벨·색상 상수"와 "날짜·가격 포매터"가 공용 모듈을 두고도 admin 컴포넌트마다 다시 복붙되면서 값까지 어긋나기 시작한**(방 렌탈/방 렌트) 양상입니다. 우선순위는 ① 갈라진 상수 통합(1번, 필수) → ② 포매터를 `formatTime.ts`/`formatPrice`로 모으기(2·3번) → ③ 모달 로직 훅화(5번) 순서를 권합니다. 6번은 지금 두셔도 됩니다.
