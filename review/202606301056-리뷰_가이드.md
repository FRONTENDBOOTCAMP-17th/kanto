# Kanto 15차 리뷰 해결 가이드 (2026-06-30 10:56)

리뷰에서 짚은 `[필수]`들을 어떻게 고치는지 단계별로 정리했습니다. 우선순위 순서대로 갑시다.

## 1. getUsedGoodsItem 소프트딜리트 누락 (3-1)

삭제된 중고 글의 상세가 직링크로 열리는 문제입니다. 렌탈이 이미 쓰는 방식(가져온 뒤 status를 보고 막기)을 그대로 따라가면 일관성도 좋고 간단합니다.

`src/services/usedGoods/usedGoods.ts`:

```ts
export async function getUsedGoodsItem(postId: number) {
  const supabase = await createClient();

  const { data } = await supabase
    .from("used_goods")
    .select(`*, posts (*, users!posts_user_id_fkey (id, name, avatar_url, auth_id, created_at))`)
    .eq("post_id", postId)
    .single();

  // 소프트딜리트된 글은 없는 것으로 취급
  if (data?.posts && (data.posts as { status?: string }).status === "deleted") {
    return null;
  }

  return data;
}
```

그리고 이 함수를 쓰는 상세 페이지(`src/app/(user)/usedgoods/[id]/page.tsx`)에서 null이면 404로 보냅니다.

```ts
import { notFound } from "next/navigation";

const item = await getUsedGoodsItem(postId);
if (!item) notFound();
```

> 쿼리 한 방에 막고 싶으면 `.eq("posts.status", "active")` 같은 중첩 필터도 가능합니다. 다만 `used_goods`를 from으로 두고 `posts`를 조인한 구조라 중첩 테이블 필터가 까다로워서, 가져온 뒤 검사하는 위 방식이 읽기 쉽고 렌탈과도 통일됩니다.

`requireAdmin`은 관리자가 아니면 throw하므로, 일반 회원이 직접 호출하면 거기서 막힙니다. `deletePost.ts`처럼 반환값(`sessionUser`)을 받아 감사 로그에 넘기면 더 좋습니다.

> 왜 화면에 버튼이 없어도 위험한가: 서버액션은 빌드되면 내부적으로 POST 엔드포인트가 됩니다. 브라우저 콘솔이나 fetch로 직접 부를 수 있으니 UI에서 안 보이게 하는 것만으로는 막지 못합니다. 인가는 반드시 서버 함수 안에서 합니다.

## 3. cleanup 크론 fail-closed로 (3-3)

"secret이 있으면 검사"가 아니라 "secret이 없으면 거부"로 뒤집습니다.

`src/app/api/posts/cleanup/route.ts`:

```ts
export async function POST(request: Request) {
  const secret = process.env.CRON_SECRET;

  // env가 비어 있으면 동작 자체를 막는다 (fail-closed)
  if (!secret) {
    return NextResponse.json(
      { error: "CRON_SECRET not configured" },
      { status: 500 },
    );
  }

  const auth = request.headers.get("authorization");
  if (auth !== `Bearer ${secret}`) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  // ...이하 삭제 로직 그대로
}
```

이렇게 두면 배포 때 env를 깜빡해도 영구삭제가 무방비로 도는 일은 없습니다. 차라리 "설정이 안 됐다"고 500을 내는 편이 훨씬 안전합니다.

## 4. 렌탈 채팅 가격 오타 (3-4)

`src/app/api/chat/[id]/route.ts:61`, 비교 문자열만 바꾸면 됩니다.

```ts
const postPrice =
  posts?.post_type === "used_goods"
    ? (posts.used_goods[0]?.price ?? null)
    : posts?.post_type === "rental"     // "rentals" → "rental"
      ? (posts.rentals[0]?.price ?? null)
      : null;
```

`posts.rentals[0]`처럼 관계 테이블을 부를 때 쓰는 `rentals`(복수)는 그대로 둡니다 — 그건 테이블명이라 맞습니다. 바꾸는 건 `post_type`과 비교하는 리터럴뿐입니다.

## 5. (팀) post_type 상수화 — 오타 재발 방지

위 4번 같은 실수를 컴파일 타임에 막는 방법입니다. 타입 상수 파일을 하나 두고 전부 거기서 가져다 씁니다.

`src/constants/postType.ts` (새 파일):

```ts
export const POST_TYPE = {
  USED_GOODS: "used_goods",
  RENTAL: "rental",
  JOBS: "jobs",
} as const;

export type PostType = (typeof POST_TYPE)[keyof typeof POST_TYPE];
```

사용하는 쪽:

```ts
import { POST_TYPE } from "@/constants/postType";

// 비교
posts?.post_type === POST_TYPE.RENTAL

// insert
.insert({ post_type: POST_TYPE.RENTAL, ... })

// 쿼리 필터
.eq("post_type", POST_TYPE.USED_GOODS)
```

이렇게 하면 `POST_TYPE.RENTALS`처럼 오타를 내는 순간 에디터와 `tsc`가 바로 빨간 줄을 그어 줍니다. 한 번에 다 바꾸기 부담되면 새로 손대는 파일부터 점진적으로 교체해도 됩니다.
