## 0. 먼저: 인덱스가 있는지부터 확인 (코드 안 고치고 효과 큼)

Supabase 대시보드의 SQL Editor에 아래를 붙여 한번 돌려보세요.

```sql
select tablename, indexname, indexdef
from pg_indexes
where schemaname = 'public'
  and tablename in ('messages','chats','common_notifications','user_blocks','posts')
order by tablename;
```

결과에서 아래가 안 보이면 추가하면 됩니다.

```sql
-- 채팅 메시지: chat_id로 거르고 created_at로 정렬하는 모든 쿼리가 이걸 씁니다
create index if not exists idx_messages_chat_created
  on messages (chat_id, created_at desc);

-- 채팅 목록 정렬
create index if not exists idx_chats_last_message_at
  on chats (last_message_at desc);

-- 알림 목록
create index if not exists idx_notifications_receiver_created
  on common_notifications (receiver_id, created_at desc);

-- 차단 역방향 조회
create index if not exists idx_user_blocks_blocked
  on user_blocks (blocked_id);
```

> `(chat_id, created_at desc)`처럼 두 컬럼을 왜 묶느냐면, 쿼리 자체가 "chat_id로 거른 뒤 created_at 순으로 정렬"하는 모양이라서 그렇습니다. 이 순서대로 인덱스를 만들어 두면 Postgres가 인덱스를 그대로 따라가면서 정렬된 50개만 읽고 멈추는데, 인덱스가 없으면 그 방의 메시지를 다 끌어모아 메모리에서 정렬한 다음에야 50개를 잘라냅니다.

## 1. 채팅방 진입 왕복 줄이기 — `api/chat/[id]/route.ts`

지금은 `posts`와 `used_goods`를 따로따로 조회하고 있는데요, `getChatDetail`이 이미 `chats → posts`를 가져오고 있으니 거기에 자식 테이블까지 임베드로 묶어버리면 됩니다.

```ts
// chats를 가져올 때 post와 그 자식까지 한 번에
const { data: chat } = await supabase
  .from("chats")
  .select(`
    *,
    posts (
      *,
      used_goods (*),
      rentals (*),
      jobs (*)
    )
  `)
  .eq("id", chatId)
  .single();
// 이러면 별도 posts 조회, used_goods 조회(왕복 2번)가 사라집니다.
```

자식 테이블 중 실제로 값이 채워지는 건 post_type에 맞는 하나뿐이고 나머지는 `null`로 들어옵니다. 화면 쪽에서는 `chat.posts.used_goods ?? chat.posts.rentals ?? chat.posts.jobs`처럼 받아 쓰면 됩니다.

## 2. 읽음 처리 한 곳으로 모으기 — `ChatRoomClient.tsx` + `useChatMessages.ts`

지금은 `useChatMessages`가 messages를 다건 UPDATE 하고, 거기에 더해 `ChatRoomClient`가 마운트와 언마운트 양쪽에서 `markChatReadAction`을 부르고 있습니다. 한 번 입장하는데 같은 일이 여러 번 도는 셈이에요.

- 먼저 **언마운트 시 호출을 제거**하고, 마운트 시 한 번만 부르도록 합니다.
- 그리고 가능하면 "messages.is_read 갱신 + chats.unread=0"을 RPC 하나로 묶으세요.

```sql
create or replace function mark_chat_read(p_chat_id bigint, p_user_id bigint)
returns void language sql security definer as $$
  update messages set is_read = true
    where chat_id = p_chat_id and sender_id <> p_user_id and is_read = false;
  update chats
    set user_id_1_unread = case when user_id_1 = p_user_id then 0 else user_id_1_unread end,
        user_id_2_unread = case when user_id_2 = p_user_id then 0 else user_id_2_unread end
    where id = p_chat_id;
$$;
```

```ts
// 클라이언트는 한 번만
await supabase.rpc("mark_chat_read", { p_chat_id: chatId, p_user_id: myId });
```

이렇게 하면 왕복도 줄고, "messages는 읽음 처리됐는데 chats unread는 아직 그대로"인 어긋남도 같이 사라집니다.

## 4. Realtime 채팅 목록 구독에 필터 — `useChatListRealtime.ts`

```ts
// 내가 user_id_1인 방 변경만 받기 (양쪽이면 채널 2개로 분리)
channel.on(
  "postgres_changes",
  { event: "*", schema: "public", table: "chats", filter: `user_id_1=eq.${myId}` },
  handler
);
```

이렇게 서버에서 미리 걸러 보내면 사용자가 늘어도 나와 상관없는 이벤트는 안 옵니다. INSERT가 올 때마다 목록 전체를 다시 받아오는 부분도, 가능하면 넘어온 payload로 목록 상태를 직접 갱신해서 전체 재조회를 줄여보세요.

## 5. 메인 인기글 limit — `services/main/main.ts`

```ts
// 각 도메인에서 메인에 쓸 만큼만
const { data } = await supabase
  .from("posts")
  .select("*, used_goods(*), users(id,name,avatar_url)")
  .eq("post_type", "used_goods")
  .eq("status", "active")
  .order("created_at", { ascending: false })
  .limit(6);
```

전체를 다 가져와서 JS에서 자르지 말고, DB에서 `order + limit`으로 잘라 받는 게 좋습니다.
