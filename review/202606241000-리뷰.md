# Kanto 코드 리뷰 (2026-06-24)

이번엔 "Kanto GO!" 번개모임이 통째로 들어오면서 지도·실시간·관리자 모더레이션까지 한꺼번에 붙었습니다. 새 화면이 많은 만큼 사용자 동선(메인 → 목록 → 상세 → 로그인 → GO!)을 관리자 화면까지 실제로 띄워 가며 따라가 봤고, 지난 11차에서 짚었던 채팅 성능 항목이 어떻게 반영됐는지부터 봤습니다.

관리자 화면은 `(admin)/layout.tsx`가 `users.role === "admin"`인 사람만 통과시키게 막아 둔 구조라, 리뷰용 계정(`kanto-review@example.com`)을 잠깐 admin으로 올려 대시보드·GO! 관리·신고·유저·채팅·게시글 관리를 한 바퀴 돌아본 뒤 원래 권한으로 되돌려 놨습니다.

두 가지 환경 메모를 남깁니다.

- **`@vis.gl/react-google-maps`가 설치 안 된 채로 `/go`를 열면 빌드가 깨집니다.** `package.json`엔 의존성이 선언돼 있지만, 클론 직후 `npm install`을 안 하면 `/go` 전체가 모듈 못 찾음 에러로 죽더라고요(`kanto-s06-go-build-error.png`). 설치 후엔 tsc·build 모두 깨끗하게 통과합니다. CI에서 `npm ci` 후 build까지 돌리면 자동으로 걸러지는 종류라, 이 부분은 가이드에 정리해 둘게요.
- **GO! 지도에 "This page can't load Google Maps correctly" 팝업**이 떴습니다. 리뷰 환경의 `.env.local`에 Google Maps 키가 없거나, 키의 허용 도메인에서 `localhost`가 빠진 탓으로 보입니다. 운영 키·도메인 설정 쪽 문제일 수 있으니 배포 환경에서 한 번 확인해 주세요.

## 3. 팀 공통 — 이번 회차 핵심 [필수]

### 3-1. [필수] GO! 관리자 서버 액션에 권한 검사가 빠졌습니다 (가장 급함)

`src/services/go/go.ts`는 `"use server"` 파일입니다. 안에 있는 사용자용 함수들(`123`, `168`, `208`, `236`행)은 전부 `getSessionUser()`로 로그인 사용자를 먼저 확인하는데, **관리자 전용 함수 두 개만 그 검사가 없습니다.**

```ts
// go.ts:270
export async function adminGetMeetups(): Promise<AdminMeetup[]> {
  const admin = createAdminClient();   // ← 바로 service-role 클라이언트, 권한 확인 없음
  ...
}

// go.ts:347
export async function adminForceEndMeetup(postId: number): Promise<void> {
  const admin = createAdminClient();   // ← 여기도 권한 확인 없음
  const { error } = await admin
    .from("posts")
    .update({ status: "inactive" } as never)
    .eq("id", postId);                 // ← 넘어온 postId를 그대로 비활성화
  ...
}
```

여기서 짚고 싶은 건 **Server Action이 `(admin)/layout.tsx` 가드를 거치지 않는 별도의 엔드포인트**라는 점입니다. 레이아웃 가드는 "관리자 화면을 그려줄지"만 결정할 뿐, 서버 액션은 그 화면을 거치지 않고도 직접 호출될 수 있으니까요. 그래서 지금 구조에서는 로그인만 한 일반 회원이 `adminForceEndMeetup`을 직접 불러 **아무 게시글이나 `postId`만 넣어 강제 비활성화**할 수 있습니다. 10차에서 짚었던 "관리자 서버 액션 role 재검증 누락"이 그대로 남은 데다 새로 만든 GO! 액션에도 같은 패턴이 반복된 거라, 이번엔 꼭 닫고 갑시다.

처방은 가이드에 `requireAdmin()` 헬퍼를 만들어 두 함수 맨 앞에서 부르는 형태로 정리해 두겠습니다. 코드베이스 전체에 아직 `requireAdmin`이 없어서(검색 0건), 한 번 만들어 두면 앞으로 모든 관리자 액션에 한 줄로 붙일 수 있습니다.

### 3-2. [필수] 비로그인 상태에서 회원 이메일과 admin 여부가 그대로 읽힙니다

리뷰용 publishable(anon) 키로 직접 호출해 보니 `users` 테이블이 이렇게 열립니다.

```
GET /rest/v1/users?select=id,email,role
→ [{"id":180,"email":"dlehdrms6896@daum.net","role":"admin"},
   {"id":200,"email":"mogoa97@daum.com","role":"user"}, ...]
```

로그인도 안 한 상태에서 **전 회원의 이메일이 노출되고, 심지어 누가 `admin` 계정인지까지 드러납니다.** 7차 때는 anon으로 `users`가 0행이라 닫혔다고 확인했던 부분인데 이번에 다시 열렸습니다. 그 사이 망고지수(KTS) 같은 공개 지표 때문에 `users` 조회를 열어야 했던 사정이 있었는지 궁금합니다. 만약 랭킹/점수를 공개로 보여주려고 푼 거라면, **공개에 필요한 컬럼(닉네임·점수·등급)만 노출하는 뷰를 따로 만들고** `email`·`role`·`auth_id`·`phone` 같은 민감 컬럼은 anon에서 다시 막는 방향이 맞습니다. 이메일이 통째로 긁히는 건 시연·배포 전에 반드시 닫아야 하는 부분이라 [필수]로 둡니다.
