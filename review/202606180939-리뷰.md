# Kanto 8차 코드 리뷰 (2026-06-18)

지난 7차(2026-06-17) 이후 `develop`에 새로 들어온 변경을 봤습니다. 이번 회차의 큰 줄기는 **안전결제(Xendit)**, **프로필 본인인증**, **계정 정지/이용 제한**, **관리자(신고·회원 관리) 화면** 네 가지입니다. 먼저 앱을 띄워 사용자·관리자 흐름을 순서대로 돌려 보고(아래 "E2E 결과"), 거기서 드러난 동작을 근거로 코드를 봤습니다.

좋은 소식부터 말씀드리면, 7차에서 제일 급했던 세 가지 — 비로그인 상세 크래시, 찜 403, 빌드 오류 더미 — 가 이번에 대부분 정리됐습니다. 다만 국제화 작업이 들어오면서 **프로필 페이지가 통째로 깨지는** 새 문제가 하나 생겼네요. 이번 회차에서 제일 먼저 손봐야 할 부분이라 생각합니다.

## 2. 이번 회차에서 가장 급한 것 — `/profile` 런타임 크래시 [필수]

로그인한 사용자가 프로필 페이지(`/profile`)에 들어가면 화면이 곧바로 에러로 덮입니다. E2E에서 잡힌 실제 메시지는 이렇습니다.

```
Runtime ReferenceError: PROVIDER_KEYS is not defined
  src/app/(user)/profile/_components/ProfileCard.tsx (37:23) @ ProfileForm
```

해당 위치를 보면 두 가지가 함께 깨져 있습니다(`ProfileCard.tsx`).

```tsx
// 18행 — formatTime 에 더 이상 존재하지 않는 이름을 import (TS2305)
import { formatSellerInfoCreatedAt } from "@/utils/formatTime";

// 37행 — 어디에도 정의/ import 되지 않은 PROVIDER_KEYS 사용 (ReferenceError)
const providerKey = PROVIDER_KEYS.find((p) => p === user.provider) ?? "email";
```

국제화(next-intl) 작업이 들어오면서 `ProfileCard`가 `useTranslations("Profile.providers")`를 쓰도록 바뀌었는데, 그 과정에서 예전에 있던 `PROVIDER_KEYS` 상수와 `formatSellerInfoCreatedAt` 유틸이 사라지거나 이름이 바뀐 것으로 보입니다. 컴포넌트는 여전히 옛 이름을 부르고 있어서, 로그인 사용자가 프로필을 열 때마다 크래시가 납니다. 프로필은 모든 사용자가 들르는 핵심 화면이라 영향이 큽니다. 게다가 이번에 새로 들어온 본인인증 모달도 이 페이지 안에 있어서 함께 막혀 버리네요.

`p` 매개변수의 implicit any(TS7006)까지 합치면 ProfileCard 한 파일에서 tsc 3건이 나오는데, 이게 곧 화면 크래시의 정체입니다. `PROVIDER_KEYS`를 다시 정의(또는 상수 모듈에서 import)하고 18행의 죽은 import를 제거하면 됩니다. 구체적인 방법은 함께 드리는 가이드에 적어 두었습니다.

> 왜 tsc가 빨간불인데 화면은 "200"으로 떴을까요? 서버는 페이지 셸을 200으로 내려보내지만, 브라우저에서 클라이언트 컴포넌트가 렌더되며 `PROVIDER_KEYS`를 부르는 순간 터집니다. HTTP 상태만 보면 멀쩡해 보여도 실제 화면은 깨져 있는 거죠. "E2E로 화면을 직접 봐야 하는" 이유이자, "tsc를 머지 게이트로 걸어야 하는" 이유이기도 합니다.

## 3. 빌드(tsc) — 4건으로 줄었지만 아직 0은 아닙니다 [필수]

`npx tsc --noEmit` 결과입니다(7차 23건 → 8차 4건).

```
src/app/(admin)/layout.tsx(95,53)   TS2339  Property 'badge' does not exist ...
src/app/(user)/profile/_components/ProfileCard.tsx(18,10)  TS2305  formatSellerInfoCreatedAt 없음
src/app/(user)/profile/_components/ProfileCard.tsx(37,23)  TS2552  PROVIDER_KEYS 없음
src/app/(user)/profile/_components/ProfileCard.tsx(37,43)  TS7006  Parameter 'p' implicit any
```

ProfileCard 3건은 2장에서 다뤘고, 남은 한 건은 어드민 사이드바입니다. `(admin)/layout.tsx`의 `navItems` 배열에 어떤 항목은 `badge`가 있고 어떤 항목은 없다 보니, `badge`가 없는 유니언 타입에서 구조분해(`{ ..., badge }`)로 꺼내려다 막힌 경우네요. 런타임에는 `undefined`라 화면이 깨지진 않지만 tsc와 프로덕션 빌드는 막습니다. 모든 항목이 `badge?: number`를 갖는 공통 타입을 하나 만들어 배열에 붙이면 깔끔하게 풀립니다(가이드 참고).

옛 오류 더미를 이만큼 걷어낸 건 분명한 진전이라 생각합니다. 남은 4건만 닫으면 7라운드 넘게 빨갛던 빌드가 드디어 초록불이 됩니다.

## 4. CI 게이트 — 같은 이야기를 다시 드리는 이유 [필수]

매 회차 빌드가 깨지는 근본 원인은 머지 전에 tsc/build를 자동으로 돌려 막는 장치가 없다는 점이라 생각합니다. 이번 프로필 크래시가 딱 그 증거네요. 국제화 PR이 들어올 때 `tsc`가 통과해야 머지되는 규칙이 있었다면, `PROVIDER_KEYS is not defined`는 그 PR 단계에서 빨간불로 걸려 main(develop)까지 오지 못했을 겁니다.

혹시 기능 마감이 급해서 CI 세팅을 미뤄둔 상황일까요? 그렇다면 충분히 이해합니다. 다만 지금은 그 미룸의 비용이 매 회차 누군가 빌드를 도로 고치는 시간으로 계속 나가고 있어서, 한 번 깔아 두면 그 시간이 통째로 사라집니다. GitHub Actions로 `tsc --noEmit` + `next build`를 PR에서 돌리고, 그 체크를 develop 브랜치 보호 규칙의 required check로 거는 것까지가 한 세트입니다. 바로 붙일 수 있는 워크플로 yml을 가이드에 넣어 두었습니다.

## 5. 새 엔드포인트 리뷰 (이번 회차 핵심)

새로 생긴 프론트엔드·서버 엔드포인트는 빠짐없이 봤습니다. 전반적으로 설계 수준이 한 단계 올라왔다는 인상이네요.

### 5-1. 안전결제 — `/payment/return` 페이지 · `/api/payment/xendit/webhook` (THLIMM)

결제처럼 비동기·외부 연동이 얽히는 곳은 같은 일이 두 번 일어나도 괜찮은가, 즉 멱등성이 제일 중요한데 그 점을 잘 잡았습니다.

- **webhook**(`route.ts`): `x-callback-token`을 `XENDIT_CALLBACK_TOKEN`과 비교해 검증하고, `transaction.status === "pending"`일 때만 상태를 전환합니다. 중복 콜백이 와도 두 번 처리되지 않게 막은 점이 좋네요. 부가 UX인 시스템 메시지 발송이 실패해도 webhook은 200으로 응답해(재전송 안 유발) 본 처리와 부가 처리를 분리한 것도 정확한 판단이라 생각합니다.
- **return 페이지**(`page.tsx`): webhook이 닿지 않는 로컬 환경을 대비해 인보이스 상태를 직접 재조회해 거래를 보정합니다. `Number(txn)` + `Number.isInteger` 가드에, webhook과의 경합 가능성을 주석으로 인지해 둔 것까지, 결제 흐름을 제대로 이해하고 짠 코드라 생각합니다.

E2E로 `/payment/return`(pending)·`/payment/return?failed=1`(failed) 상태 화면이 정상 렌더되는 것까지 확인했습니다. 실제 결제 왕복은 Xendit 키가 필요해 강사 환경에서 끝까지 돌리진 못했고, 그 한계는 그대로 적어 둡니다.

- [제안] `transaction.ts`의 `postSystemMessage`는 채팅 unread 카운트를 읽고 +1 해서 쓰는 비원자적 갱신입니다. 채팅 설계 문서에서 1:1 구조라 허용한다고 정리해 둔 맥락과 같은 선상이라 당장 버그는 아니지만, 결제 메시지와 일반 메시지가 동시에 들어오면 카운트가 한 번 누락될 수 있습니다. 나중에 RPC 원자 증가로 합치면 결제·채팅 양쪽이 같이 깔끔해집니다.

### 5-2. 본인인증 — `/api/profile/verification` (dongkeun)

POST(코드 발송)·PATCH(코드 검증) 구조가 또렷하고, 좋은 판단이 여럿 보입니다. 인증 코드를 Redis에 TTL 180초로 저장하되 Redis가 없으면 메모리 Map으로 폴백하고, `auth.getUser()`로 요청자를 먼저 검증하고, 이름·이메일·코드의 빈 값 검증도 빠짐없이 합니다. 발송이 실패하면 저장한 코드를 지워(orphan 방지) 깔끔하게 롤백하는 것도 좋네요.

- [제안] 이메일(Resend)이 설정 안 된 환경에서 응답에 `devCode`(실제 인증번호)를 그대로 내려보냅니다(159행). 로컬 개발 편의로는 유용하지만, 운영에서 혹시 `RESEND_API_KEY`가 비어 배포되면 인증번호가 클라이언트로 노출됩니다. `process.env.NODE_ENV !== "production"`일 때만 `devCode`를 포함하도록 한 줄 가드를 두면 안전합니다.
- [사소] `memoryVerificationCodes` Map은 모듈 스코프라, Vercel처럼 인스턴스가 여러 개로 뜨는 환경에선 POST를 처리한 인스턴스와 PATCH를 처리한 인스턴스가 달라 코드가 안 맞을 수 있습니다. Redis가 붙어 있으면 실제로는 Redis가 진실원본이라 문제되지 않지만, 메모리 폴백은 단일 인스턴스(로컬) 전용이라는 점만 주석으로 남겨두면 다음 사람이 헷갈리지 않습니다.

### 5-3. 관리자 화면 — `/admin` · `/admin/reports` · `/admin/users(/[id])` (THLIMM)

이번 회차에서 가장 완성도가 높았던 부분이라 생각합니다. 대시보드는 총 회원/활성/신규/게시글 지표, 신규 가입 추이 그래프, 카테고리 분포 도넛, 지역별 신규글, 인기글 Top5, 신고 처리 큐와 유형 분포까지, 운영자가 한눈에 볼 정보가 잘 짜여 있네요(E2E K20). 신고 관리·회원 관리·회원 상세도 순서대로 정상 동작했습니다(K21·K22).

- `adminUsers.ts`의 `getAdminUsers`는 신고 수를 앱 쪽에서 `Map`으로 집계하는데, 지금 데이터 규모에선 충분합니다. 다만 회원·신고가 커지면 신고 수는 DB 집계(뷰나 RPC)로 내리는 게 좋아지는 시점이 옵니다. 지금 당장 고칠 필요는 없고 기억만 해두면 됩니다.
- [사소] 어드민 사이드바 `badge` 타입(3장)만 정리하면 이 영역은 빌드까지 깨끗해집니다.

### 5-4. 계정 정지/이용 제한 (soyu)

`SuspendedModal`·`SuspendedBanner`·`useSuspended`에 미들웨어까지, 정지된 사용자를 막는 흐름을 컴포넌트·훅·미들웨어로 나눠 둔 구성이 좋네요. 이번엔 테스트 계정이 정상 상태라 정지 계정으로 끝까지 돌려보진 못했습니다. 다음 회차에 정지 계정을 하나 만들어 배너·모달·차단까지 화면으로 확인하겠습니다.

## 6. 팀 공통 / 협업

- **국제화(next-intl) 도입은 큰 작업이었고 방향도 좋습니다.** 다만 이번처럼 여러 컴포넌트의 상수·유틸을 함께 옮기는 리팩터는 한 군데(ProfileCard)가 옛 이름을 부르는 순간 바로 런타임 크래시로 이어집니다. 이런 광범위 리팩터일수록 머지 전 tsc가 중요한데, 그 안전망(4장 CI)이 없어서 깨진 채로 develop까지 와버렸습니다. 기능과 안전망은 같이 가야 한다고 생각합니다.
- 작성자 분포는 이번에도 4명에 고루 퍼져 있습니다(결제·어드민 THLIMM, 본인인증·프로필 dongkeun, 정지 soyu 등). 좋은 흐름이네요.
- 커밋 컨벤션(`feat:`/`fix:`/`chore:`)도 잘 지켜지고 있습니다. `chor:` 오타가 몇 건 보이는데(예: "chor: 국제화 …"), 사소하지만 통일하면 좋겠습니다.
