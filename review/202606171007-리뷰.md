## 2. 이번에 새로 발견한 것 — E2E 실측 (가장 중요)

### 2-1. [필수] 상세 페이지 3종이 **비로그인 방문자에게 모두 크래시**합니다

로그아웃 상태에서 목록→상세로 들어가 봤더니 세 종류 상세가 전부 서버 런타임 에러로 죽습니다.

| 페이지 | 터지는 위치 | 코드 |
|--------|------------|------|
| 중고 상세 | `UsedGoodsDetail.tsx:43` | `const accession = data.posts.users.created_at` |
| 구인 상세 | `job/[id]/page.tsx:38` | `authorAuthId={job.posts.users.auth_id}` |
| 방렌트 상세 | `rental/[id]/page.tsx:40` | `authorAuthId={rental.posts.users.auth_id}` |

에러 메시지는 모두 `Cannot read properties of null (reading 'auth_id' / 'created_at')`입니다.

원인을 끝까지 따라가 봤습니다.

1. 이번에 `users` RLS를 닫으면서 **anon은 자기 행이 없으니 `users`를 한 행도 못 읽습니다.**
2. 상세 페이지는 작성자 정보를 `posts(*, users(*))` 같은 **중첩 조인(embed)** 으로 가져옵니다.
3. 비로그인(anon) 요청에선 그 `users(...)` 조인이 **null** 로 돌아옵니다(실측: anon 토큰으로 `jobs?select=posts(users(...))` 호출 시 users가 비어 옵니다).
4. 그런데 페이지는 `data.posts.users.created_at`, `job.posts.users.auth_id`처럼 **옵셔널 체이닝 없이** 바로 접근하니 크래시가 납니다.

로그인하면 정상입니다. 인증 사용자는 다른 사람의 `users` 행을 읽을 수 있어서(실측: id 198 계정이 id 181 김도혁 행 조회에 성공) 조인이 채워지고, 상세가 멀쩡히 뜹니다.

- 비로그인 중고 상세(에러): `images/2026-06-17/kanto-K2b-usedgoods-detail.png`
- 비로그인 방렌트 상세(에러): `images/2026-06-17/kanto-K7-rental-detail.png`
- 비로그인 구인 상세(에러): `images/2026-06-17/kanto-K8-job-detail.png`
- 로그인 후 구인 상세(정상): `images/2026-06-17/kanto-K10b-job-detail-loggedin.png`

이게 왜 심각하냐면, **처음 들어온 손님과 검색엔진 크롤러는 전부 로그아웃 상태**이기 때문입니다. 상품/구인/방 상세는 서비스의 핵심 진입 화면인데, 로그인 안 한 사람에겐 전부 에러 화면이 뜨는 셈입니다. `[필수]`로 둡니다.

단순 버그가 아니라 **결합도 문제**라 생각합니다. `users` 테이블 하나가 "민감정보 보호(email·phone)"와 "공개 작성자 표시(name·avatar)"라는 두 책임을 함께 지고 있다 보니, 보안을 위해 통째로 닫자 공개 화면이 깨진 겁니다. 해결 방향은 가이드에 적어 뒀는데, 핵심은 **공개해도 되는 컬럼만 anon에 열어주는 것**입니다(좁은 정책 또는 공개용 뷰). 코드 쪽 응급조치로 옵셔널 체이닝과 폴백도 함께 권합니다.

### 2-2. [필수] 찜하기(좋아요)가 현재 동작하지 않습니다

로그인 상태로 중고 상세에서 찜 버튼을 눌러봤더니, 하트가 잠깐 켜졌다가 도로 꺼집니다(낙관적 업데이트 후 롤백). 왜 롤백되는지 보려고 인증 사용자 권한으로 직접 insert를 쏴봤습니다.

```
POST /rest/v1/common_likes  (인증 사용자 토큰)
→ HTTP 403
{"code":"42501","message":"new row violates row-level security policy for table \"common_notifications\""}
```

찜을 넣으면 `common_likes`에 INSERT가 일어나고, 거기에 걸린 **알림 생성 트리거가 `common_notifications`에 INSERT를 시도하다 RLS에 막혀** 트랜잭션 전체가 롤백됩니다. 그래서 찜이 저장되지 않습니다. 찜 취소(delete)는 트리거를 안 타서 204로 잘 됩니다.

`toggleLike`(`src/services/likeToggle.ts`)는 잘못이 없습니다. 문제는 DB 트리거가 만드는 알림이 RLS를 못 넘는 쪽에 있습니다. 찜 알림은 "찜한 사람"이 "글쓴이"에게 보내는 건데, 보내는 사람이 받는 사람의 알림 행을 직접 못 만들게 RLS가 막는 건 자연스러운 동작이라 생각합니다. 그래서 **알림을 만드는 트리거 함수를 `SECURITY DEFINER`로** 두는 게 표준 해법입니다. 가이드에 SQL 예시를 넣어 뒀습니다. `[필수]`.

> 참고: E2E 캡처(`kanto-K9a/K9b`)는 로그인 상태 중고 상세가 정상 렌더되는 모습입니다. 찜 토글만 위 이유로 반영이 안 됩니다.

## 4. 멤버별 — 응집도·결합도 / React·Next·TS

### 김도혁 (DoHyuk-Centric) — 이번 회차 14커밋

이번 변경의 대부분을 맡았습니다. 잘한 점부터 짚습니다.

**잘한 점 (응집 설계)**
- **`InteractionButtons` 추출** — 찜·공유·신고 세 버튼을 응집된 컴포넌트 하나로 묶어, 상세 페이지의 데스크톱/모바일 두 자리에서 재사용합니다. 같은 마크업을 두 번 쓰던 걸 한 곳으로 모은 좋은 리팩토링입니다. `aria-label`·`aria-pressed`까지 챙긴 접근성도 칭찬합니다.
- **`getUserLikeReportStatus`** — 찜 여부와 신고 여부를 한 서버 함수에서 `Promise.all`로 **병렬 조회**합니다. 두 상태를 한 책임으로 모았고, 순차 await로 늘어지지 않게 한 점도 좋습니다.
- **`view.ts`의 `increment_view_count` RPC** — 조회수를 "읽어서 +1 후 쓰기"가 아니라 **DB RPC로 원자 증가**시킵니다. 동시 조회 때 카운트가 덮어써지는 경합을 피하는 정석입니다. 작고 응집도 높은 래퍼라 마음에 듭니다.
- **`leaveChatAction`** — `"use server"` 서버 액션으로 분리하고, `getUser` 인증 체크 뒤 user_id_1/2를 판별해 `_left` 플래그를 세웁니다. 책임이 한곳에 모여 있습니다.

**결합도/설계 [제안]**
- **`InteractionButtons`의 상태가 이중화**돼 있습니다. `isLiked`는 컴포넌트 안에 있고, `likeCount`는 부모(`UsedGoodsDetail`)에 있어 `onLikeChange` 콜백으로 동기화합니다. 게다가 상세 페이지엔 이 컴포넌트가 **데스크톱용·모바일용 2개** 동시에 마운트돼 각자 자기 `isLiked`를 들고 있습니다. 지금은 한 번에 하나만 보여서 티가 안 나지만, "하나의 사실(찜 여부)"이 세 군데(부모 likeCount + 두 인스턴스의 isLiked)로 흩어진 구조입니다. `isLiked`도 부모로 끌어올려 한 곳에서 관리하거나, 반응형으로 **한 인스턴스만** 렌더하는 편이 단일 출처 원칙에 맞습니다.
- **`toggleLike`가 브라우저 클라(`@/lib/supabase`)로 DB에 직접 insert/delete**하고 `user_id`를 클라이언트에서 넘깁니다. RLS가 막아주긴 하지만, 찜 같은 쓰기는 **서버 액션**으로 옮기면 인증·트리거·검증을 서버 한곳에서 일관되게 다룰 수 있습니다(2-2의 트리거 문제도 서버에서 더 잡기 쉬워집니다).

### 4-4. [필수] RentalCreateForm — 편의시설 목록이 두 군데서 어긋남 (빌드깨짐)

이번 회차 새 빌드 에러 3건의 원인입니다. 오늘 주제인 단일 출처·결합에 딱 맞는 예라 따로 뺍니다.

`type/rental/rentalDetail.ts`에 편의시설을 **단일 출처**로 잘 정의해 뒀습니다.

```ts
export const AMENITIES = ["주차","엘리베이터","에어컨","세탁기","냉장고",
  "반려동물 허용","금고","TV","인터넷","수영장","헬스장","보안요원"] as const;
export type Amenity = (typeof AMENITIES)[number];   // 12종
```

그런데 정작 `RentalCreateForm.tsx`에서 토글 함수는 **손으로 다시 쓴 다른 목록**을 받습니다.

```ts
const toggleAmenity = (item: "와이파이" | "에어컨" | "주차" | "주방") => { ... }
```

이 4개 중 "와이파이"·"주방"은 `AMENITIES`에 없습니다. 그래서 `AMENITIES.map((item) => toggleAmenity(item))`에서 "엘리베이터" 같은 값은 4종 타입에 안 맞고, 토글 안에서 setAmenities에 "와이파이"를 넣는 것도 `Amenity`에 안 맞아 **TS2345 3건**이 납니다.

`AMENITIES`라는 단일 출처를 이미 만들어 두고 안 쓴 케이스입니다. 토글 인자 타입을 `Amenity`로 바꾸기만 하면 끝납니다.

```ts
const toggleAmenity = (item: Amenity) => { ... }
```

이렇게 하면 나중에 편의시설을 추가하거나 빼도 `AMENITIES` 한 곳만 고치면 타입이 따라옵니다. 이게 단일 출처의 이점입니다. `[필수]`(빌드 복구).

## 5. React·Next·TypeScript 종합 — 잘 쓰는 부분 / 더 볼 부분

**잘 쓰고 있는 것**
- **서버/클라이언트 컴포넌트 역할 분리**가 대체로 명확합니다. 상세 데이터 fetch는 서버에서 하고, 상호작용(찜·공유)은 `"use client"` 컴포넌트로 내립니다.
- **`as const` + 인덱스 타입으로 리터럴 유니온 파생**(AMENITIES/RENT_TYPES/ROOM_TYPES) — TS를 제대로 쓰는 패턴입니다(4-4는 그걸 안 쓴 게 문제지, 패턴 자체는 좋습니다).
- **서버 액션(`"use server"`)** 도입(`leaveChatAction`) — 채팅 같은 mutation을 서버로 옮기는 좋은 방향입니다. 이 패턴을 찜/신고에도 넓히면 좋겠습니다.
- **조회수 RPC** — 클라가 아니라 DB에서 원자적으로 증가시킵니다. Next 서버 컴포넌트와 Supabase RPC 조합을 잘 썼습니다.

**더 볼 부분 (위에서 다룬 것 요약)**
- 서버 컴포넌트에서 브라우저 클라 사용: `usedgoods/[id]/page.tsx`가 관련매물 쿼리를 `@/lib/supabase`(브라우저 싱글톤)로 합니다(1차 잔재). 서버 클라(`@/utils/supabase/server`)로 통일하세요. `[제안]`
- 렌더 중 스토어 mutate(GlobalLayout), throttle 없는 scroll 리스너(ScrollToTop), cleanup 없는 fetch effect(ChatRoom)는 4장을 참고하세요.
- 작성자 정보 접근에 옵셔널 체이닝이 일관되지 않습니다(2-1). 같은 페이지 안에서도 `users?.auth_id`와 `users.created_at`이 섞여 있습니다.

## 6. 팀 공통 — 빌드 게이트와 분담

- **빌드가 7라운드째 빨간불**이고, 매번 "새로 머지된 코드"가 원인입니다(5차 회원탈퇴/SearchBar, 6차 admin/rental import, 7차 RentalCreateForm amenities). 같은 패턴이 반복된다는 건 사람 실수가 아니라 **머지 전에 거르는 장치가 없다**는 신호라 생각합니다. `.github/workflows`가 아직 없는데, `tsc --noEmit`과 `build`를 PR 필수 체크로 거는 워크플로 하나면 이런 깨짐이 main/develop에 못 들어옵니다. 6차에 권했던 건데 아직이라 다시 올립니다. 예시 yml은 가이드에 넣어 뒀습니다.
- 작성자 분포는 이번엔 김도혁 14 / soyu 2로 한쪽에 몰렸습니다. 단정하진 않겠습니다. 이번 작업(상세 재작성·채팅 나가기)이 한 사람 영역에 모이도록 팀이 합의한 건가요, 아니면 자연스럽게 몰린 건가요? 합의한 분담이면 문제없고, 그게 아니라면 다음 스프린트에서 영역을 나눠보는 걸 권합니다.
- 좋은 신호도 있습니다. 6차의 usedgoods select 타입 에러를 `94a1b11`로 직접 고쳤고, users RLS도 닫았습니다. 리뷰 반영이 이어지고 있습니다.
