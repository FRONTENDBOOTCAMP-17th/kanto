## 2. 보안 — `users` 테이블 RLS (3차에도 재실측) `[필수]`

2차에 이어 다시 anon 키로 직접 호출했습니다. 결과가 같습니다.

```
GET /rest/v1/users?select=id,name,email,phone,auth_id  (apikey = NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY)
→ HTTP 200
  [{"id":101,"name":"김민준","phone":"010-3829-4751",...},
   {"id":178,"name":"이동근","email":"dglee6896@gmail.com","auth_id":"974f7df4-..."},
   {"id":179,"name":"임태형","email":"twin0105205@naver.com","auth_id":"1412370d-..."}]
```

publishable 키는 브라우저 JS에 그대로 노출되는 공개 키라 RLS가 유일한 방어선입니다. 지금은 **누구나 키만으로 전체 회원의 이메일·전화·auth_id를 긁어갈 수 있는** 상태라, 다른 어떤 항목보다 먼저 막아야 합니다. 같은 테스트에서 `chats`는 anon으로 0행(RLS 정상)이었으니, `users`에도 "본인 행만 select" 또는 공개 컬럼 뷰 정책을 추가하면 됩니다. SQL은 가이드에 있습니다.

### 박소유 (soyu) — 검색창 공동 컴포넌트 `SearchBar` + `formatTime`

**잘한 점**
- 중고/방렌트가 각자 갖고 있던 검색바를 `SearchBar` 하나로 합쳤습니다. `children` 슬롯으로 카테고리/필터 드롭다운을 끼워 넣고, `showLocation`(지역 드롭다운)은 `locationFilter`·`onLocationChange`가 둘 다 있을 때만 켜지게 한 설계가 깔끔합니다. 호출하는 쪽이 필요한 것만 넘기면 되니 재사용성이 좋아요. E2E에서 두 목록 모두 정상 렌더 확인했습니다(`images/3rd-01`, `3rd-02`).
- `formatTime.ts`의 `formatTimeAgo`에 주(`<5주`)·달(`<12달`)·년 단위를 더해 `923f7d0`에서 N달전/N년전까지 자연스럽게 끊기게 했습니다. `formatChatListTime`의 어제/N일전/N주전/N달전 분기도 경계가 정확합니다.

**`[제안]` 모바일·데스크톱 입력이 둘 다 DOM에 그려집니다**
`SearchBar`는 `md:hidden`(모바일)과 `hidden md:flex`(데스크톱) 두 벌의 `<input placeholder="검색어를 입력해주세요">`를 동시에 렌더합니다(CSS로 한쪽만 보임). 화면엔 문제없지만, 같은 placeholder 입력이 두 개라 테스트나 접근성 도구가 "어느 게 진짜냐" 헷갈립니다(E2E에서도 첫 매칭이 숨겨진 모바일 입력이라 `visible=true`로 따로 잡아야 했어요). 지금 단계에선 그대로 둬도 되지만, 입력 `value`/`onChange`가 한 state를 공유하니 **둘이 같은 상태를 가리키는 점**은 인지해 두세요. 나중에 `useMediaQuery`로 한쪽만 마운트하면 더 깔끔해집니다.

**`[사소]`** 드롭다운 위치를 `getBoundingClientRect()`로 클릭 시점에 계산해 `position: fixed`로 띄웁니다(`SearchBar`·`FilterDropdown` 공통). 드롭다운 연 채 스크롤하면 위치가 안 따라옵니다. 2차 `RentalList`에서 짚었던 것과 같은 패턴이라, 공통 컴포넌트로 빠진 지금 한 번에 고치기 좋은 위치가 됐습니다.

### 임태형 (THLIMM) — 이미지 업로드 공통화 `useImageUpload` / `ImageUploadField` / `useCreateUsedGoodsForm`

**잘한 점**
- `CreateUsedGoodsForm`에서 이미지 업로드·미리보기·삭제 로직을 `useImageUpload` 훅과 `ImageUploadField` 프레젠테이션 컴포넌트로 깨끗하게 분리했습니다(`b631b4d`, `55402b1`, `abdec08`). 폼 컴포넌트는 이제 마크업만 남고, 제출/상태 로직은 `useCreateUsedGoodsForm`(`0ba03b4`)로 빠져서 각 파일이 한 가지 일만 합니다. 2차 때 "글쓰기 폼이 길다"고 했던 부분을 정확히 이 방향으로 푼 거라 좋습니다.
- `useCreateUsedGoodsForm`의 제출 흐름이 탄탄합니다: posts insert 실패 → 중단, 이미지 업로드 실패 → **방금 만든 posts 행을 delete**(보상), used_goods insert 실패 → 중단. CreateJobForm에서 칭찬했던 보상 패턴을 여기서도 지켰습니다.
- `removeImage`에서 `URL.revokeObjectURL`로 미리보기 URL을 그때그때 해제하는 것도 좋습니다.

**`[필수]` `useImageUpload`의 언마운트 정리(cleanup)가 실제론 아무것도 안 지웁니다**
```ts
// useImageUpload.ts:10
useEffect(() => {
  return () => {
    imagePreviews.forEach((url) => URL.revokeObjectURL(url));
  };
}, []);   // ← 의존성 []
```
의존성이 `[]`라, 이 cleanup이 기억하는 `imagePreviews`는 **첫 렌더 시점의 값(=빈 배열)** 으로 고정됩니다(stale closure). 그래서 사진을 여러 장 올린 뒤 페이지를 떠나도 그 blob URL들은 해제되지 않고 메모리에 남습니다. 의도(언마운트 시 전부 정리)와 실제 동작이 어긋난 거예요.
- 함수형 cleanup이 항상 최신 배열을 보게 하려면 `imagePreviews`를 `useRef`에 같이 담아 ref로 정리하거나, 미리보기 추가/삭제 시점에서만 revoke하고 언마운트 effect는 빼는 방법이 있습니다. 가이드에 ref 버전 예시를 넣었습니다.

**`[제안]` 제출 검증과 사용자 피드백**
- `handleSubmit`이 `if (!productCategory || !condition || !preferredLocation) return;`로 조용히 멈춥니다. 사용자는 버튼을 눌렀는데 아무 반응이 없어 보여요. 필수값이 비면 어떤 항목이 비었는지 알려주면(또는 해당 필드에 표시) UX가 좋아집니다.
- 이미지 업로드를 `for ... of`로 **순차** 업로드합니다(한 장 끝나야 다음 장). 장수가 많아지면 느려요. `Promise.all`로 병렬화하면 빨라지는데, 그러면 "중간 실패 시 보상" 로직이 살짝 복잡해지니 지금 단계에선 알아만 두세요.

### 김도혁 (DoHyuk-Centric) — 이용약관 + Notion CMS + `GlobalLayout`

**잘한 점**
- 약관 본문을 코드에 하드코딩하지 않고 **Notion을 CMS로 써서** `getNotionContent(pageId)`로 받아 `react-markdown` + `remark-gfm`으로 렌더하는 구조(`dfc4241`, `1d64d72`, `3f0de01`)가 좋습니다. `getNotionContent`가 `pageId`를 인자로 받게 일반화해서 운영정책/서비스/개인정보/청소년 4개 페이지가 같은 함수를 재사용합니다. `export const revalidate = 3600`으로 ISR 캐시까지 건 것도 깔끔합니다.
- `GlobalLayout`(`f751199`)으로 Header/Footer/ScrollToTop을 한 곳에서 조건부 렌더하고, 각 페이지에 흩어져 있던 중복을 `de9625a`에서 제거했습니다. `/terms`·`/login`·`/signup`에서 전역 헤더/푸터를 숨기는 의도가 명확합니다. E2E에서 `/terms`가 전역 네비 없이 약관 전용 레이아웃으로 뜨는 것 확인했습니다.
- `TermsNav`의 모바일 드롭다운/데스크톱 탭 분기, 현재 경로 하이라이트(`pathname === href`)도 꼼꼼합니다.

**`[필수]` Notion 호출이 실패하면 페이지 전체가 에러 화면으로 죽습니다**
E2E에서 `/terms/policy`는 HTTP 200으로 라우팅됐지만, 화면은 **"Runtime APIResponseError — API token is invalid"** 전체 에러로 떴습니다(`images/3rd-03-terms-policy.png`).
- 솔직히 밝히면, 이번 리뷰 환경의 `.env.local`에 `NOTION_API_KEY`·`NOTION_*_PAGE_ID`가 **아예 없어서** 제 쪽 키 문제로 난 것이라, 학생 코드가 틀렸다는 뜻은 아닙니다. 운영 키로는 정상일 가능성이 큽니다.
- 다만 이 화면이 보여주는 진짜 문제는 **Notion(외부 API)이 죽거나 키가 만료되면 약관 페이지가 통째로 5xx 에러 화면이 된다**는 점입니다. 약관·정책은 "항상 보여야 하는" 정적 성격이 강하니, `getNotionContent`를 `try/catch`로 감싸 실패 시 안내 문구나 fallback 텍스트를 보여주는 게 안전합니다. `error.tsx`(라우트 에러 바운더리)를 `terms` 폴더에 두는 것도 방법입니다. 가이드에 두 방법 다 정리했습니다.

**`[제안]` env 누락 시 빌드/런타임 가드**
`process.env.NOTION_TERMS_POLICY_PAGE_ID!`처럼 non-null 단언(`!`)으로 쓰는데, 값이 없으면 `undefined`가 그대로 Notion에 넘어가 런타임에서 터집니다. 시작 시 필수 env를 한 번 검증하거나(없으면 명확한 메시지), 위 try/catch와 묶어 처리하면 좋습니다.

## 5. 팀 공통

- **라우트 단·복수 혼선이 3차에도 그대로입니다**(`/chats`, `/jobs`가 여전히 404). 2차에서 권했던 `routes.ts` 상수화가 아직 안 들어왔는데, 검색창·약관·이미지업로드를 공통 컴포넌트로 잘 뽑은 그 손길을 라우트에도 한 번 적용하면 이 부류 버그는 컴파일 단계에서 사라집니다(`ROUTES.chat`, `ROUTES.jobs`). 영향(정상 등록 후 404, 채팅 버튼 404)에 비해 수정량이 작습니다.
- **클라이언트 직접 DB 쓰기**가 여전히 여러 곳입니다(`RentalList` 찜, `useCreateUsedGoodsForm`이 `user_id`를 클라가 들고 insert). RLS가 받쳐주면 동작은 하지만 신뢰 경계가 클라로 내려와 있습니다. 글쓰기·찜은 점진적으로 서버 액션으로 옮기길 권합니다(2차와 동일).
- **리팩토링 방향 자체는 아주 좋습니다.** 검색바·이미지 업로드·폼 훅·약관 레이아웃 모두 "중복 제거 + 관심사 분리" 방향이라, 코드가 1·2차보다 확실히 정돈됐습니다. 다만 `[필수]` 백로그(① RLS ② 라우트 ③ 롤백/고아 ⑤ 서버조회·discord·빌드)가 누적되고 있으니, 새 기능과 백로그 정리를 번갈아 가는 리듬을 추천합니다.
- 커밋 컨벤션(`feat`/`refactor`/`chore`)·PR 머지 흐름은 건강합니다. 리뷰 브랜치를 짧게 만들고 머지 후 삭제하는 흐름도 잘 지켜지고 있습니다.
