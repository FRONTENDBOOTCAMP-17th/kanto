## 2. 이번 빌드 — 6라운드째 빨간불입니다

먼저 좋은 소식. 5차 에러 중 api/user import는 사라졌습니다. 그건 잘 닫았어요.

그런데 `npx tsc --noEmit`가 여전히 실패합니다(에러 22건). 5차와 마찬가지로 원인의 대부분이 **이번에 새로 들어온 코드**고, 5차에서 짚은 "한쪽이 구조를 바꿨는데 쓰는 쪽이 안 따라온" 패턴이 또 보입니다. 분포부터 보면 21건이 `src/app/` 아래, 1건이 `services/rental/rental.ts`입니다.

### 가. [필수] `admin/_components/DashboardClient.tsx` — 에러 19건의 진원지

`DashboardClient.tsx:11`이 `../_lib/constants`에서 `DashboardData` 타입을 import하는데, **`_lib/constants.ts`는 그 이름을 export하지 않습니다**(TS2305). `DashboardData`가 없으니 컴포넌트 함수 시그니처(`DashboardClient.tsx:164` `{ data }: { data: DashboardData }`)가 깨지고, 그 여파로 내부 `.map(...)` 콜백들의 파라미터(`s`,`r`,`l`,`i`,`c`,`t`,`p`,`m`)가 전부 implicit any가 되어 TS7006/TS7053이 18건 더 따라옵니다.

여기서 눈에 띄는 게 하나 있습니다. **`DashboardClient.tsx`는 어디에서도 import되지 않습니다**(`grep -rn "DashboardClient" src/` → 자기 정의 한 줄뿐). 실제 대시보드 화면(`admin/page.tsx`)은 `HeaderSection`/`KpiCards`/`TrendAndDonut`/`ReportQueue` 같은 분리된 작은 컴포넌트들을 씁니다. 그러니 `DashboardClient.tsx`는 리팩토링 과정에서 쪼개고 남은 **죽은 파일**로 보입니다. 죽은 파일이어도 tsc는 컴파일 대상에 넣으니 빌드 전체를 막아요. 안 쓰는 파일이면 지우고, 쓸 거면 `DashboardData` 타입을 정의해 import를 맞춰야 합니다. 어느 쪽인지부터 정하는 게 먼저입니다.

### 나. [필수] `rental/create/page.tsx:3` + `rental.ts:95` — 또 삭제된 `@/lib/supabaseServer`

```ts
// src/app/(user)/rental/create/page.tsx:3
import { createSupabaseServerClient } from "@/lib/supabaseServer";
```

5차에서 api/user가 똑같은 모듈을 import해서 깨졌던 그 자리입니다. 그 라우트는 고쳤는데, 이번에 새로 들어온 방렌트 글쓰기 페이지(`rental/create/page.tsx`)와 `services/rental/rental.ts:95`이 **삭제된 그 모듈을 다시 import**하고 있어요(TS2307 + TS2304). THLIMM이 서버 클라이언트를 `@/utils/supabase/server`로 단일화하면서 `supabaseServer.ts`를 지웠는데, 새 코드가 옛 이름을 또 쓴 거죠. 5차에 ESLint 룰까지 걸어뒀는데 이게 또 빠져나간 걸 보면, 룰이 새 파일을 못 잡고 있는 것 같습니다(4장에서 다시 봅니다). 고침은 `createClient`로 바꾸면 됩니다 — 가이드에 정확한 코드가 있습니다.

### 다. [필수] `usedgoods/page.tsx:63` — Filters에 넘기는 prop 이름이 또 어긋남 (TS2322)

`usedgoods/page.tsx:63`은 `defaultSearch`/`defaultCategory`/`defaultLocation`을 넘기는데, `UsedGoodsFilters`의 Props는 `givenSearch`/`defaultCategory`/`defaultLocation`입니다. `defaultSearch`라는 이름이 Props에 없어서 안 맞아요. 한쪽 이름을 다른 쪽에 맞추면 됩니다(둘 다 `givenSearch`로 통일 추천).

### 라. [필수] `usedgoods/[id]/page.tsx:63` — 5차 carryover (TS2322)

5차에 짚은 그대로입니다. 상세 select 결과 타입이 `UsedGoods` 타입에 안 맞아요(users alert 컬럼 누락). 아직 안 고쳐졌습니다.

네 건 다 5차에서 본 그림과 같습니다. 누가 코드를 못 짠 게 아니라, **구조를 바꾼 쪽(모듈 삭제·컴포넌트 분리·users 컬럼)과 그걸 쓰는 쪽이 같이 안 움직인** 자국이에요. 이건 4장에서 한 번 더 이야기하겠습니다.

## 4. 팀 공통 — 진전은 크고, 빌드 게이트만 아직 빈자리입니다

먼저 진전부터. 이번 라운드는 칭찬할 게 많습니다.

- `/chats` 404를 **라우트 제거로 원천 해소**(DoHyuk) — 네 라운드 끌던 항목을 구조적으로 닫았습니다.
- 어드민 RLS 우회 아키텍처를 **권한 게이트 뒤에서만** service role을 쓰도록 안전하게 설계(THLIMM).
- redis로 로그인 무차별 시도를 막는 **보안 장치 도입**(DoHyuk).
- 약관 Notion CMS 자동화로 운영 부담 감소(soyu).

각자 한 영역씩 들고 가는 모양이 이어지고 있네요. 5차에 물었던 것을 한 번 더 드립니다. 이 분담은 팀에서 합의해 나눈 건가요, 아니면 자연스럽게 자리 잡은 건가요? 어느 쪽이든 지금 흐름은 좋습니다.

다만 하나는 분명히 짚겠습니다. **빌드가 여섯 라운드째 빨간불입니다.** 매번 깨지는 자리에 공통점이 있어요.

- 5차: 회원탈퇴 라우트·SearchBar가 새로 들어오며 깨짐
- 6차: 어드민 DashboardClient(죽은 파일)·방렌트 새 페이지가 들어오며 깨짐

매 회차 **"이번에 새로 머지된 코드"가 빌드를 막습니다.** 기존 코드가 퇴행하는 게 아니라, 새 PR이 tsc를 통과하지 않은 채 develop에 들어오는 거예요. 이건 사람 문제가 아니라 **게이트가 없는 문제**라 봅니다. 사람이 매번 기억해서 막을 순 없으니까요. 그래서 5차에 이어 다시 권합니다. **PR이 develop에 머지되기 전에 `tsc`와 `build`를 자동으로 돌리는 GitHub Actions를 이번 주에 걸어두면**, 위 네 건 같은 에러는 머지 자체가 막혀서 develop이 늘 초록불을 유지합니다. 워크플로 파일 예시를 가이드에 적어뒀습니다. 다그치려는 게 아니라, 이번에 진전이 컸던 만큼 이 한 칸만 채우면 빌드 이슈가 구조적으로 사라진다는 이야기입니다.

그리고 2-나(삭제된 모듈 재import)가 5차에 이어 또 난 건, THLIMM이 걸어둔 ESLint 룰(`@/lib/supabaseServer` import 금지)이 새 파일을 못 잡고 있다는 신호입니다. 룰의 적용 범위(`files`/`overrides`)가 새 라우트·서비스 파일까지 포함되는지 한번 점검해 보면 좋겠습니다.
