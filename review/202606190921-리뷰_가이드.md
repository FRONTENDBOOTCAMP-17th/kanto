# Kanto 9차 리뷰 — 해결 가이드 (2026-06-19)

이번 리뷰에서 짚은 것들을 어떻게 고치면 좋은지 단계별로 적었습니다. 1번이 시연 전 최우선입니다.

---

## 1. [필수] 관리자 페이지·서버 액션 인가 막기

### 왜 막아야 하는가

지금 관리자 화면들은 `createAdminClient()`(service-role 키)로 데이터를 읽습니다. 이 키는 Supabase의 RLS(행 수준 보안)를 **통째로 우회**합니다. 평소엔 RLS가 "내 데이터만 보이게" 막아 주지만, service-role 앞에선 그 빗장이 풀려 모든 행이 다 보입니다. 그러니 이 페이지에 들어오는 사람이 진짜 관리자인지 페이지가 직접 확인해 주지 않으면, 일반 사용자가 주소만 알면 남의 채팅·회원정보를 그대로 보게 됩니다. 실제로 그렇게 새는 걸 E2E로 확인했습니다.

지금은 가드(아래 코드)가 `admin/page.tsx`와 `admin/reports/page.tsx` 두 곳에만 손으로 붙어 있고, 새로 만든 페이지에는 빠져 있습니다. 페이지마다 복붙하는 방식은 새 페이지를 만들 때 한 번만 깜빡해도 구멍이 나니, **헬퍼 하나로 모으는 것**을 권합니다.

### 1단계 — `requireAdmin()` 헬퍼 만들기

`src/utils/auth/requireAdmin.ts` 같은 위치에 한 곳에 모읍니다. (지금 페이지들이 쓰는 `createClient`(서버용, 쿠키 기반)를 그대로 씁니다 — 이건 RLS를 우회하지 않는, 로그인 세션을 읽는 클라이언트입니다.)

```ts
import { redirect } from "next/navigation";
import { createClient } from "@/utils/supabase/server";

// 현재 로그인 사용자가 admin 인지 확인한다. 아니면 즉시 리다이렉트.
// 반환값(현재 사용자 정보)은 필요하면 페이지에서 쓸 수 있다.
export async function requireAdmin() {
  const supabase = await createClient();

  const {
    data: { user },
  } = await supabase.auth.getUser();
  if (!user) redirect("/login");

  const { data: userRow } = await supabase
    .from("users")
    .select("id, role")
    .eq("auth_id", user.id)
    .single();

  if (userRow?.role !== "admin") redirect("/");

  return userRow; // { id, role }
}
```

### 2단계 — 모든 어드민 페이지 맨 앞에서 호출

`page.tsx`는 서버 컴포넌트라, 데이터를 가져오기 **전에** 한 줄만 부르면 됩니다.

```tsx
// admin/posts/page.tsx
import { requireAdmin } from "@/utils/auth/requireAdmin";
import { getAdminPosts } from "@/services/admin/adminPosts";
import AdminPostsClient from "@/app/(admin)/admin/posts/_components/AdminPostsClient";

export default async function AdminPostsPage() {
  await requireAdmin();          // ← 이 한 줄. admin 이 아니면 여기서 리다이렉트되고 아래는 실행 안 됨
  const posts = await getAdminPosts();
  return <AdminPostsClient posts={posts} />;
}
```

같은 식으로 `admin/chats/page.tsx`, `admin/chats/[id]/page.tsx`, `admin/users/page.tsx`, `admin/users/[id]/page.tsx`에도 첫 줄에 `await requireAdmin();`을 넣습니다. (이미 가드가 있는 `admin/page.tsx`·`admin/reports/page.tsx`도 이 헬퍼로 바꾸면 코드가 짧아지고 한곳에서 관리됩니다.)

> 더 깔끔하게 가고 싶으면 `(admin)/layout.tsx`(모든 어드민 페이지를 감싸는 레이아웃)에서 `await requireAdmin()`을 한 번 부르는 방법도 있습니다. 레이아웃이 먼저 실행되니 페이지마다 안 붙여도 됩니다. 다만 Next.js에서 레이아웃 가드만으로는 충분치 않은 경우(병렬 라우트 등)가 있어, 안전하게 가려면 **민감한 페이지에는 페이지에도 한 줄 더** 두는 편을 권합니다.

### 3단계 — 서버 액션에도 가드

서버 액션은 누구나 호출할 수 있는 엔드포인트라, 페이지를 막아도 액션이 뚫려 있으면 의미가 없습니다. `togglePostStatus` 맨 앞에도 같은 헬퍼를 둡니다.

```ts
// admin/posts/_actions/togglePostStatus.ts
"use server";
import { requireAdmin } from "@/utils/auth/requireAdmin";
import { setPostStatus } from "@/services/admin/adminPosts";

export async function togglePostStatus(postId: number, status: "active" | "inactive") {
  await requireAdmin();          // ← admin 이 아니면 여기서 막힘
  await setPostStatus(postId, status);
}
```

### 확인 방법

고친 뒤, role이 `user`인 계정으로 로그인한 상태에서 `/admin/posts`·`/admin/chats/1` 같은 주소로 직접 들어가 보세요. 이제는 `/`(또는 `/login`)으로 튕겨 나가야 정상입니다.

## 2. [제안] `blockUserAction` — 참여자인지 먼저 확인

지금은 차단 후 채팅의 "나갔음" 플래그를 이렇게 찍습니다.

```ts
const isUser1 = chat.user_id_1 === userData.id;
await supabase.from("chats")
  .update(isUser1 ? { user_id_1_left: true } : { user_id_2_left: true })
  .eq("id", chatId);
```

문제는 내가 그 채팅방 참여자가 **아닐 때**입니다. `isUser1`이 false가 되면서 엉뚱하게 `user_id_2_left = true`가 찍혀, 내가 끼지도 않은 채팅에 손을 대게 됩니다. 참여자인지 먼저 확인하고 갈라주면 됩니다.

```ts
const { data: chat } = await supabase
  .from("chats")
  .select("user_id_1, user_id_2")
  .eq("id", chatId)
  .single();
if (!chat) throw new Error("채팅을 찾을 수 없습니다.");

const isUser1 = chat.user_id_1 === userData.id;
const isUser2 = chat.user_id_2 === userData.id;
if (!isUser1 && !isUser2) throw new Error("이 채팅방의 참여자가 아닙니다."); // ← 추가

await supabase.from("chats")
  .update(isUser1 ? { user_id_1_left: true } : { user_id_2_left: true })
  .eq("id", chatId);
```

## 3. [제안] `getAdminChatMessages` — 없는 채팅방을 404로

상세 페이지는 `if (!chat) notFound();`를 기대하는데, 정작 서비스가 `.single()`을 써서 "id가 없으면 에러를 던지는" 형태라 `chat`이 null로 오지 못하고 500으로 떨어질 수 있습니다. `maybeSingle()`로 바꾸면 "없으면 null"이 되어 페이지의 `notFound()`와 잘 맞물립니다.

```ts
// services/admin/adminChats.ts
admin
  .from("chats")
  .select(`...`)
  .eq("id", chatId)
  .maybeSingle(),   // .single() → .maybeSingle()
```

## 4. [제안] 비밀번호 재설정 — 계정 존재 여부를 흘리지 않기

지금은 이름+이메일이 안 맞으면 404 "일치하는 계정이 없습니다"를 줍니다. 이러면 공격자가 이메일을 바꿔 넣어 가며 "이 이메일이 가입돼 있나?"를 떠볼 수 있습니다(account enumeration). 이럴 땐 계정이 있든 없든 같은 응답을 주는 게 안전합니다.

```ts
const matched = await findMatchingUser(name, email);

// 계정이 없어도 동일한 응답을 준다(존재 여부를 노출하지 않음).
// 메일은 매칭됐을 때만 실제로 보낸다.
if (matched) {
  const code = createVerificationCode();
  const key = getResetKey(email);
  await saveVerificationCode(key, code);
  // ... 기존 메일 발송 로직 ...
}

return NextResponse.json({
  message: "입력하신 정보로 안내 메일을 보냈습니다. 메일을 확인해주세요.",
  expiresIn: VERIFICATION_TTL_SECONDS,
});
```

> 필수는 아닙니다. 시연 이후 보안을 더 단단히 하고 싶을 때 적용하면 됩니다.

## 5. [제안] `/admin/posts` 목록 페이지네이션

`getAdminPosts`가 글 전체를 한 번에 끌어옵니다. 지금은 데이터가 적어 괜찮지만, 글이 쌓이면 한 화면에 수백~수천 건을 내려보내 느려집니다. `range`로 페이지를 나누는 게 가장 간단합니다.

```ts
export async function getAdminPosts(page = 0, size = 20): Promise<AdminPost[]> {
  const admin = await createAdminClient();
  const from = page * size;
  const { data, error } = await admin
    .from("posts")
    .select("id, title, post_type, status, view_count, created_at, users(name)")
    .order("created_at", { ascending: false })
    .range(from, from + size - 1);   // ← 페이지 단위로
  if (error) throw error;
  return (data ?? []).map(/* ... 기존 매핑 ... */);
}
```

클라이언트에서 페이지 번호를 올리며 더 불러오거나, 무한 스크롤을 얹으면 됩니다. 시연 이후 여유가 생길 때 권합니다.
