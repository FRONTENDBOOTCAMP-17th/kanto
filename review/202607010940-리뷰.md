## 2. 이번에 실측으로 다시 확인된 보안 문제 (최우선)

### [필수·보안] `users` 테이블이 anon 키로 통째로 읽힙니다

`.env.local`의 publishable(anon) 키만으로 이렇게 요청하면:

```
GET /rest/v1/users?select=* → HTTP 206, content-range 0-2/122
```

전체 122행이 그대로 내려옵니다. 노출되는 컬럼은 `email`, `phone`, `role`, `auth_id`인데, 더 신경 쓰이는 건 여기에 필터를 걸면 **어떤 계정이 관리자인지 정확히 골라낼 수 있다**는 점입니다.

```
GET /rest/v1/users?role=eq.admin&select=email,role
→ admin@kanto.com, (이동근님 계정), mark6896@naver.com, asdf1234@naver.com ...
```

박소유님 계정은 `super_admin`으로 이메일·전화까지 노출됩니다. 공격자 입장에서는 "누가 관리자인지" + "그 이메일"을 손에 쥔 셈이라, 크리덴셜 공격(비밀번호 대입·피싱)의 정확한 표적이 생깁니다. 이건 12차에서 한 번 짚었던 회귀가 아직 닫히지 않은 상태입니다.

반대로 나머지 민감 테이블은 RLS가 잘 걸려 있어서, `messages`·`chats`·`transactions`·`common_likes`·`common_reports`·`user_blocks`는 anon으로 전부 0행이었습니다(정상). `posts`·`meetups`·`meetup_participants`는 공개 의도라 열려 있는 게 맞고요. 딱 `users` 하나만 열려 있습니다. 해결 방향은 가이드 문서에 RLS 정책과 공개 뷰 분리 예시로 적어 뒀습니다.

## 5. 조원별 리뷰

### 박소유 (soyu) — 이번 delta의 대부분

이번 변경의 사실상 전부를 맡으셨습니다. 로직은 손대지 않고 주석만 걷어낸 뒤 `tsc`·`build`가 무사히 통과하는 걸 확인하고 올렸으니 안전한 정리였습니다. 두 가지만 짚을게요.

- **[사소]** 주석을 지운 자리에 빈 줄과 후행 공백이 대량으로 남아서(GlobalLayout·FloatingChatWidget·UnifiedBanner 등 연속 빈 줄), diff가 지저분해졌습니다. 정리를 하는 김에 빈 줄까지 같이 정돈하면 좋고, 이건 손으로 하기보다 prettier + eslint의 `no-multiple-empty-lines`·`no-trailing-spaces`로 한 번에 잡는 게 낫습니다.
- **[제안]** "주석을 전부 지운다"는 방침 자체는 다시 생각해 볼 만합니다. `// 이미 참여 중 — 멱등 처리`(go.ts) 같은 **의도를 설명하는 주석은 남기는 편**이 나중에 코드를 다시 볼 팀원에게 도움이 됩니다. 지워도 되는 건 "코드를 그대로 읽어 주는" 주석이고요.

### 김도혁 (DoHyuk) — 15차 [필수] ①②③가 담당분

이번 delta엔 실질 변경이 없어서 15차에 남긴 세 가지가 그대로입니다. 다만 먼저 칭찬부터 하면, **`requireAdmin` 헬퍼(user.ts:48)를 코드베이스 전반에 잘 정착시키셨습니다** — deletePost·togglePostStatus·applySanction·liftSanction·reports·admin chats·go 관리자 액션까지 광범위하게 붙어 있어서 9~12차에 짚었던 관리자 인가 문제가 거의 다 닫혔습니다. 설계도 role이 admin/super_admin 아니면 throw하는 식으로 깔끔하고요.

문제는 딱 두 파일이 이 패턴에서 빠져 있다는 겁니다.

- **[필수·보안]** `bulkPostActions.ts`의 세 함수(bulkTogglePostStatus/bulkRestorePosts/bulkDeletePosts)와 `bulkUserActions.ts`가 `createAdminClient`(service-role)로 바로 들어갑니다. `getCurrentAdminId()`는 부르지만 null을 반환만 할 뿐 **차단은 안 합니다.** 그래서 일반 로그인 회원이 서버액션을 직접 호출하면 남의 글·회원을 일괄 조작할 수 있습니다. 이미 갖고 계신 헬퍼라, 각 함수 첫 줄에 `await requireAdmin()`만 넣으면 닫힙니다.
- **[필수]** `getUsedGoodsItem`(usedGoods.ts:91)에 삭제 필터가 없습니다. 형제 함수 `getUsedGoodsDetail`(:83)·`getRentalDetail`엔 `.neq("status","deleted")`가 있는데 이 함수만 빠졌어요. 그래서 삭제된 글도 데이터가 truthy로 돌아오는 바람에 `page.tsx:18`의 notFound()를 안 타고, `UsedGoodsDetail.tsx:55`에서 `data.posts.users`를 읽으려는데 posts가 null이라 500이 납니다(E2E 실측). 삭제글이 화면에 "노출"되는 것보다, notFound로 안 빠지고 서버가 터지는 게 더 문제입니다.
- **[필수]** `api/posts/cleanup/route.ts:5`는 시크릿이 있을 때만 검증하고, 없으면 그냥 통과해서 영구 삭제를 실행합니다(fail-open). CRON_SECRET이 비어 있으면 아무나 이 엔드포인트로 글을 지울 수 있어요. `if (!secret) return 500`으로 막는(fail-closed) 방향이 맞습니다.

### 이동근 (dongkeun) — 15차 [필수] ④가 담당분

- **[필수]** `api/chat/[id]/route.ts:61`의 `posts?.post_type === "rentals"`가 오타입니다. 코드베이스 전역은 단수 `"rental"`을 씁니다(rental.ts:53,145 / main.ts:25 / posts/[id]/route.ts:48). 이 분기가 매칭이 안 돼서 렌탈 채팅에서 `postPrice`가 항상 null로 내려갑니다. `"rentals"` → `"rental"` 한 글자입니다.
- **[제안]** eslint의 `ChatRoom.tsx:46` set-state-in-effect, `PaymentCard.tsx:37` `Date.now()` 렌더 중 호출도 채팅/결제 영역이라 함께 손보면 좋겠습니다(3장 참고).

### 임태형 (THLIMM)

이번 delta엔 머지 커밋만 있어 실질 변경이 없습니다. 15차에 남긴 **삭제 정책 혼재**(사용자 삭제=hard delete `api/posts/[id]:56` vs 어드민=soft delete)가 여전한데, 이건 버그라기보다 팀이 합의할 사안이라, 하나로 통일할지 또 통일한다면 어느 쪽으로 갈지 팀에서 한 번 정하고 가면 좋겠습니다.

## 6. 팀 공통

- **보안·버그 픽스가 정리보다 먼저입니다.** 정리 커밋과 기능 커밋을 분리한 흐름 자체는 좋은데, 이번처럼 주석-only 커밋이 [필수] 4건 위에 얹히면 리뷰 대조가 "변화 없음"으로만 나옵니다. 남은 나흘은 순서를 뒤집어서 위 우선순위부터 닫고 정리는 그다음에 가는 게 좋겠습니다.
- **CI에 eslint 추가** — react-compiler 룰 error 9건이 지금 PR을 그냥 통과합니다.
- **`post_type` 리터럴 상수화** — "rental"/"jobs"/"used_goods" 문자열이 곳곳에 흩어져 있어서, 이번 "rentals" 같은 오타가 컴파일 타임에 안 걸립니다. 유니온 타입 상수 하나로 모으면 tsc가 오타를 잡아 줍니다.
