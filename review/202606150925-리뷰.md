## 2. 이번 빌드 — 아직 빨간불입니다 (5라운드째)

좋은 소식부터. 4차의 옛 에러(getLikeList 반환형, browser-client status)는 **사라졌습니다.** 그건 잘 닫았습니다.

다만 `npx tsc --noEmit`와 `npm run build`가 여전히 실패합니다. 원인은 4차와 다른 **새 것**인데, 공통점이 하나 있어요. 셋 다 **각자 만든 것이 머지될 때 서로 안 맞춰진** 데서 났습니다. 하나씩 보겠습니다.

### 가. [필수] `api/user/route.ts:2` — 삭제된 모듈을 import (TS2307)

```ts
// src/app/api/user/route.ts:2
import { createSupabaseServerClient } from "@/lib/supabaseServer";
```

`@/lib/supabaseServer`는 이미 삭제된 모듈입니다. THLIMM이 서버 Supabase 클라이언트를 `@/utils/supabase/server`의 `createClient`로 단일화하면서 `supabaseServer.ts`를 지웠습니다(커밋 `28c609d`). 이 단일화 자체는 옳고 좋은 방향이고요. 그런데 DoHyuk이 새로 만든 회원탈퇴 라우트(`api/user/route.ts`)는 삭제된 옛 모듈을 그대로 import하고 있어서, 두 PR이 머지될 때 서로 안 맞춰졌습니다.

고침은 import를 새 클라이언트로 바꾸고 호출부(8·50줄)도 `createClient()`로 맞추면 됩니다. 정확한 코드는 가이드에 적었습니다.

### 나. [필수] SearchBar prop 불일치 — TS2322 4건 + TS7006 6건

`src/components/common/SearchBar.tsx`가 받는 props(`onSearch`/`showLocation`/`children`)와, 그걸 쓰는 `JobFilters`/`RentalFilters`/`UsedGoodsFilters`가 넘기는 props(`searchInput`/`onSearchChange`/`onSearchSubmit`/`locationFilter`/`onLocationChange`)가 서로 다릅니다. 그래서 타입이 안 맞아 TS2322 4건, 콜백 파라미터 `e`·`v`가 implicit any라 TS7006 6건이 납니다.

이 Filters들은 `job/page.tsx:43`, `rental/page.tsx:51`, `usedgoods/page.tsx:49`에서 실제로 쓰이는 핫패스입니다. 화면이 직접 의존하는 컴포넌트라 빌드가 막혀요. SearchBar의 props 타입을 넓혀 Filters가 넘기는 모양을 받아주거나, 반대로 Filters를 SearchBar의 기존 API로 되돌리거나, 둘 중 하나로 통일해야 합니다.

### 다. [필수] `usedgoods/[id]/page.tsx:63` — 상세 쿼리 select가 새 컬럼을 빠뜨림 (TS2322)

users 타입에 `alert_*` 컬럼이 새로 생겼는데, 상세 페이지의 `posts.users` select는 그 컬럼을 가져오지 않습니다. 그러니 넓어진 users 타입과 select 결과 타입이 안 맞아요. select에 alert 컬럼을 추가하거나, 이 자리에서 쓰는 타입을 필요한 필드만으로 좁혀주면 됩니다.

세 건 다 공통점이 보입니다. 누가 코드를 잘못 짠 게 아니라, 한쪽이 구조를 바꿨는데(모듈 삭제, users 컬럼 추가, SearchBar API) 그걸 쓰는 다른 쪽이 같이 안 움직인 자국이에요. 이건 4장에서 한 번 더 이야기하겠습니다.

## 4. 팀 공통 — "함께 가기"가 눈에 띄게 좋아졌습니다, 빌드 게이트 하나만 더

지난 회차들에서 작업이 한 사람(김도혁)에게 몰려 보여 함께 가자고 했었는데, 이번엔 다릅니다.

- soyu가 알림 시스템 전체(Realtime 포함)를 냈고,
- dongkeun이 수정·신고·토스트 기능을 끝까지 완성했고,
- THLIMM이 서버 클라이언트 단일화 같은 횡단 수정을 맡았습니다.

커밋 숫자 격차(87 vs 8~15)는 여전히 있지만, **내용 면에서는** 각자 진짜 기능을 들고 있다는 게 코드에 드러납니다. 이 변화는 칭찬할 일이라 봅니다. 하나만 묻고 싶네요. 이 분담은 팀에서 합의해서 나눈 건가요, 아니면 자연스럽게 이렇게 자리 잡은 걸까요? 어느 쪽이든, 지금처럼 각자 한 영역을 책임지고 가는 모양이 좋습니다.

다만 이번 빌드깨짐 세 건이 한 가지를 분명히 보여줍니다. 셋 다 "한쪽이 구조를 바꿨는데 그걸 쓰는 다른 쪽이 같이 안 움직여서" 났어요.

- THLIMM이 `supabaseServer`를 지웠는데 → DoHyuk의 새 라우트가 옛 모듈을 import (2-가)
- users에 `alert_*` 컬럼이 생겼는데 → usedgoods 상세 select가 안 따라옴 (2-다)
- SearchBar API가 바뀌었는데 → Filters가 옛 모양으로 호출 (2-나)

전에 페어 프로그래밍을 한 번 권한 적이 있는데, 이번 세 건이 정확히 "둘이 같이 봤으면 안 났을" 사례들이에요. 모듈을 지우거나, 공통 컴포넌트 API를 바꾸거나, 공유 테이블에 컬럼을 더할 때는 **그걸 쓰는 사람과 짧게라도 같이 보거나 공유**하고 가면 이런 갭이 안 생깁니다.

그리고 한 가지는 분명하게 짚겠습니다. **빌드가 다섯 라운드째 빨간불입니다.** develop이 빌드가 안 되면 곧 배포도 안 됩니다. 지금 macOS 로컬에서는 통과하더라도 CI나 배포 환경(Linux)에서 더 잘 깨지고요. 다그치려는 게 아니라, 이번엔 많이 닫은 만큼 이 한 가지를 같이 닫자는 겁니다. 빌드 복구가 미뤄진 이유가 따로 있나요? 없다면, **새 기능을 머지하기 전에 `tsc`/`build`를 통과시키는 규칙**(GitHub Actions에 `tsc` + `build` 두 줄)을 이번 주에 걸어두면 좋겠습니다. 가이드에 예시를 적어뒀습니다.
