# Kanto 9차 코드 리뷰 (2026-06-19)

월요일 중간 시연을 앞두고, 8차(2026-06-18) 이후 `develop`에 들어온 변경을 봤습니다. 이번 회차는 들어온 양도 많고(머지 기준 70여 커밋) 무게도 묵직합니다. 채팅방 유저 신고·차단, 글쓰기 본인인증 모달, 중고 판매완료·예약중, 거래후기, 관리자 글 관리(`/admin/posts`)·채팅 관리(`/admin/chats`), 비밀번호 재설정까지 한 번에 들어왔네요. 늘 하던 대로 앱을 먼저 띄워 사용자 → 관리자 순서로 흐름을 돌려 보고(아래 "E2E 결과"), 거기서 드러난 동작을 근거로 코드를 봤습니다.

먼저 좋은 소식을 말씀드리면, 8차에서 제일 급했던 세 가지가 모두 정리됐습니다. `/profile` 크래시도 고쳐졌고, tsc가 8라운드 만에 0건으로 떨어졌으며, 그렇게 미뤄지던 CI 게이트도 드디어 붙었습니다. 이건 분명한 진전이라 생각하고, 아래에서 하나씩 짚겠습니다.

다만 새로 들어온 관리자 화면에서 한 가지 무거운 문제를 봤습니다. 시연 전에 이것부터 막는 게 좋겠습니다.

## 2. 이번 회차에서 가장 급한 것 — 관리자 페이지 인가(authz) 누락 [필수]

새로 들어온 관리자 화면을 보다가, 일반 사용자가 관리자 데이터에 그대로 접근할 수 있는 구멍을 발견했습니다. E2E로 실제로 재현해 본 결과라 확실합니다.

테스트 계정(role = `user`, 즉 관리자가 아닌 평범한 회원)으로 로그인한 상태에서 `/admin/chats/11` 주소로 직접 들어가 봤더니, 아무 제지 없이 **다른 회원 두 명(김도혁·이동근)의 1:1 채팅 전문**이 화면에 그대로 떴습니다(E2E K30, 캡처 `kanto-K30_admin_chats_11.png`). `/admin/posts`·`/admin/chats`·`/admin/users`도 마찬가지로 리다이렉트 없이 200으로 렌더됐습니다.

원인은 인가 가드가 페이지마다 손으로 붙어 있는데, 일부 페이지에만 붙어 있어서입니다. 확인해 보면 가드가 있는 페이지는 두 곳뿐입니다.

```tsx
// admin/page.tsx, admin/reports/page.tsx — 가드 있음
const { data: { user } } = await supabase.auth.getUser();
if (!user) redirect("/login");
const { data: userRow } = await supabase.from("users").select("role").eq("auth_id", user.id).single();
if (userRow?.role !== "admin") redirect("/");
```

그런데 이번에 새로 들어온 `admin/posts/page.tsx`·`admin/chats/page.tsx`·`admin/chats/[id]/page.tsx`, 그리고 이전부터 있던 `admin/users/page.tsx`·`admin/users/[id]/page.tsx`에는 이 가드가 빠져 있습니다. 예컨대 글 관리 페이지는 이렇게 곧장 데이터를 읽습니다.

```tsx
// admin/posts/page.tsx — 가드 없음
export default async function AdminPostsPage() {
  const posts = await getAdminPosts();   // 바로 service-role 로 전체 글 조회
  return <AdminPostsClient posts={posts} />;
}
```

여기서 더 무거운 건, 이 화면들이 `createAdminClient()`(service-role 키)로 데이터를 읽는다는 점입니다. service-role 키는 **RLS를 통째로 우회**합니다(`utils/supabase/admin.ts` 주석에도 적혀 있죠). 즉 평소 RLS가 막아 주던 "남의 채팅·남의 회원정보"가, 이 페이지에 들어오는 순간 가드가 없으면 그대로 흘러나옵니다. 미들웨어(`middleware.ts`)도 `/admin`을 따로 막지 않아서, 로그인만 했으면(심지어 비로그인이라도 일부 경로는) 주소만 알면 들어올 수 있는 상태입니다.

같은 문제가 서버 액션에도 하나 더 있습니다.

```ts
// admin/posts/_actions/togglePostStatus.ts
"use server";
export async function togglePostStatus(postId: number, status: "active" | "inactive") {
  await setPostStatus(postId, status);   // caller 가 admin 인지 확인하지 않음
}
```

`setPostStatus`는 다시 service-role 클라이언트로 `posts.status`를 바꿉니다. 서버 액션은 사실상 누구나 호출할 수 있는 HTTP 엔드포인트라, caller가 관리자인지 확인하지 않으면 일반 사용자가 임의의 글을 `inactive`로 내려 버릴 수 있습니다.

결국 **읽기(채팅·회원·글 조회)도 쓰기(글 상태 변경)도 관리자 확인이 빠진 채 RLS를 우회하는 service-role을 쓰고 있어서** 영향이 큽니다. 시연에서 관리자 화면을 보여줄 예정이라면 더더욱, 그 전에 막아 두는 게 좋겠습니다.

해법은 어렵지 않습니다. 페이지마다 가드를 복붙하지 말고 `requireAdmin()` 헬퍼 하나로 모으면 빠뜨릴 일도 없어집니다. 서버 액션 맨 앞에도 같은 헬퍼를 한 줄 두면 됩니다. 구체적인 코드는 함께 드리는 가이드(1번)에 적어 두었습니다.

## 3. 멤버별 리뷰 (신규 코드)

8차 이후 들어온 코드를 작성자별로 봤습니다. 전반적으로 설계 수준이 또 한 단계 올라왔다는 인상이라, 잘된 부분은 그대로 짚고 손볼 곳을 함께 적습니다.

### 3-1. 이동근(@dongkeun99) — 비밀번호 재설정 `/api/auth/reset-password`

비밀번호 재설정처럼 "로그인 전 미인증 상태에서 본인을 확인해야 하는" 흐름은 실수가 나기 쉬운데, 여기는 꽤 단단하게 짜였습니다.

- POST(코드 발송) → PATCH(코드 검증) → PUT(재설정)으로 단계가 또렷하고, **PUT에서 인증번호를 다시 검증**합니다(146행). 클라이언트가 PATCH를 건너뛰고 곧장 PUT을 때려도 통과되지 않게 막은 점이 좋네요. 인증 단계를 우회하는 공격을 정확히 의식한 코드라 생각합니다.
- 비밀번호 규칙(`PASSWORD_REGEX`)을 회원가입과 동일하게 맞췄고, 메일 발송 실패 시 저장한 코드를 지워(orphan 방지) 롤백합니다.
- `devCode`를 `NODE_ENV !== "production"`일 때만 노출하고, 운영에서 메일이 미구성이면 코드 노출 대신 에러로 막는 가드(84~93행)까지 들어가 있습니다. 8차에서 말씀드린 부분을 그대로 반영했네요.

- [제안] `findMatchingUser`는 이름이 안 맞아도(404), 메일은 보내지 않으니 큰 문제는 아니지만, "일치하는 계정이 없습니다"를 그대로 내려보내면 공격자가 어떤 이메일이 가입돼 있는지 떠보는(account enumeration) 단서가 됩니다. 이럴 땐 존재 여부와 무관하게 "입력하신 정보로 안내 메일을 보냈습니다"처럼 동일한 응답을 주는 편이 안전합니다. 지금 단계에서 필수는 아니고, 더 단단하게 가고 싶을 때 기억해 두면 좋은 패턴입니다.

### 3-2. 박소유(@soyupark1997) — 관리자 채팅 관리 `/admin/chats`·`/admin/chats/[id]`

채팅 목록·상세를 운영자 시점에서 보여주는 화면인데, 메시지를 날짜 구분선으로 묶고 보낸 사람 기준으로 좌우 정렬하는 등 채팅 뷰 자체는 깔끔하게 잘 만들었습니다(E2E K24). `getAdminChatMessages`에서 채팅 정보와 메시지를 `Promise.all`로 동시에 가져오는 것도 좋네요.

- [필수] 이 페이지에는 **2번에서 말씀드린 관리자 가드가 빠져 있습니다.** `adminChats.ts`가 service-role로 모든 채팅을 읽기 때문에, 가드가 없으면 일반 사용자에게 남의 1:1 채팅이 그대로 노출됩니다. 가장 민감한 데이터라 우선순위가 제일 높습니다.
- [사소] `getAdminChatMessages`의 `chats ... .single()`은 해당 id가 없으면 에러를 던집니다. 지금은 `notFound()`로 받지 못하고 500으로 떨어질 수 있어서, `maybeSingle()`로 바꾸면 "없는 채팅방"을 깔끔하게 404로 처리할 수 있습니다(상세 페이지가 이미 `if (!chat) notFound()`를 기대하고 있으니 잘 맞물립니다).

### 3-3. 김도혁(@DoHyuk-Centric) — 관리자 글 관리 `/admin/posts`

글 목록을 한 화면에 모아 카테고리·작성자·조회수·상태를 보여주고, 상태를 토글로 바꾸는 흐름이 운영자 입장에서 직관적입니다(E2E K23). `getAdminPosts`에서 `users(name)` 조인 결과가 배열로 올 수도, 객체로 올 수도 있는 경우를 둘 다 처리해 둔 것도 꼼꼼하네요.

- [필수] 여기도 **관리자 가드 누락**이고, 더해서 `togglePostStatus` 서버 액션에 caller 검증이 없습니다(2번). 읽기뿐 아니라 "글 내리기" 쓰기까지 일반 사용자가 호출할 수 있어, 가드를 페이지·액션 양쪽에 둬야 합니다.
- [제안] `getAdminPosts`가 글 전체를 정렬만 해서 한 번에 다 끌어옵니다. 지금 데이터 규모에선 괜찮지만, 글이 쌓이면 한 화면에 수백~수천 건을 내려보내게 됩니다. 시연 이후 여유가 생기면 페이지네이션(또는 무한 스크롤)을 얹는 걸 권합니다.

### 3-4. 임태형(@THLIMM) — 채팅방 유저 신고/차단

차단 액션(`blockUserAction`)이 좋은 판단을 여럿 담고 있습니다. `auth.getUser()`로 요청자를 먼저 확인하고, 차단하는 사람의 `users.id`를 **클라이언트가 보낸 값이 아니라 서버에서 직접 조회해** 쓰며(13~18행), 차단 row는 `upsert`에 `ignoreDuplicates`를 줘서 중복 차단도 안전하게 처리합니다. 신고 여부 조회(`report.ts`의 `checkReported`)를 `single()`에서 `maybeSingle()`로 바꿔 "신고 이력 0건"일 때 나던 406을 없앤 것도 정확한 수정이네요.

- [제안] `blockUserAction`은 차단 후 `chats`의 `user_id_1_left`/`user_id_2_left`를 갱신하는데, **요청자가 그 채팅방의 참여자인지 확인하지 않습니다.** 지금은 `isUser1 = chat.user_id_1 === userData.id`로 갈라서, 만약 내가 참여자가 아닌 다른 채팅방 id를 넘기면 `isUser1`이 false가 되어 엉뚱하게 `user_id_2_left = true`가 찍힙니다. 7차 때 `leaveChatAction`에서 같은 패턴을 말씀드렸는데 결이 같습니다. `chat.user_id_1 === userData.id || chat.user_id_2 === userData.id`로 참여자인지 먼저 확인하고, 둘 다 아니면 막아 주면 깔끔합니다.

## 4. 팀 공통 / 협업

- **인가 로직이 페이지마다 복붙되는 구조가 이번 구멍의 근본 원인이라 생각합니다.** 같은 6~7줄짜리 role 체크를 페이지마다 손으로 붙이다 보니, 새 페이지(`posts`·`chats`)를 추가하면서 자연스럽게 빠진 거죠. 이런 "빠뜨리면 보안이 뚫리는" 코드일수록 한 곳(`requireAdmin()` 같은 헬퍼나 그룹 레이아웃)에 모아 두는 게 안전합니다. 새 어드민 페이지를 만들 때 가드를 기억해야 하는 게 아니라, 기본적으로 막혀 있고 헬퍼를 부르는 형태가 좋습니다(가이드 1번).
- **CI 게이트가 붙은 건 팀 전체에 큰 자산이라 생각합니다.** 다만 지금 게이트는 tsc·build(타입·빌드)까지만 봅니다. 이번 인가 누락처럼 타입은 멀쩡한데 동작이 위험한 경우는 게이트가 잡지 못하니, 그 빈틈을 코드 리뷰와 E2E가 메워야 한다는 점을 팀이 함께 의식하면 좋겠습니다.
- 작성자 분포는 이번에도 4명에 고루 퍼져 있습니다(재설정 dongkeun, 채팅관리 soyu, 글관리 DoHyuk, 신고/차단 THLIMM). 좋은 흐름이네요.
- 커밋 컨벤션은 대체로 잘 지켜지는데, `chor:`(→`chore:`) 오타가 이번에도 몇 건 보입니다. 사소하지만 통일하면 좋겠습니다.

## 6. 다음 회차(시연)까지 우선순위

1. **[필수] 관리자 페이지·서버 액션 인가 막기** — `/admin/posts`·`/admin/chats(/[id])`·`/admin/users(/[id])`에 관리자 가드 추가, `togglePostStatus`에 caller 검증 추가. `requireAdmin()` 헬퍼로 모으는 걸 권합니다(2장·가이드 1번). **시연 전 최우선.**
2. **[제안] `blockUserAction` 참여자 검증** — 차단 시 요청자가 그 채팅방 참여자인지 먼저 확인(3-4).
3. **[제안] `getAdminChatMessages`의 `single()` → `maybeSingle()`** — 없는 채팅방을 404로(3-2).
4. **[제안] 비밀번호 재설정 account enumeration 완화** — 계정 존재 여부와 무관한 동일 응답(3-1).
5. **[제안] `/admin/posts` 목록 페이지네이션** — 글 누적 대비(3-3).

8차의 급한 불 세 개(프로필 크래시·빌드·CI)를 한 번에 다 잡고 들어온 회차라, 안정성 면에서 확실히 단단해졌습니다. 남은 건 새로 들어온 관리자 화면의 인가 한 가지인데, 헬퍼 하나로 모으면 금방 막힙니다. 그것만 정리하면 시연에서 관리자 기능까지 자신 있게 보여줄 수 있는 단계라 생각합니다. 화이팅입니다.
